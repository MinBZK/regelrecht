## Context

[RFC-022](./rfc-022) decides the architecture: chronolexogram types, the cell model, the two cross-cutting fields, and cross-cell queries. This RFC carries the *how*, the schema and engine work that follows from those decisions, kept separate so RFC-022 stays an architecture record rather than a build plan. RFC-022 is the source of truth for *why* each item exists; this RFC says where it lands in the codebase.

It also collects the three reused-mechanism amendments RFC-022 flagged but did not specify, so the open security question (RFC-009 key-reuse) lives next to the work it gates rather than in the architecture RFC.

Nothing here is built yet. The schema is at v0.5.x and `schema/latest` points to it; there is no `chronicles/` directory, no chronicle schema, no `decision_type` vocabulary file, and no engine support for cells, lexostatus, or the Blauwe-Knop feature-blocks.

## Decision

### Schema version bump

This work lands as a new schema version, `schema/v0.5.x` → `schema/v0.6.0`, with the `schema/latest` symlink updated. The bump is close to purely additive (see RFC-022 Tradeoffs):

- Opening `decision_type` from a closed enum to `type: string` breaks no consumer (an unrecognised value was unrepresentable before, so nothing relied on the field being closed) and stays backward-compatible for producers (every existing value still validates).
- The new optional `modality` and `extensions` blocks and the new `chronicle.json` are fully additive.
- Cross-cell queries require no schema change; they reuse the existing `source` block with a runtime resolution rule.

The minor digit moves rather than the patch digit because relaxing a field's type is a meaningful contract change worth signalling, not because it breaks consumers. The bump lands in Phase 1; until then v0.5.x remains the validated schema.

### Phase 1: Schema

- Bump `schema/v0.5.x` → `schema/v0.6.0`; update the `schema/latest` symlink.
- Open `decision_type` in `schema.json` from a closed enum to `type: string` (no inline enum).
- Add `schema/v0.6.0/vocabulary/decision_type.yaml`: the recommended vocabulary (the nine historical values plus `BETALINGSVERPLICHTING`, `STRAFBESCHIKKING`, `BESTUURLIJKE_BOETE` as `id`/`label`/`description` entries), modelled on `corpus/annotations/_vocabulary/ambiguity.yaml` (RFC-018 Decision 9).
- Wire one read of that file into both the editor `decision_type` picker and `just validate`; validation MAY warn on an out-of-vocabulary value but MUST NOT reject (mirroring `validate-annotations`).
- Add the `produces.modality` optional block.
- Add `produces.extensions` as a free-form namespaced map. Within it, constrain `blauwe_knop.visible_from_stage` to a closed enum (the RFC-008 stage set for Awb-route decretypes), so a typo or out-of-range value fails schema validation rather than silently activating.
- Add `schema/v0.6.0/chronicle.json` for chronicle-stream files, including the optional `references_decision` event field (RFC-022 §3.3).
- No schema change for cross-cell queries; the existing `source` block is reused, with runtime-context-driven transport selection (Phase 2).
- All changes additive; no existing regulation YAML breaks.

### Phase 2: Engine support

- New `packages/engine/src/chronicle.rs`: loading, validating, and triggering chronicle-stream entries.
- New types in `packages/engine/src/types.rs`: `ChronicleStream`, `Modality`.
- Update `packages/engine/src/resolver.rs`: resolve `source.regulation` following the canonical order in RFC-022 §4.2 (data-source tier, then loaded-regulation tier, then cell tier; regulation always wins over a same-id cell; a shadowing cell-id is a load-time error). The new work is the cell tier: when no loaded regulation matches, dispatch via the security context's transport (Blauwe Knop when it has a `blauwe_knop_client` block, FSC when it has an `fsc_client` block). Each query evaluates the requested lexostatus via the receiving cell's declared chronolexoreductie. RFC-022 §4.2 is normative; do not re-specify the precedence here.
- New `packages/engine/src/security_context/blauwe_knop_source.rs`: when activated by the security-context feature-block, exposes a Blauwe-Knop-source endpoint that, on a verified citizen-pull, runs the cell's declared lexostatus-reduction, **enforces the `visible_from_stage` guard per candidate decretogram** (`current_stage >= visible_from_stage`, excluded besluiten simply absent), and serialises an FCID-response signed with the security context's signing key. This **load-time cross-field check** (a route-bearing decretype may not set `visible_from_stage` below its route-populating stage) is what RFC-022 §3.2 gates the "experimental" marker on; until it ships, a production security context must not activate the source for any route-bearing decretype.
- New `packages/engine/src/security_context/blauwe_knop_client.rs`: when activated, resolves cross-cell queries by pulling the named cell's Blauwe-Knop-source on behalf of the active citizen session.
- New `packages/engine/src/lexostatus.rs`: loading cell `capabilities.yaml` declarations, executing chronolexoreductions, serialising to the requested transport format.
- Update the RFC-013 Execution Receipt builder (`packages/engine/src/receipt.rs`) to include loaded chronicle-streams and lexostatus-definitions alongside regulations.

### Phase 3: Vocabulary and docs

- Update [law-format](/concepts/law-format), [multi-org-execution](/concepts/multi-org-execution), [execution-provenance](/concepts/execution-provenance) to use the cell/chronicle/lexogram vocabulary.
- Add glossary entries for the chronolexogram terms (*lexogram*, *decretogram*, *executogram*, *chronicle*, *reductie*, *lexostatus*, cell).

### Phase 4: Pilot

- One chronicle-stream YAML under `chronicles/` as a working example.
- One integration spec using `extensions.<namespace>`. The first integration is `blauwe_knop` (a cell-as-source for the Blauwe Knop Connect standard, with FCID as the payload type); see the Blauwe-Knop-source appendix of the [CJIB pilot proposal](/concepts/cjib-blauwe-knop-source-proposal) for stage-binding, route derivation, and field-level mapping.

### Amendments to existing RFCs

RFC-022 reuses three existing mechanisms in ways that ask for a small canonical amendment to the corresponding RFCs. The amendments land as follow-up commits after the CJIB pilot workshop confirms the operational details.

- **RFC-007 (Cross-Law Execution).** The `source.regulation` resolver is extended so that a value naming a *cell* (not a regulation in the loaded corpus) dispatches via the runtime-selected cross-cell transport. RFC-007 currently knows only the cross-law case. The amendment is to the resolver, not to the `source` object (the object originates in RFC-001 §9 and is codified in the schema). Additive; does not break RFC-007's existing use.
- **RFC-013 (Execution Provenance).** The Execution Receipt's `loaded_regulations` array is generalised to record loaded chronicle-streams and lexostatus-definitions alongside regulations. A rename/generalisation of an existing field; additive.
- **RFC-009 (Multi-Org Execution / FSC).** This is the one item with a genuine open question. RFC-022 raises whether the cell's FSC private key may also serve as the JWS-signing key for Blauwe-Knop FCID-responses. RFC-009 keeps payload-signing (§5) and transport-authentication (§6, FSC-mTLS) in separate layers and does not contemplate one key serving both, so key-reuse is a **new, security-relevant decision**, not a clarification. The canonical amendment must treat it as an open security question (key-usage / EKU constraints, certificate-vs-raw-key, coupled rotation, separation of duties) and may well conclude that separate key pairs are required. To keep this off the critical path, the pilot uses a **dedicated JWS signing key** (separate from the FSC-mTLS key), so the signing design works regardless of how the amendment resolves; key-reuse remains a later optimization, not a pilot blocker.

## Why

### Benefits

- **RFC-022 stays an architecture record.** A reader judging the design is not wading through schema diffs and Rust module names; a reader implementing it has the build plan in one place.
- **The open security question lives next to the work it gates.** The RFC-009 key-reuse decision is in the implementation RFC, not buried in the architecture RFC's context.
- **The bump is signalled honestly.** A minor-version move marks a contract change without falsely implying a break.

### Tradeoffs

- **Two RFCs to keep in step.** If the architecture in RFC-022 shifts, the affected-components table here must follow. Acceptable: the table is mechanical, and splitting it out is what keeps RFC-022 stable enough to be worth referencing.
