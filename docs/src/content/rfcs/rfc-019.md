---
title: "RFC-019: Units of Measurement on Values"
---

**Status:** Proposed
**Date:** 2026-06-01
**Authors:** Tim de Jager

## Context

Values in our law YAML carry an implicit unit that lives only in a field's name and
in convention. A `definitions` block looks like this:

```yaml
definitions:
  drempelinkomen_alleenstaande:
    value: 3971900        # = тВм39.719  тАФ stored in eurocent
  percentage_drempelinkomen_alleenstaande:
    value: 0.01896        # = 1,896%   тАФ stored as a decimal ratio
```

Nothing in `3971900` says whether it is euros or eurocent, and nothing in `0.01896`
says whether it is a ratio (0тАУ1) or a percentage (0тАУ100). A reader тАФ human or
machine тАФ that guesses wrong is off by a factor of 100. The units are present in
the law text itself; they should be present in the machine-readable representation
too, so a value cannot be misinterpreted during a calculation.

### Current situation

- A unit mechanism already exists, but only on `execution.input[]`, `execution.output[]`
  and `parameters[]` via `type_spec.unit`. The schema enum is
  `["eurocent", "years", "weeks", "months", "days"]`.
- The **`definitions` block** тАФ the constants shown above тАФ is schema-free
  (`{ "type": "object", "additionalProperties": true }`) and cannot carry a unit at all.
  This is the main gap: it is exactly where misinterpretation originates.
- The engine **ignores units in arithmetic**. `ADD`/`SUBTRACT`/`MULTIPLY`/`DIVIDE`
  (`packages/engine/src/operations.rs`) are unit-blind; today `eurocent + days` succeeds
  silently. The only place a unit does anything is rounding `eurocent` outputs to
  integers at the API boundary (`packages/engine/src/service.rs`).
- Percentages are stored inconsistently тАФ sometimes as a ratio (`0.137`), sometimes
  as a whole percent (`30`) тАФ and carry no unit.

### Constraints

- Roughly all existing laws declare no units. The change must not break them: laws must
  keep validating and executing exactly as before until they opt in by adding annotations.
- The unit rules used during static validation and during runtime execution must never
  diverge. A single source of truth is required.

## Decision

We introduce **units of measurement as a first-class, statically- and dynamically-checked
property of values**, built on the existing `type_spec.unit` field and extended to the
`definitions` block.

### 1. Unit model

A new module `packages/engine/src/units.rs` defines the model. Units are grouped into
dimensions:

| Dimension   | Units                          | Example              |
|-------------|--------------------------------|----------------------|
| Money       | `euro`, `eurocent`             | `3971900` (eurocent) |
| Time        | `years`, `months`, `weeks`, `days` | `12` (months)    |
| Ratio       | `ratio` (dimensionless, 0тАУ1)   | `0.137`              |
| Percentage  | `percentage` (0тАУ100)           | `13.7`               |
| Count       | `count` (dimensionless)        | aantal kinderen      |
| (none)      | `Unknown` (not declared)       | a bare literal       |

`Unit` is a Rust enum with an explicit `Unknown` variant. **A value whose unit is
`Unknown` never triggers a check** тАФ this is the backward-compatibility cornerstone:
laws without annotations behave exactly as today.

`euro` and `eurocent` are distinct units; mixing them is flagged (this is the original
euro-vs-eurocent concern). Likewise `ratio` and `percentage` are distinct and both
allowed; the law declares which it means.

### 2. Algebra

A single function `combine(op, lhs, rhs) -> Result<Unit, UnitMismatch>` encodes the
(deliberately simplified) algebra. If either operand is `Unknown`, the result is
`Unknown` and no error is ever produced.

| Operation                | Rule                                                       | Error on known, conflicting units |
|--------------------------|------------------------------------------------------------|-----------------------------------|
| `+`, `тИТ`, `MIN`, `MAX`   | both operands same unit тЖТ that unit                        | yes (`eurocent + days`, `euro + eurocent`) |
| comparisons (`<`, `==`, тАж) | both operands same unit                                  | yes |
| `├Ч`                      | `X ├Ч ratio = X`, `X ├Ч count = X`, `ratio ├Ч ratio = ratio`  | yes for any other known combination (e.g. `eurocent ├Ч eurocent`, `eurocent ├Ч percentage`) |
| `├╖`                      | `X ├╖ ratio = X`, `X ├╖ count = X`, same-dimension ├╖ same-dimension = `ratio` (e.g. `money ├╖ money`)  | yes for any other known combination |
| `IF` / branches          | all branches same unit тЖТ that unit                         | yes if branches disagree |

We do **not** track compound units (e.g. `eurocent/maand`). Combinations the table
cannot resolve between two known units are reported as a mismatch rather than carried
as a derived unit. In practice divisions are by dimensionless counts (e.g. a yearly
amount ├╖ 12), which the rules above accept.

### 3. Shared inference: the validator uses the engine

A second function in `units.rs`:

```rust
fn infer_unit(expr: &ActionValue, symbols: &SymbolUnits) -> Result<Unit, UnitMismatch>
```

recursively walks the operation tree (`ActionValue` / `ActionOperation`) applying
`combine()`. It is **value-independent** тАФ it inspects only the declared units of
symbols, not concrete numbers тАФ so it naturally visits *all* branches of an `IF`.

Both checking paths call this same function, so the static and runtime rules cannot
drift apart:

| Path                          | How it uses `infer_unit`                                                                 |
|-------------------------------|------------------------------------------------------------------------------------------|
| **Static** (`validate` bin)   | New step 3 after serde + JSON-schema. Runs `infer_unit` over every action/output expression, across all branches, without executing the law. |
| **Runtime** (`engine.rs`, `evaluate_action`) | The engine runs `infer_unit` per action during execution (before dispatching to `operations.rs`); a `UnitMismatch` propagates as an `EngineError`. |

This directly answers "let the validation script use the engine code": the `validate`
binary already lives in the engine crate and deserializes to `ArticleBasedLaw`; it
reuses the very same inference the runtime uses.

### 4. Symbol table

`infer_unit` resolves a symbol тЖТ unit table per article, built from:

| Source                              | Unit comes from                                                                 |
|-------------------------------------|---------------------------------------------------------------------------------|
| `definitions[].type_spec.unit`      | declared locally (new тАФ see schema change)                                      |
| `execution.input[].type_spec.unit`  | declared locally on the input                                                   |
| `parameters[].type_spec.unit`       | declared locally                                                                |
| literals (`value: 0.137`)           | `Unknown`                                                                       |

A symbol not found тЖТ `Unknown` тЖТ never an error.

**Implementation status (initial landing).** The symbol table is built from
*locally declared* units only. Resolving a cross-law `source` (`regulation` +
`output`) input to the **source law's output unit** тАФ so a mismatch between what
a law expects and what the referenced law actually produces is caught тАФ is
**deferred to follow-up** (it needs a multi-law/registry view). Until then a
sourced input must declare its own `type_spec.unit` locally to participate in
checking (as `wet_op_de_zorgtoeslag` does). This is a missed-check gap, not a
correctness risk: an un-resolved unit is `Unknown` and never errors.

### 5. Schema (`schema/v0.5.3/`)

A new schema version (versions are immutable; existing laws stay on v0.5.2):

- **`definitions` becomes structured.** Each constant may optionally carry a `type` +
  `type_spec`, reusing the existing `baseField` shape that inputs/outputs already use.
  The simple form (`naam: 123`) remains valid for backward-compatibility.
- **`type_spec.unit` enum extended** to
  `["eurocent", "euro", "ratio", "percentage", "count", "years", "months", "weeks", "days"]`.

### 6. Rust types

- `Definition::Structured` gains `type_spec: Option<TypeSpec>` (`packages/engine/src/article.rs`).
  `TypeSpec` stays the carrier тАФ no new parser.
- New `EngineError::UnitMismatch { operation, left, right, message }` (`packages/engine/src/error.rs`),
  surfaced with article/construct context through the existing `TracedError`.

### 7. Reporting

- **Static** (alongside schema output):
  - known, conflicting units тЖТ `FAIL: тАж unit mismatch in ADD: eurocent + days`
  - amount/percentage without a unit тЖТ `WARN: тАж missing unit on amount 'drempelinkomen_alleenstaande'`
    (does not break CI)
- **Runtime**: `EngineError::UnitMismatch` propagates like any other engine error.

### 8. Rollout

- Schema v0.5.3 + engine change land in one PR. The corpus is untouched: every value is
  `Unknown`, so there are no new errors or warnings until a law is annotated.
- Laws are then annotated per-law in follow-up PRs, starting with `wet_op_de_zorgtoeslag`
  as the reference.

## Why

### Benefits

- A value can no longer be silently misinterpreted (euro vs eurocent, ratio vs percentage)
  once its field is annotated тАФ caught both before runtime (validation) and at runtime
  (hard error), so a wrong benefit amount cannot be paid out.
- One source of truth (`combine` + `infer_unit`) shared by the validator and the engine;
  static and runtime checks cannot diverge.
- Fully backward-compatible: `Unknown` units never fail, so adoption is gradual and per-law.
- Builds on the existing `type_spec.unit` field and the existing `validate` flow rather
  than adding a parallel mechanism (DRY).

### Tradeoffs

- The runtime check sits in the engine core (`evaluate_action` in `engine.rs`, ahead of
  the `operations.rs` dispatch). Mitigated by `infer_unit` being value-independent and run per action.
- The simplified algebra rejects combinations that produce compound units rather than
  modelling them. Accepted: such cases are rare and arguably should be modelled explicitly.
- `ratio` and `percentage` both remaining valid leaves two conventions in the corpus.
  Accepted to avoid a corpus-wide rewrite; a future RFC may pick a canonical form.

### Alternatives Considered

**Alternative 1: Static validation only (no runtime check)**
- The validator infers and checks units; the engine stays unit-blind.
- Rejected: cross-law inputs derive their unit from another law's output at runtime, which
  the local static check cannot always see; and with warnings-only for missing units, gaps
  slip through. A runtime hard error is the defense-in-depth backstop.

**Alternative 2: Runtime enforcement only**
- Units checked only during execution.
- Rejected: misses the chance to catch mistakes before deployment, and units are
  fundamentally a static property of fields that should be checkable without executing.

**Alternative 3: Full dimensional algebra with compound units**
- Track derived units like `eurocent/maand` through every operation.
- Rejected for now: significantly more complex for little real-world benefit; the
  simplified rules cover the actual patterns in the corpus (amount ├Ч percentage, amount ├╖ count).

**Alternative 4: Units as metadata/annotation only (no enforcement)**
- Annotate definitions purely for human and tooling clarity.
- Rejected: does not prevent actual calculation errors, which is the point of the request.

### Implementation Notes

- The unit rules live exclusively in `packages/engine/src/units.rs`. Neither the validator
  nor `operations.rs` should re-implement any rule.
- Tests: unit tests on `combine()` (all dimension pairs) and `infer_unit()` (nested
  expressions, `IF` branches, cross-law source); a BDD scenario with a law that deliberately
  adds `eurocent + days` expecting `UnitMismatch`; and `wet_op_de_zorgtoeslag` fully annotated
  as a smoke test that a correct law stays green.
