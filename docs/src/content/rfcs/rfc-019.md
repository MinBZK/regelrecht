title: "RFC-019: Law End Dates (valid_to) and Expired-Reference Handling"
---

**Status:** Draft
**Date:** 2026-06-08
**Authors:** Tim de Jager
**Depends on:** RFC-001 (YAML Schema), RFC-003 (IoC for Delegated Legislation), RFC-007 (Cross-Law Execution Model)

## Context

A law version in regelrecht carries a `valid_from` (optional) and a `publication_date`
([RFC-001](/rfcs/rfc-001) §5). There is **no end date**, and version selection
([RFC-003](/rfcs/rfc-003), "Temporal model for open term resolution") filters on the **lower bound
only** (`valid_from <= calculation_date`, newest wins). The consequence: **the most recent version
of any law is treated as valid indefinitely.** A law can only "end" by being superseded — never by
expiring or being repealed.

This is wrong for real Dutch law, where laws can end **without a successor**:

- **Horizonbepaling.** The Tijdelijke wet maatregelen covid-19 (BWBR0044337) had a sunset clause
  (art. VIII lid 1: provisions *vervallen* three months after entry into force unless extended; lid 3
  governs extension). When the fifth extension was rejected, the measures lapsed (19 mei 2022) with
  **no successor**.
- **Repeal devolved to another layer.** The Wet op de openluchtrecreatie (BWBR0006548) was
  *gefaseerd ingetrokken* per 1 januari 2008; responsibility moved to municipalities, with **no
  national successor** ([Eerste Kamer 29.829](https://www.eerstekamer.nl/wetsvoorstel/29829_gefaseerde_intrekking_wet_op)).

**Dependents break too ("verweesde regelingen").** When a delegating regulation is repealed, the
regulations resting on it lapse *van rechtswege* (Aanwijzing 243 / 6.24). The Tijdelijke regeling
maatregelen covid-19 (BWBR0044416) lost its grondslag when the Twm lapsed and expired one day later.
In regelrecht's `source.regulation` model ([RFC-007](/rfcs/rfc-007), [RFC-001](/rfcs/rfc-001) §6)
this is a dangling cross-law reference: the engine would still return the repealed law and compute on
no-longer-valid law — a **silent wrong answer**.

This RFC adds the end date and the basic handling of a reference to an ended law. A reference to a
*changed or ended* law is not always dead — eerbiedigende werking or a statische verwijzing can keep
an earlier version applicable. That nuance, and resolving a reference at a date other than the global
`calculation_date`, is the subject of **RFC-020 (date-aware reference resolution)**. RFC-019 covers
only the case at the global calculation date.

### Scope note: instrument validity ≠ substantive scope

`valid_from`/`valid_to` describe **instrument validity** — when the *text* is in force. This is
distinct from the **substantive scope** a norm governs (e.g. a 2023-only price cap), which is
expressed in the rule's own conditions/parameters and field `temporal` metadata
([RFC-001](/rfcs/rfc-001) §8). RFC-019 only adds instrument validity; it does not infer substantive
scope.

## Decision

### 1. `valid_to`: instrument end date

Add an optional law-level `valid_to`, mirroring `valid_from` ([RFC-001](/rfcs/rfc-001) §5):

```yaml
$id: tijdelijke_wet_maatregelen_covid19
bwb_id: BWBR0044337
valid_from: '2020-12-01'
valid_to: '2022-05-18'    # art. VIII: vervalt met ingang van 19 mei 2022 → laatst van kracht 18 mei
```

**Inclusive bound + "met ingang van" conversion.** `valid_to` is the **last calendar day on which
the law is in force** (inclusive), consistent with `valid_from` and with BWB's `einddatum`
("geldend t/m"). A statutory clause "*vervalt met ingang van D*" is therefore **exclusive** and maps
to `valid_to = D − 1`.

**Semantics.** `valid_to` is set **only** when the law genuinely ends with no successor — a
horizonbepaling or a repeal by another instrument. It is **not** set merely because a newer version
exists: consecutive versions still end implicitly via the newest-wins rule ([RFC-003](/rfcs/rfc-003)).
Setting `valid_to` on every version is rejected (see Alternatives).

**Harvester signal.** BWB publishes an `einddatum` per consolidation, but for ordinary succession
that is merely the day before the next consolidation starts. The genuine-end signal is **the final
consolidation (no later one exists) carrying a finite `einddatum` ≠ the `9999-12-31` sentinel**. The
harvester reads `einddatum` today and discards it; it should write `valid_to` exactly in that case.
(Origin/main PR #764 already skips laws with no consolidated text, so the harvester is the right
layer.)

### 2. Selection gains an upper bound

Extend [RFC-003](/rfcs/rfc-003)'s temporal filter. The rule, precisely ordered:

1. Among versions with `valid_from <= calculation_date`, select the newest (unchanged).
2. If the selected version has a `valid_to` and `calculation_date > valid_to`, the law is **not in
   force** → return nothing. **Do not fall through to an older version** — a repealed law does not
   resurrect a prior version.

In force iff `valid_from <= calculation_date <= valid_to` (inclusive). Additive: laws without
`valid_to` behave exactly as today.

### 3. Expired references: behaviour and honest diagnostics

A cross-law reference whose target is not in force at the `calculation_date` resolves to nothing, so
the dependent does not silently compute on dead law. The engine must distinguish *why* and **report
only what the data proves** ([RFC-015](/rfcs/rfc-015): the engine must not invent legal conclusions):

| Cause | Trace fact (the engine may state) | Must NOT state |
|---|---|---|
| `$id` absent from corpus | "regeling `<id>` niet gevonden" | "geen grondslag" |
| all versions `valid_from > calculation_date` | "geen versie van `<id>` van kracht op `<datum>` (nog niet in werking)" | — |
| selected version `valid_to < calculation_date` | "geen versie van `<id>` van kracht op `<datum>`; laatst geldig t/m `<valid_to>`" | "vervallen zonder opvolger" |

We never assert "geen grondslag / zonder opvolger": eerbiedigende werking, a statische verwijzing, or
an alternative grondslag may keep the law alive (→ RFC-020). The resolver therefore exposes the
*reason* (`NotFound` / `NotYetInForce` / `EndedOn(date)`) and the `CrossLawReference` `PathNode`
([RFC-007](/rfcs/rfc-007) trace types; [RFC-013](/rfcs/rfc-013) per-output provenance) records the
corresponding fact.

**Fatal vs. continue.** Consistent with [RFC-012](/rfcs/rfc-012) (fail-fast default, opt-in
tolerance), an unresolved required reference is an error by default. A reference (or open term, cf.
`required: false` in [RFC-003](/rfcs/rfc-003)) may be declared optional, in which case an ended/
missing target yields a null/taint that propagates like [RFC-012](/rfcs/rfc-012)'s `UNTRANSLATABLE`,
and a downstream requirement decides the outcome ("geen recht") instead of crashing.

### 4. Receipt and conformance

- The execution receipt's `loaded_regulations` scope ([RFC-013](/rfcs/rfc-013)) gains `valid_to`
  alongside the existing `valid_from`, recording the full validity window.
- A `valid_to` filtering test joins the **Temporal** conformance level ([RFC-014](/rfcs/rfc-014)),
  which already covers `valid_from` filtering.

## Worked examples

### A. `valid_to` + upper bound — Prijsplafond energie 2023 (CEK23, BWBR0047628)

The price-cap calculation (art. 2.2 plafondtarief, art. 2.3 volumegrens) with an instrument end date
(art. 8.1 lid 2). Tariffs are the regulation's own figures (supply component, excl. energiebelasting):

```yaml
$id: subsidieregeling_plafond_energietarieven_kleinverbruikers_2023
regulatory_layer: MINISTERIELE_REGELING
bwb_id: BWBR0047628
valid_from: '2022-12-15'
valid_to: '2026-12-31'          # art. 8.1 lid 2: "vervalt met ingang van 1 januari 2027"
                                # (exclusive) → laatst van kracht 31 december 2026 (inclusive)
articles:
  - number: '2.2'               # plafondtarief
    machine_readable:
      execution:
        parameters:
          - { name: verbruik_kwh, type: number, required: true }
          - { name: contracttarief_kwh, type: amount, required: true }
        output:
          - name: korting_elektriciteit
            type: amount
        actions:
          # korting = verbruik tot de volumegrens (art 2.3) × (contracttarief − plafondtarief), ≥ 0
          - output: korting_elektriciteit
            value:
              operation: MAX
              values:
                - 0
                - operation: MULTIPLY
                  values:
                    - operation: MIN
                      values: [$verbruik_kwh, 2900]          # art. 2.3 volumegrens
                    - operation: SUBTRACT
                      values: [$contracttarief_kwh, 0.24755]  # art. 2.2 plafondtarief
```

`calculation_date` in `[2022-12-15, 2026-12-31]` resolves the regeling; `2027-01-01` or later returns
nothing (instrument expired). (The price cap also has a *substantive* scope of 2023 only; per the
scope note, that is modeled in the rule, not via `valid_to`.)

### B. Genuine end without successor — Twm horizonbepaling

```yaml
$id: tijdelijke_wet_maatregelen_covid19
valid_from: '2020-12-01'
valid_to: '2022-05-18'    # art. VIII: vervalt met ingang van 19 mei 2022 → laatst van kracht 18 mei
```

A `source.regulation` reference to this law for a `calculation_date >= 2022-05-19` resolves to
nothing (§2), and the trace states the fact, not a verdict (§3).

### C. Expired dependent — orphaned reference

A regulation R reads an output from law B via `source: { regulation: B, output: … }`. With B carrying
`valid_to`, a calculation at a date after B's end resolves B to nothing → R yields no outcome, and the
`CrossLawReference` node records "geen versie van B van kracht op `<datum>`, laatst geldig t/m
`<valid_to>`" — instead of computing on the repealed B.

## Testing

Extends the existing harness rather than inventing one. `features/bezwaartermijn.feature` already runs
a cross-law/date chain on a BESCHIKKING. New cucumber-rs scenarios:

1. *valid_to upper bound*: a law with `valid_to` resolves before the end date and returns nothing
   after it (no fall-through to an older version).
2. *expired cross-reference*: dependent R references ended law B; at a `calculation_date` after B's
   `valid_to`, R yields no outcome and the trace states the fact (§3).

Plus a `valid_to` filtering case in the **Temporal** conformance level ([RFC-014](/rfcs/rfc-014)).

## Why

### Benefits

- **Expresses laws that end.** Repealed/expired laws (Twm, CEK23, WOR) become representable; the
  engine stops treating the latest version as eternal.
- **No silent wrong answers.** A reference to an ended law surfaces honestly instead of computing on
  no-longer-valid law.
- **Small and additive.** One field + an upper bound + an honest diagnostic. Laws without `valid_to`
  are unchanged; date-aware resolution is deferred to RFC-020.

### Tradeoffs

- **Authoring discipline.** `valid_to` must be set *only* for genuine ends, mirroring RFC-003's
  existing convention that implementing regulations carry an explicit `valid_from`.
- **Receipt/conformance churn**: receipt scope and the Temporal level need the new field and a case.

### Alternatives Considered

**`valid_to` on every version (mirroring the next `valid_from`).** Rejected: redundant with
newest-wins selection and easy to make inconsistent. End dates carry meaning only for genuine
termination.

**Engine infers "repealed/no successor" and reports it.** Rejected: violates
[RFC-015](/rfcs/rfc-015) (zero domain knowledge) and is unprovable — eerbiedigende werking or an
alternative grondslag may keep the law alive. The engine reports data facts only.

**Fall through to an older version once the latest has ended.** Rejected: a repealed law does not
resurrect a prior version; if an earlier version still applies, that is the date-aware case in
RFC-020, expressed explicitly — not an implicit fallback.

### Implementation Notes

- Schema: additive `valid_to` (law level); new minor schema version. Non-breaking.
- Engine: `ArticleBasedLaw.valid_to: Option<String>`; extend `select_version_for_date` with the
  inclusive upper bound and the no-fall-through rule; resolver returns a typed reason
  (`NotFound` / `NotYetInForce` / `EndedOn`).
- Harvester: write `valid_to` from BWB `einddatum` only for the final consolidation when it is finite
  (≠ `9999-12-31`).
