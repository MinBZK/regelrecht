**Status:** Draft
**Date:** 2026-06-08
**Authors:** Tim de Jager
**Depends on:** RFC-001 (YAML Schema), RFC-003 (IoC for Delegated Legislation), RFC-007 (Cross-Law Execution Model), RFC-008 (Awb Administrative Procedures)

## Context

A law version in regelrecht carries a `valid_from` (optional) and a `publication_date`
([RFC-001](/rfcs/rfc-001) §5). There is **no end date**, and version selection
([RFC-003](/rfcs/rfc-003), "Temporal model for open term resolution") filters on the **lower bound
only** (`valid_from <= calculation_date`, newest wins). The consequence: **the most recent version
of any law is treated as valid indefinitely.** A law can only "end" by being superseded — never by
expiring or being repealed. This RFC closes that gap and the date-resolution problems around it.

This is not a hypothetical. Dutch law routinely ends without a successor, and the underlying
mechanisms are codified in the Aanwijzingen voor de regelgeving.

### Real cases this cannot express today

**1. Horizonbepaling — a law that expires by design.** The Tijdelijke wet maatregelen covid-19
(BWBR0044337) contained a sunset clause: artikel VIII lid 1 — its provisions *vervallen* three
months after entry into force unless extended; lid 3 governs extension. When the Eerste Kamer
rejected the fifth extension, the measures lapsed on **19 mei 2022** with **no successor**. The
*horizonbepaling* instrument and the cleanup guidance for spent regulations are codified
([Aanwijzing 6.24/6.25](https://www.kcbr.nl/beleid-en-regelgeving-ontwikkelen/aanwijzingen-voor-de-regelgeving/hoofdstuk-6-wijziging-en-intrekking-van-regelingen/ss-63-intrekken-en-vervallen-van-regelingen/aanwijzing-624-vervallen-uitvoeringsregelingen)).
Modeled today, this law would answer queries for 2023 as if still in force.

**2. Repeal devolved to another layer.** The Wet op de openluchtrecreatie (BWBR0006548) was
*gefaseerd ingetrokken* per 1 januari 2008; responsibility moved to municipalities, with **no
national successor**
([Eerste Kamer 29.829](https://www.eerstekamer.nl/wetsvoorstel/29829_gefaseerde_intrekking_wet_op)).

**3. Orphaned dependents ("verweesde regelingen").** When a delegating regulation is repealed, the
regulations resting on it lapse *van rechtswege*, **unless** a new basis is created or a basis
exists in another regulation
([Aanwijzing 243 / 6.24](https://www.kcbr.nl/beleid-en-regelgeving-ontwikkelen/aanwijzingen-voor-de-regelgeving/hoofdstuk-6-wijziging-en-intrekking-van-regelingen/ss-63-intrekken-en-vervallen-van-regelingen/aanwijzing-624-vervallen-uitvoeringsregelingen)).
The Tijdelijke regeling maatregelen covid-19 (BWBR0044416) lost its grondslag when the Twm lapsed
and expired one day later. In regelrecht's `source.regulation` model
([RFC-007](/rfcs/rfc-007), [RFC-001](/rfcs/rfc-001) §6) this is a dangling cross-law reference: the
engine would still return the repealed law and compute on no-longer-valid law — a silent wrong
answer.

### A "missing" reference is not legally the same as "no grondslag"

Three codified mechanisms mean that a reference to a law that has *changed or ended* does **not**
automatically break — evaluation can, and often must, continue against an **earlier** version:

- **Eerbiedigende werking / overgangsrecht**
  ([Aanwijzing 5.64](https://www.kcbr.nl/beleid-en-regelgeving-ontwikkelen/aanwijzingen-voor-de-regelgeving/hoofdstuk-5-bijzondere-bestanddelen-van-regelingen/ss-514-overgangsrecht/aanwijzing-564-eerbiedigende-en-uitgestelde-werking)):
  the old law stays applicable to existing cases and pending procedures.
- **Statische vs. dynamische verwijzing**
  ([Aanwijzing 3.47](https://www.kcbr.nl/beleid-en-regelgeving-ontwikkelen/aanwijzingen-voor-de-regelgeving/hoofdstuk-3-aspecten-van-vormgeving-31-364/ss-33-aanhaling-en-verwijzing-327-351/aanwijzing-347-dynamische-en-statische-verwijzing)):
  a *static* reference points to the text "zoals dat luidde op <datum>" and survives repeal; a
  *dynamic* reference follows the current text. Dynamic is the hoofdregel; static is the exception,
  typically in transitional law.
- **Alternatieve grondslag** ([Aanwijzing 243](https://www.kcbr.nl/beleid-en-regelgeving-ontwikkelen/aanwijzingen-voor-de-regelgeving/hoofdstuk-6-wijziging-en-intrekking-van-regelingen/ss-63-intrekken-en-vervallen-van-regelingen/aanwijzing-624-vervallen-uitvoeringsregelingen)):
  a second regulation can carry the basis.

### What the existing RFCs already give us

This RFC deliberately builds on, and does not duplicate, accepted decisions:

| Already decided | Where | Used here for |
|---|---|---|
| `valid_from` + `publication_date` metadata | [RFC-001](/rfcs/rfc-001) §5 | start of validity |
| Field-level temporal metadata (`type: period`, `period_type`, `reference`) | [RFC-001](/rfcs/rfc-001) §8 | **substantive** temporal scope (distinct from instrument validity) |
| `regelrecht://` / `source.regulation` references | [RFC-001](/rfcs/rfc-001) §6, [RFC-007](/rfcs/rfc-007) | the references that can dangle |
| `valid_from <= calculation_date` selection, newest-wins; lex superior → lex posterior priority | [RFC-003](/rfcs/rfc-003) | the filter we extend with an upper bound |
| Date types & arithmetic (`DATE`, `DATE_ADD`, `DAY_OF_WEEK`, `AGE`) | [RFC-007](/rfcs/rfc-007) §3 | computing an applicable date |
| **"The lifecycle is law"**; multiple event dates per case (`aanvraag_datum`, `besluit_datum`, `bekendmaking_datum`); heroverweging (Awb 7:11) modeled as ex nunc | [RFC-008](/rfcs/rfc-008) | the legal source of *which date applies* |
| Multiple sources per `$id`, priority number, tie = load error | [RFC-010](/rfcs/rfc-010) | alternative grondslag |
| `UNTRANSLATABLE` taint propagates like `NaN`; fail-fast default, opt-in tolerance | [RFC-012](/rfcs/rfc-012) | precedent for "continue with a marked value" |
| Per-output provenance; `valid_from` in the execution receipt scope | [RFC-013](/rfcs/rfc-013) | conservative failure reporting; receipt gains `valid_to` |
| Zero built-in domain knowledge | [RFC-015](/rfcs/rfc-015) | the engine must not *infer* "repealed/no successor" |

The gap is narrow and specific: **(1)** no end date and no upper bound in selection; **(2)** a
reference cannot be resolved at a date *different from the global `calculation_date`* and have that
date propagate; **(3)** no defined behaviour or honest diagnostic when a referenced law has ended.

## Decision

### 1. Three distinct temporal axes

Conflating these is the root bug. The RFC names them explicitly:

| Axis | Question it answers | Where it lives |
|---|---|---|
| **Instrument validity** | When is this *text* in force? | `valid_from` / **`valid_to`** (law-level) |
| **Substantive scope** | Which period/facts does the norm *govern*? | field `temporal` metadata ([RFC-001](/rfcs/rfc-001) §8), parameters, scope |
| **Applicable-date selection** | *Which version-date* do we evaluate against? | computed by law (Awb/overgangsrecht), passed per reference |

The Subsidieregeling bekostiging plafond energietarieven kleinverbruikers 2023 (CEK23, BWBR0047628)
shows all three at once, and is the worked example throughout:

- **Instrument validity**: art. 8.1 lid 2 "*vervalt met ingang van 1 januari 2027*" → `valid_to: '2026-12-31'`.
- **Substantive scope**: the price cap governs **calendar year 2023** only.
- **Applicable-date selection**: a dependent computing a 2023 benefit must resolve the cap *as it applied in 2023*.

### 2. `valid_to`: instrument end date

Add an optional law-level `valid_to`, mirroring `valid_from` ([RFC-001](/rfcs/rfc-001) §5). The
worked example is the Twm horizonbepaling in §B below.

**Inclusive bound + "met ingang van" conversion.** `valid_to` is the **last calendar day on which
the law is in force** (inclusive), consistent with `valid_from` and with BWB's `einddatum`
("geldend t/m"). A statutory clause "*vervalt met ingang van D*" is therefore **exclusive** and maps
to `valid_to = D − 1`. (Example A below: CEK23 art. 8.1 lid 2 "vervalt met ingang van 1 januari
2027" → `valid_to: '2026-12-31'`.)

**Semantics (faithful to the user's rule).** `valid_to` is set **only** when the law genuinely ends
with no successor — a horizonbepaling (Twm art. VIII) or a repeal by another instrument. It is
**not** set merely because a newer version exists: consecutive versions still end implicitly via the
newest-wins rule ([RFC-003](/rfcs/rfc-003)). Setting `valid_to` on every version would be redundant
and is rejected (see Alternatives).

**Harvester signal.** BWB publishes an `einddatum` per consolidation, but for ordinary succession
that is merely the day before the next consolidation starts. The genuine-end signal is **the final
consolidation (no later one exists) carrying a finite `einddatum` ≠ the `9999-12-31` sentinel**. The
harvester reads `einddatum` today and discards it; it should write `valid_to` exactly in that case.
(Origin/main PR #764 already skips laws with no consolidated text — withdrawn/future/announced — so
the harvester is the right layer for this.)

### 3. Selection gains an upper bound

Extend [RFC-003](/rfcs/rfc-003)'s temporal filter. The rule, precisely ordered:

1. Among versions with `valid_from <= as_of`, select the newest (unchanged from RFC-003).
2. If the selected version has a `valid_to` and `as_of > valid_to`, the law is **not in force** →
   return nothing. **Do not fall through to an older version** — a repealed law does not resurrect a
   prior version.

The bound is inclusive (see §2): in force iff `valid_from <= as_of <= valid_to`. `as_of` is the
applicable date for *this* resolution (§4), defaulting to the global `calculation_date`.

This is additive: laws without `valid_to` behave exactly as today. It also slots into the existing
priority resolution — an ended candidate drops out *before* lex superior/lex posterior
([RFC-003](/rfcs/rfc-003)), so a co-existing alternative grondslag is chosen automatically (scenario
*c*; mechanism reused, not reinvented).

### 4. Date-aware references: `as_of` that propagates

Today a single global `calculation_date` selects every version. But *which* version-date applies is
a **legal** question whose answer can differ per reference and must flow down the subtree beneath it.
We add an optional `as_of` to a cross-law reference / open-term resolution:

```yaml
input:
  - name: standaardpremie
    source:
      regulation: regeling_standaardpremie
      output: standaardpremie
      as_of: $besluit_datum        # resolve the referenced law as it stood on this date
```

- **Default = dynamic** (no `as_of`): follows the global `calculation_date`. This is the hoofdregel
  ([Aanwijzing 3.47](https://www.kcbr.nl/beleid-en-regelgeving-ontwikkelen/aanwijzingen-voor-de-regelgeving/hoofdstuk-3-aspecten-van-vormgeving-31-364/ss-33-aanhaling-en-verwijzing-327-351/aanwijzing-347-dynamische-en-statische-verwijzing))
  and matches current behaviour.
- **`as_of: <date or $param>` = static / ex tunc / eerbiedigend**: resolves the referenced law (and
  everything beneath that reference) at the given date. A literal date models a *statische
  verwijzing* ("zoals dat luidde op <datum>"); a parameter (e.g. `$besluit_datum`) models ex tunc /
  eerbiedigende werking.

**The applicable date is itself computed by law, not by engine heuristic** ([RFC-015](/rfcs/rfc-015)
zero domain knowledge). [RFC-008](/rfcs/rfc-008) already establishes the Awb lifecycle as law and
carries the relevant event dates (`aanvraag_datum`, `besluit_datum`, `bekendmaking_datum`) as stage
inputs. ex nunc is the default at heroverweging (Awb 7:11 lid 1: *"Indien het bezwaar ontvankelijk
is, vindt op grondslag daarvan een heroverweging van het bestreden besluit plaats."* — RFC-008
models this as ex nunc); ex tunc / eerbiedigende werking are the exceptions, expressed by passing an
earlier event date as `as_of`. RFC-019 refines RFC-008's default; it does not contradict it.

### 5. Ended / unresolved references: behaviour and honest diagnostics

When a reference resolves to nothing, the engine must distinguish *why* and must **report only what
the data proves** ([RFC-015](/rfcs/rfc-015): the engine must not invent legal conclusions):

| Cause | Trace fact (the engine may state) | Must NOT state |
|---|---|---|
| `$id` absent from corpus | "regeling `<id>` niet gevonden" | "geen grondslag" |
| all versions `valid_from > as_of` | "geen versie van `<id>` van kracht op `<as_of>` (nog niet in werking)" | — |
| selected version `valid_to < as_of` | "geen versie van `<id>` van kracht op `<as_of>`; laatst geldig t/m `<valid_to>`" | "vervallen zonder opvolger" |

We never assert "geen grondslag / zonder opvolger": overgangsrecht, a static reference, or an
alternative grondslag may keep the law alive — facts the engine cannot know. The resolver therefore
exposes the *reason* (e.g. `NotFound` / `NotYetInForce` / `EndedOn(date)`) and the
`CrossLawReference` `PathNode` ([RFC-007](/rfcs/rfc-007) trace types; [RFC-013](/rfcs/rfc-013)
per-output provenance) records the corresponding fact.

**Fatal vs. continue.** Consistent with [RFC-012](/rfcs/rfc-012) (fail-fast default, opt-in
tolerance), an unresolved required reference is an error by default. A reference (or open term, cf.
`required: false` in [RFC-003](/rfcs/rfc-003)) may be declared optional, in which case an ended/
missing target yields a null/taint that propagates like [RFC-012](/rfcs/rfc-012)'s `UNTRANSLATABLE`,
and a downstream requirement decides the outcome ("geen recht") instead of crashing.

### 6. Receipt and conformance

- The execution receipt's `loaded_regulations` scope ([RFC-013](/rfcs/rfc-013)) gains `valid_to`
  alongside the existing `valid_from`, so a sealed receipt records the full validity window.
- A `valid_to` filtering test and date-aware (`as_of`) resolution tests join the **Temporal**
  conformance level ([RFC-014](/rfcs/rfc-014)), which already covers `valid_from` filtering.

## Worked examples

Each mechanism below is grounded in a real regulation and shown as YAML consistent with the existing
corpus (e.g. `corpus/regulation/nl/wet/algemene_wet_bestuursrecht/1994-01-01.yaml`,
`corpus/regulation/nl/ministeriele_regeling/regeling_standaardpremie/`).

### A. `valid_to` + upper bound — Prijsplafond energie 2023 (CEK23, BWBR0047628)

The price-cap calculation (art. 2.2 plafondtarief, art. 2.3 volumegrens) as an executable regeling
with an instrument end date (art. 8.1 lid 2). Tariffs are the regulation's own figures (supply
component, excl. energiebelasting):

```yaml
$id: subsidieregeling_plafond_energietarieven_kleinverbruikers_2023
regulatory_layer: MINISTERIELE_REGELING
bwb_id: BWBR0047628
publication_date: '2022-12-15'
valid_from: '2022-12-15'
valid_to: '2026-12-31'          # art. 8.1 lid 2: "vervalt met ingang van 1 januari 2027"
                                # (exclusive) → laatst van kracht 31 december 2026 (inclusive)
articles:
  - number: '2.2'               # plafondtarief
    text: |-
      Het maximale tarief bedraagt € 0,24755 per kWh elektriciteit
      en € 0,85734 per m3 gas.
    machine_readable:
      execution:
        parameters:
          - { name: verbruik_kwh, type: number, required: true }
          - { name: contracttarief_kwh, type: amount, required: true }
        output:
          - name: korting_elektriciteit
            type: amount
            temporal: { type: period, period_type: year }   # substantive scope: het jaar 2023
        actions:
          # korting = verbruik tot de volumegrens (art 2.3) × (contracttarief − plafondtarief),
          # nooit negatief
          - output: korting_elektriciteit
            value:
              operation: MAX
              values:
                - 0
                - operation: MULTIPLY
                  values:
                    - operation: MIN
                      values: [$verbruik_kwh, 2900]   # art. 2.3 volumegrens
                    - operation: SUBTRACT
                      values: [$contracttarief_kwh, 0.24755]   # art. 2.2 plafondtarief
```

Selection behaviour (§3): `as_of` in `[2022-12-15, 2026-12-31]` resolves the regeling; `as_of` of
`2027-01-01` or later returns nothing. (Axis-separation, per §1: the *instrument* lives until 2027,
but the *substantive* cap governs only 2023 via the `temporal` metadata.)

### B. Genuine end without successor — Twm horizonbepaling

```yaml
$id: tijdelijke_wet_maatregelen_covid19
bwb_id: BWBR0044337
valid_from: '2020-12-01'
valid_to: '2022-05-18'    # art. VIII: vervalt met ingang van 19 mei 2022 → laatst van kracht 18 mei
```

Any `source.regulation` reference to this law with `as_of >= 2022-05-19` resolves to nothing (§3),
and the trace states the fact, not a verdict (§5).

### C. Ex nunc vs. ex tunc — Awb 7:11 heroverweging (builds on RFC-008)

The `bezwaartermijn` chain is already modeled and tested
(`corpus/.../algemene_wet_bestuursrecht/1994-01-01.yaml`, `features/bezwaartermijn.feature`). The
heroverweging on objection (Awb 7:11) re-evaluates the original decision. Whether it applies the law
of the *besluit* moment (ex tunc) or of the *beslissing op bezwaar* moment (ex nunc) is selected by
which event date is passed as `as_of`:

```yaml
# Default (ex nunc): heroverweging follows current law — no as_of, uses calculation_date
input:
  - name: oorspronkelijk_recht
    source: { regulation: vreemdelingenwet_2000, output: verblijfsvergunning_verleend }

# Exception (ex tunc / eerbiedigende werking): evaluate the law as it stood at the besluit moment
input:
  - name: oorspronkelijk_recht
    source:
      regulation: vreemdelingenwet_2000
      output: verblijfsvergunning_verleend
      as_of: $besluit_datum     # RFC-008 stage input; propagates through the referenced subtree
```

### D. Statische verwijzing — frozen text by date

```yaml
# "... de Wet X, zoals die luidde op 31 december 2000 ..."
input:
  - name: oude_definitie
    source:
      regulation: wet_x
      output: definitie
      as_of: '2000-12-31'   # static: survives later repeal of wet_x; date propagates down
```

### E. Alternatieve grondslag — reuse of existing priority

No new mechanism. Two regulations declaring `implements` for the same open term
([RFC-003](/rfcs/rfc-003)), or two sources for the same `$id` ([RFC-010](/rfcs/rfc-010)), are already
priority-resolved. With §3, an ended candidate (its `valid_to < as_of`) drops out of the temporal
filter before priority resolution, so the surviving grondslag is selected — matching Aanwijzing 243
("tenzij … een grondslag bestaat in een andere regeling").

## Testing

Yes — RFC-008's material is already exercised, and RFC-019 extends that harness rather than inventing
one:

- **Existing**: `features/bezwaartermijn.feature` runs the Awb hook/override/date chain on a
  BESCHIKKING at `calculation date "2026-01-01"`. The Awb lifecycle, hooks (3:46, 6:7, 6:8) and the
  Vreemdelingenwet art. 69 override are modeled in the corpus.
- **New BDD scenarios** (same cucumber-rs style):
  1. *valid_to upper bound*: a law with `valid_to` resolves before the end date and returns nothing
     after it (no fall-through to an older version).
  2. *ended cross-reference*: dependent A references ended law B; at `as_of` after B's `valid_to`,
     A yields no outcome and the trace node states "geen versie van B van kracht op `<as_of>`,
     laatst geldig t/m `<valid_to>`".
  3. *ex nunc vs ex tunc*: the same heroverweging reference resolves to different versions for
     `as_of: $besluit_datum` vs the default, when two versions of the target law are loaded.
  4. *statische verwijzing*: `as_of: '<fixed date>'` keeps resolving after the target's `valid_to`.
- **Conformance**: add `valid_to` filtering and `as_of` resolution cases to the **Temporal** level
  ([RFC-014](/rfcs/rfc-014)).

## Why

### Benefits

- **Expresses laws that end.** Repealed/expired laws (Twm, CEK23, WOR) become representable; the
  engine stops treating the latest version as eternal.
- **Legally faithful references.** ex nunc/ex tunc, statische verwijzing and eerbiedigende werking
  are modeled by *which date* a reference resolves at — grounded in the Aanwijzingen and Awb, not in
  engine heuristics ([RFC-015](/rfcs/rfc-015)).
- **No silent wrong answers.** A dangling reference to an ended law surfaces honestly instead of
  computing on no-longer-valid law.
- **Mostly reuse.** Alternative grondslag, priority, date arithmetic, taint propagation, provenance
  and the lifecycle all come from accepted RFCs; the new surface is `valid_to`, the upper bound, and
  `as_of`.

### Tradeoffs

- **`as_of` propagation** adds a second date dimension to resolution; the engine must thread an
  `as_of` (defaulting to `calculation_date`) through cross-law resolution and cycle detection.
- **Authoring discipline.** `valid_to` must be set *only* for genuine ends, mirroring RFC-003's
  existing convention that implementing regulations carry an explicit `valid_from`.
- **Receipt/conformance churn**: receipt scope and the Temporal conformance level need the new field
  and cases.

### Alternatives Considered

**`valid_to` on every version (explicit end mirroring the next `valid_from`).**
Rejected: redundant with newest-wins selection and easy to make inconsistent. End dates carry
meaning only for genuine termination.

**Engine infers "repealed/no successor" and reports it.**
Rejected: violates [RFC-015](/rfcs/rfc-015) (zero domain knowledge) and is unprovable — overgangsrecht
or an alternative grondslag may keep the law alive. The engine reports data facts only.

**A single global date for all version selection (status quo).**
Rejected: cannot express ex tunc, statische verwijzing, or eerbiedigende werking, all of which
require resolving a referenced law at a date other than the global one.

**Dates computed outside the engine.**
Rejected for the same reason [RFC-007](/rfcs/rfc-007) rejected it: the date rules (Awb, overgangsrecht)
are themselves law and should be executed, not hard-coded.

### Implementation Notes

- Schema: additive `valid_to` (law level) and `as_of` (on `source`); new minor schema version
  (constructs from [RFC-007](/rfcs/rfc-007)/[RFC-008](/rfcs/rfc-008) landed in v0.5.0). Non-breaking.
- Engine: `ArticleBasedLaw.valid_to: Option<String>`; extend `select_version_for_date` with the
  inclusive upper bound and the no-fall-through rule; thread `as_of` through `get_law_for_date` /
  `get_article_by_output` and the cross-law resolution path; resolver returns a typed reason
  (`NotFound` / `NotYetInForce` / `EndedOn`).
- Harvester: write `valid_to` from BWB `einddatum` only for the final consolidation when it is finite
  (≠ `9999-12-31`).

## Open Questions (to verify against the law)

1. **Where does the applicable-date rule actually live?** ex nunc/ex tunc is largely *jurisprudential*
   (e.g. ABRvS), and overgangsrecht usually sits in the *specific* law, not in the Awb. RFC-008 already
   labels Awb 7:11 heroverweging "ex nunc"; that label is doctrine, not statutory text. How do we
   represent a jurisprudential default faithfully without inventing it as engine behaviour?
2. **Can multiple event dates legitimately coexist in one beoordeling** (e.g. ex tunc for the merits,
   ex nunc for a procedural element)? RFC-008 already carries multiple stage dates — confirm against
   the Awb which feeds `as_of` for which sub-question.
3. **How far does `as_of` propagate** down a reference subtree, and what happens when an inner
   reference sets its own `as_of` (nested static reference inside a dynamic one)?
4. **Substantive scope vs instrument validity** — should the engine enforce that a result is only
   produced when *both* the instrument is in force and the `as_of` falls in the substantive period,
   or is that the law author's responsibility?
5. **Interaction with hooks/overrides** ([RFC-007](/rfcs/rfc-007)): hook and override candidates are
   already temporally filtered by `valid_from`; confirm `valid_to` + `as_of` apply identically there.
