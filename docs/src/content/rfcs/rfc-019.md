# RFC-019: Chronolexogram types in the schema and corpus

**Status:** Proposed
**Date:** 2026-05-26
**Authors:** Anne Schuth

## Terminology

This RFC adopts the vocabulary from the [Chronolexografie](https://chronolexografie.nl/) position paper (Olthof & Van Andel, december 2025) and the [Nieuwland — een ontwerp voor een digitale rechtsstaat](https://achterkantvandeoverheid.nl/) program (Achterkant van de Overheid, december 2025).

| Term | Working translation | Definition (paraphrased from the position paper) |
|---|---|---|
| **lexogram** | law-recording | vastlegging van een (mogelijke, toekomstige) wijziging in wet- of regelgeving |
| **decretogram** | decision-recording | vastlegging van een concreet besluit of beschikking bij toepassing van wetgeving |
| **executogram** | execution-recording | vastlegging van daadwerkelijke levering of afhandeling van een zaak of dienst |
| **chronolexocel** (cel) | cell | domein waarin chronolexogrammen worden gemaakt, bewaard en verwerkt |
| **chronolexokroniek** (kroniek) | chronicle | tijdsgeordende reeks van chronolexogrammen binnen één cel |
| **chronolexoreductie** (reductie) | reduction | inherent destructieve activiteit van hergebruik van feiten op een later moment |
| **lexostatus** | rechtstoestand-projectie | rechtstoestand zoals gezien vanuit een bepaald (gevraagd) perspectief |

## Context

The Chronolexografie position paper proposes a coherent vocabulary for how Dutch government organizations record the rechtstoestand digitally. It distinguishes three types of recording (lexogram, decretogram, executogram), and places them in cells with their own chronicles. The wider [Nieuwland](https://achterkantvandeoverheid.nl/) design embeds this in a broader picture of registreren, interpreteren, ontwerpen en normeren.

RegelRecht implements parts of this vocabulary already, without using its names. The 23 regulations in `corpus/regulation/` are de facto lexograms. A BESCHIKKING produced by an engine is a decretogram. Execution traces (RFC-013, Draft) are chronicle-like ordered records of how a decretogram came about. RFC-009 (Multi-Organisation Execution, Proposed) treats each engine as having its own identity that maps neatly onto a cell.

What is missing is the third type. RegelRecht has no concept of an executogram: the recording of a factual handling such as a payment received, a service delivered, a kwijtschelding granted. These are not regulation outputs. They are recordings of facts that a cell learns from its surrounding systems, signs, and may share with other cells. Without an explicit slot for them, integrations that need them (with claim-collection systems, payment surfaces, citizen-facing overzichten) end up putting executograms into ad hoc places, or worse, conflating them with decretograms.

This RFC introduces the three chronolexogram types as first-class concepts in the schema and the corpus layout. It also extends the `decision_type` enum so that decretograms in the financial-enforcement domain (payment obligations, criminal-law settlements, administrative fines) have proper labels.

The RFC is deliberately generic. It does not commit RegelRecht to any specific external standard for serialising chronolexograms outbound, and it does not assume any specific integration partner. Concrete integrations (with Mijn Betaaloverzicht via FCID, or any other claims standard) belong in separate integration documents that reference this RFC for their conceptual ground.

### Why this is necessary

Three pressures converge.

First, the corpus today cannot express the financial-enforcement domain. None of the current `decision_type` values (TOEKENNING, AFWIJZING, GOEDKEURING, GEEN_BESLUIT, ALGEMEEN_VERBINDEND_VOORSCHRIFT, BELEIDSREGEL, VOORBEREIDINGSBESLUIT, ANDERE_HANDELING, AANSLAG) describes a payment obligation, a strafbeschikking, or a bestuurlijke boete. Authoring Wahv, Wet OM-afdoening or sectoral boete-wetten requires new enum values.

Second, several upcoming integrations need to serialise both decretograms and executograms as part of one stream of events (Mijn Betaaloverzicht being the immediate driver, but the pattern is general). Without a separate executogram concept, the schema would need to absorb event-types that are not regulation outputs.

Third, Chronolexografie offers a vocabulary that aligns RegelRecht with the wider Nieuwland conversation. Naming the engine a cell, the trace a chronicle entry, the execution a reduction, and the output a lexostatus is not just cosmetic. It signals that RegelRecht participates in the same architectural picture that Olthof, Van Andel, Hotting and others are building, instead of describing the same things in private idiom.

### Design principles

1. **The corpus is the lexogram store.** A YAML file under `corpus/regulation/` is a lexogram. The sequence of `valid_from` versions for one regulation is its lexogram-chronicle. No new schema needed for lexograms themselves.
2. **A decretogram is an engine output with `legal_character: BESCHIKKING`.** New `decision_type` values cover the financial-enforcement domain.
3. **An executogram is not a regulation output.** It does not belong in `produces`. It gets its own top-level corpus directory and its own schema.
4. **The engine is a cell.** This is a recognition, not a new code path. RFC-009's engine identity already provides cell identity; RFC-013's traces are already chronicle-like.
5. **Outbound serialisation is integration-specific.** This RFC does not specify how a decretogram or executogram looks on the wire when crossing into a non-RegelRecht system. That is the job of an integration document.

## Decision

### 1. Three chronolexogram types in the schema and corpus

**Lexogram** — already exists.

A YAML file under `corpus/regulation/<jurisdiction>/<type>/<name>/<valid_from>.yaml` is a lexogram. The collection of versions for one regulation is its lexogram-chronicle. No new schema is needed. The recognition is documentational: existing concepts pages and the [law-format guide](/concepts/law-format) get one paragraph each calling this what it is.

**Decretogram** — engine output with `produces.legal_character: BESCHIKKING`.

The existing `BESCHIKKING` value of `legal_character` already labels engine outputs as decretograms. What is missing is `decision_type` values for the financial-enforcement domain. This RFC adds them (see §2). The execution trace from RFC-013 is the decretogram body; the engine signature from RFC-009 §5 is the decretogram signature.

**Executogram** — new top-level corpus directory.

A new corpus directory `corpus/executogram/` holds executogram-stream definitions. Each file declares: what factual events the cell records, who is the competent authority for those recordings, and (optionally) how each event maps onto an outbound integration target. The engine validates these files at startup just as it validates regulation files, but they are not regulations: they are recording specifications.

A minimal example:

```yaml
$id: payments_received
competent_authority: <organisation>
events:
  - name: payment_received
    source: external_intake
    fields:
      case_reference: $external.case_reference
      amount_cents: $external.amount_cents
      received_at: $external.received_at
```

A separate JSON schema for executogram files lives at `schema/v0.6.0/executogram.json`.

This separation respects Chronolexografie's principle that registratie and interpretatie are different acts. A regulation YAML is interpretation: rules applied to inputs producing a decision. An executogram-stream file is registratie: a specification for recording facts that the cell learns from its environment. Mixing the two in one schema would erase a juridically meaningful distinction.

### 2. `produces.decision_type` enum extension

Add the following values to `produces.decision_type` (decretograms only):

- `BETALINGSVERPLICHTING` — generic financial obligation imposed by a bestuursorgaan
- `STRAFBESCHIKKING` — criminal-law settlement under Wetboek van Strafvordering art. 257a
- `BESTUURLIJKE_BOETE` — sectoral administrative fine
- `INCASSO_BESCHIKKING` — explicit collection order, typically after non-payment of an underlying obligation
- `INTREKKING_BESCHIKKING` — withdrawal or annulment of an earlier decision

`legal_character: BESCHIKKING` already covers all of these. No new `legal_character` value is added.

These values describe decretograms only. The recording of a payment received, a kwijtschelding granted, or a service delivered is an executogram and lives in `corpus/executogram/`, not in any regulation's `produces`.

### 3. Optional integration hooks on `produces`

To support concrete integrations (without committing the schema to any specific one), two optional fields are added to `produces`:

- `outbound_emit: bool` (default `false`) — when true, the engine signals downstream integrations that this decretogram is eligible for outbound serialisation
- `outbound_category: string` (optional) — a free-form tag that integrations can use to route or classify the decretogram (for example a category in a specific outbound standard)

The schema does not constrain what `outbound_category` values mean. That is for the integration document to specify. For example, an MBO/FCID integration document would say: "FCID requires a category per line; use `outbound_category` with one of ALGEMEEN, ADMINISTRATIEKOSTEN, VERHOGING, RENTE."

This keeps the schema neutral. If another claims standard appears later with different categories, no schema change is needed.

### 4. The engine as a cell

A RegelRecht engine deployed at an organisation is, in Chronolexografie terms, a chronolexocel. The cell:

- holds a lexogram-chronicle (its loaded regulations and their versions)
- produces decretograms (engine outputs with `BESCHIKKING`-character)
- records executograms (via `corpus/executogram/` definitions)
- maintains its own traces as chronicles (RFC-013)
- signs its outputs with its FSC key (RFC-009 §5)
- exchanges lexostatuses with other cells via FSC contracts (RFC-009)

In Chronolexografie terms, an engine execution is a **chronolexoreductie**: the cell filters the relevant chronolexograms from its chronicles and combines them via the regulation's `machine_readable` logic into a **lexostatus** for a particular request. The "destructive" character of reduction is the same as RegelRecht's execution: only the subset of facts relevant to the requested perspective surfaces in the output.

This is a naming recognition, not a code change. The mapping is:

| Chronolexografie | RegelRecht today |
|---|---|
| chronolexocel | engine deployment with an identity (RFC-009 §2) |
| chronolexokroniek | per-regulation trace store |
| chronolexogram | one entry in the trace store (a decretogram) or one recorded event (an executogram) |
| chronolexoreductie | one engine execution |
| lexostatus | the output of an engine execution |
| chronolexosfeer | the federation of cells (RFC-009 federated mode) |

The intent is to use this vocabulary consistently in future documentation so that readers familiar with Chronolexografie and Nieuwland recognise the architecture immediately.

### 5. Multiple cells per organisation

In Dutch practice, a single organisation may run several cells with different competence-bases. For example, an organisation that executes both criminal-law and administrative-law tasks may want one cell per legal regime, with separate signing keys and separate chronicles. RFC-009's engine identity already supports multiple identities per deployment.

This RFC does not prescribe a cell-per-organisation mapping. Concrete deployments choose. The schema and corpus layout described here are neutral to that choice.

### 6. Out of scope

This RFC is deliberately small. It does not address:

- **Any specific outbound serialisation format** (FCID, JSON-LD, ActivityPub, etc.). Those belong in integration documents.
- **Any specific integration partner** (CJIB, Mijn Betaaloverzicht, a sectoral regulator). Those belong in integration documents and casus documents.
- **Trust and signing mechanics.** Already in RFC-009 §5.
- **Execution provenance and trace structure.** Already in RFC-013.
- **Federation transport.** Already in RFC-009 (FSC).
- **The legal basis for inter-organisation data exchange.** Per-case, not at engine level.
- **A statutory Wet gegevensboekhouding** (Nieuwland §7.3.2). That is a legislative artefact; this RFC is an engineering one.

## Why

### Benefits

- **Conceptual integrity.** Lexogram, decretogram and executogram are explicit. A future reader of any corpus YAML or executogram YAML sees the type for what it is, without inference from context.
- **The financial-enforcement domain becomes expressible.** Wahv, OM-strafbeschikking, sectoral bestuurlijke boetes can now be authored as regulations with appropriate `decision_type` values.
- **Integrations don't drag domain assumptions into the schema.** The schema stays neutral; integration documents do the mapping. New integrations don't force schema changes.
- **Alignment with the Nieuwland conversation.** Using the Chronolexografie vocabulary signals participation, not parallel reinvention.
- **`corpus/executogram/` opens a clean place for recording-specifications.** Even without any external integration, having a place for "this cell records these facts" is a missing piece in RegelRecht today.

### Tradeoffs

- **The `decision_type` enum extension is breaking for downstream consumers of the enum.** Schema bump to v0.6.0 is required.
- **Introducing `corpus/executogram/` adds a second top-level corpus category.** Tooling (validators, harvesters, the editor) needs to learn about it. The cost is one-time; the conceptual gain is permanent.
- **`outbound_emit`/`outbound_category` are deliberately vague.** They are hooks for integrations to interpret. A reader of the schema cannot know what `outbound_category: "ALGEMEEN"` means without consulting an integration document. This is the price of schema neutrality.
- **Adopting Chronolexografie vocabulary in docs requires a vocabulary shift across many pages.** This RFC accepts that as a one-time documentation cost.

### Alternatives Considered

**Alternative 1: Bake one specific outbound standard into the schema** (the first draft of this RFC bundled FCID-specific fields).
- Rejected: ties the schema to one standard's lifecycle. When that standard versions or is replaced, the schema breaks. Keeping the schema neutral and routing standard-specific logic through integration documents avoids that coupling.

**Alternative 2: Treat executograms as a kind of regulation output and conflate with `decision_type`.**
- Rejected: erases the registratie/interpretatie distinction that Chronolexografie makes central. A payment received is not a decision; calling it one for schema convenience is wrong.

**Alternative 3: Skip the executogram concept; let integrations invent it locally.**
- Rejected: each integration would invent slightly different recording-formats. A common slot in the corpus is the cheap and durable answer.

**Alternative 4: Don't adopt Chronolexografie vocabulary; describe the same architecture in private RegelRecht idiom.**
- Rejected: the vocabulary is good, it is the result of careful work by people we already collaborate with, and adopting it costs nothing technical. The alternative would be a parallel idiom for the same concepts, increasing translation friction with the wider Nieuwland conversation.

### Implementation Notes

This RFC is the architecture decision. Schema and engine work are in follow-up PRs.

#### Phase 1: Schema

- Bump `schema/v0.5.2` → `schema/v0.6.0`. Update `schema/latest` symlink.
- Add new `decision_type` enum values.
- Add `outbound_emit` and `outbound_category` optional fields under `produces`.
- Add `schema/v0.6.0/executogram.json` for executogram-stream files.

#### Phase 2: Engine support for executogram-streams

- New: `packages/engine/src/executogram.rs` — loading, validating, and triggering executogram-stream entries on external events.
- New: `ExecutogramStream` struct in `packages/engine/src/types.rs`.

#### Phase 3: Vocabulary update in existing docs

- Update [concepts/law-format](/concepts/law-format) to note that a regulation file is a lexogram.
- Update [concepts/multi-org-execution](/concepts/multi-org-execution) to note that an engine identity is a cell identity.
- Update [concepts/execution-provenance](/concepts/execution-provenance) to note that a trace is a chronicle entry.
- Add a short glossary entry for chronolexogram, lexogram, decretogram, executogram, lexostatus.

#### Phase 4: First executogram-stream and the first integration

- One executogram-stream YAML under `corpus/executogram/` as a working example.
- An integration document under `docs/integrations/` that uses the new hooks. The first integration is Mijn Betaaloverzicht / FCID; see that document for specifics.
