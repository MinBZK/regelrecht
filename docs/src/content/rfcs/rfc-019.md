# RFC-019: Chronolexogram types in the schema and the cell model

**Status:** Proposed
**Date:** 2026-05-26
**Authors:** Anne Schuth
**Depends on:** RFC-002 (Authority), RFC-007 (Cross-Law Execution), RFC-008 (AWB Procedures), RFC-009 (Multi-Org Execution), RFC-010 (Federated Corpus), RFC-013 (Execution Provenance)

## Context

RegelRecht turns Dutch law into executable specifications. The accumulated design captures concepts that scale across thousands of regulations: `legal_character: BESCHIKKING`, `decision_type`, `competent_authority` (RFC-002), Inversion of Control for delegated legislation (RFC-003), cross-law execution (RFC-007), the AWB lifecycle as a first-class construct (RFC-008), multi-organisation federation via FSC (RFC-009), federated corpus distribution (RFC-010), and chronicle-like execution provenance (RFC-013). The set of regulations currently in machine-readable form is small relative to the Dutch corpus and grows opportunistically; what matters here is that the conceptual coverage is broad enough to host very different domains under one engine.

Three things are still missing from that conceptual coverage. First, the financial-enforcement domain (Wahv, OM-strafbeschikking, sectoral bestuurlijke boetes) has no fitting `decision_type` and so cannot be expressed today. Second, there is no place in the repository to record *factual events* (payments received, kwijtscheldingen, deurwaardertrajecten); these are not regulation outputs and do not belong in `produces`. Third, there is no clean separation between **normative content** (regulations) and **registration specifications** (which facts a deployment records). Today no slot exists for the latter, so integrations either invent a local format or push the recording into a regulation YAML, conflating two different kinds of vastlegging.

In december 2025 the Denktank Achterkant van de Overheid published [Nieuwland](https://achterkantvandeoverheid.nl/) and the [Chronolexografie position paper](https://chronolexografie.nl/) (Olthof & Van Andel). Together they offer a coherent vocabulary for digitally recording the rechtstoestand: three types of recording (*lexogram*, *decretogram*, *executogram*), held by an organisational and juridical unit called a *chronolexocel*. The vocabulary matches the architecture RegelRecht is already building.

This RFC introduces:

1. The three chronolexogram types as first-class concepts in the schema and repository layout.
2. The cell as a juridical unit (the engine being one possible component of a cell, not the cell itself).
3. Two cross-cutting fields on chronolexograms: modality and integration extensions.
4. A `source.kind: lexostatus_query` for direct cross-cell queries, replacing the need for adapter-regulations.

The RFC is neutral about any specific external standard. Concrete integrations (with Mijn Betaaloverzicht via FCID, or any other claims standard) live in separate integration documents that reference this RFC.

This RFC **does not** introduce a `bezwaarbaar` field. An earlier draft did so. Closer reading of [RFC-008](./rfc-008) showed that the AWB lifecycle (with `BESLUIT`, `BEKENDMAKING`, `BEZWAARTERMIJN`, `BEZWAAR` stages, AWB-3:46/6:7/6:8 hooks, and explicit handling of nested besluiten op bezwaar) already models everything a `bezwaarbaar` field would have duplicated, and does so more correctly: a bezwaartermijn cannot start before bekendmaking, which a per-rule field cannot express. The route from a chronolexogram back to its bezwaar-stage is derived through RFC-008, not declared on the rule. §3.3 below specifies how.

### Design principles

1. **The corpus is the lexogram store.** Only normative content lives in `corpus/regulation/`.
2. **The chronicle store is separate from the corpus.** Registration specifications live in `chronicles/`. A registration-spec is not a regulation.
3. **A decretogram is an engine output with `legal_character: BESCHIKKING`.** New `decision_type` values cover the financial-enforcement domain.
4. **An executogram is not a regulation output.** It does not belong in `produces`. It is declared in a chronicle-stream file.
5. **A cell is a juridical unit.** The engine is one possible component of a cell.
6. **Integration configuration lives in the cell, not in the law.** Where outbound configuration must touch the regulation, it is namespaced under `extensions` so the dependency is explicit.
7. **Rechtsbescherming uses the AWB lifecycle from RFC-008**, not a parallel mechanism in this RFC.

## Terminology

This RFC adopts the vocabulary from the [Chronolexografie](https://chronolexografie.nl/) position paper. Most terms (*lexogram*, *decretogram*, *executogram*, *chronicle*, *reduction*, *lexostatus*) are introduced in context where they are first needed; full definitions live in the [glossary](/reference/glossary).

One distinction needs to be set straight up front because it is easy to misread:

A **chronolexocell** ("cell") is the *juridical and organisational domain* that holds chronicles, owns signing keys, and is the competent authority that other cells contract with via RFC-009 FSC. The cell's identity *is* its `competent_authority` per [RFC-002](./rfc-002); the cell concept simply names the unit that bears that authority across both regulation execution and chronicle-stream recording.

A **RegelRecht engine** is a *computational component* that can run inside a cell. The cell is not the engine, the engine is not the cell. A cell may contain one engine, several engines, an engine alongside a legacy system, or no engine at all. Throughout this RFC, where the difference matters, "cell" refers to the juridical unit and "engine" refers to the binary.

## Decision

### 1. Three chronolexogram types

Chronolexografie distinguishes three types of vastlegging. RegelRecht maps each to a distinct place in the schema and repository layout.

#### 1.1. Lexograms — already exist

A *lexogram* is the recording of a (possible, future) wijziging in wet- of regelgeving. A YAML file under `corpus/regulation/<jurisdiction>/<type>/<name>/<valid_from>.yaml` is a lexogram. The sequence of `valid_from` versions for one regulation is its lexogram-chronicle.

No new schema is needed. The contribution here is documentational: existing concepts pages get one paragraph each calling this what it is.

#### 1.2. Decretograms — `decision_type` extension

A *decretogram* is the recording of a concrete besluit at the application of legislation. In the schema, a decretogram is engine output with `produces.legal_character: BESCHIKKING`. The execution trace from RFC-013 is the decretogram body; the cell's FSC signature from RFC-009 §5 is the decretogram signature.

A decretogram is **lifecycle-aware** through RFC-008. The `procedure_id` field on `produces` (introduced by RFC-008) selects the AWB procedure under which the BESCHIKKING is taken. The decretogram is not a single emission; it accumulates outputs through RFC-008's stages (BESLUIT, BEKENDMAKING, BEZWAARTERMIJN, ...). Integrations that emit chronolexograms outbound (such as FCID) bind to specific stages; see §3.2 and the [MBO/FCID integration document](/integrations/mbo-fcid) for the mapping.

The existing `produces.decision_type` enum has nine values (TOEKENNING, AFWIJZING, GOEDKEURING, GEEN_BESLUIT, ALGEMEEN_VERBINDEND_VOORSCHRIFT, BELEIDSREGEL, VOORBEREIDINGSBESLUIT, ANDERE_HANDELING, AANSLAG). None describes the financial-enforcement domain. This RFC adds three values, each a distinct *type* of decretogram:

- `BETALINGSVERPLICHTING` — generic financial obligation imposed by a bestuursorgaan
- `STRAFBESCHIKKING` — criminal-law settlement under Wetboek van Strafvordering art. 257a
- `BESTUURLIJKE_BOETE` — sectoral administrative fine

That is the entire extension. Three values, deliberately small.

Two earlier candidates are excluded:

- `INCASSO_BESCHIKKING` conflated an executoriale titel (Awb 4:114 dwangbevel) with a primary besluit. A dwangbevel is closer to an executogram than to a fresh BESCHIKKING; until a concrete case appears that needs a label for a dwangbevel-as-decretogram, we do not invent one.
- `INTREKKING_BESCHIKKING` treated a *modality* as a *type*. An intrekking is a *nested decision* in the sense of RFC-008 (Open Question 5, resolved): the intrekking is itself a BESCHIKKING with its own AWB lifecycle, that as a side effect changes the original beschikking's state. The schema does not need a separate `decision_type` for this; the intrekking-besluit is just another BESCHIKKING that references the original via `modality.is_intrekking_van` (§3.1).

#### 1.3. Executograms — `chronicles/` directory

An *executogram* is the recording of feitelijke levering of afhandeling: a payment received, a kwijtschelding granted, a service delivered. An executogram is **not** a regulation output. It does not belong in any `produces` block.

Executogram-stream definitions live in a new top-level directory `chronicles/`, parallel to `corpus/`. A chronicle-stream file declares: which facts a cell records, who is competent for them, how each is named, what fields it carries. Minimal example:

```yaml
$id: payments_received
competent_authority: <organisation-id>
chronicle: cjib_main_chronicle      # which chronicle these events join
events:
  - name: payment_received
    intake: external_intake          # where the event enters the cell
    fields:
      case_reference: $external.case_reference
      amount_cents: $external.amount_cents
      received_at: $external.received_at
```

A separate JSON schema for chronicle-stream files lives at `schema/v0.6.0/chronicle.json`.

The relocation matters. A regulation is normative content. A chronicle-stream is a registration spec. Putting them in the same tree invites tooling to treat one as the other, and invites readers to think "the law for recording payments". A chronicle-stream is no such thing.

The name `chronicles/` is chosen because the content is closer to "which events does the cell's chronicle accept" than to "what does the cell do" (which would suggest a broader cell-config). Future RFCs may introduce a wider `cell-config/`; for now `chronicles/` is the minimum needed.

**Federation and reproducibility.** Chronicle-stream files participate in the same federated-corpus mechanism as regulations (RFC-010): a cell can host its chronicle-streams in its own repository alongside the regulations it implements. They also participate in the [RFC-013](./rfc-013) Execution Receipt mechanism on the same footing as regulations: when a chronicle-stream contributes to an execution (typically as the source of `BetalingVerwerkt`-style events that an engine consults during reduction), its content-hash and version are recorded in the Receipt's `scope.loaded_artifacts`. Without this, an execution that depends on a chronicle-stream cannot be reproduced months later.

### 2. The cell as juridical unit

A cell in this RFC is the organisational and juridical domain that:

- holds one or more chronicles
- owns the signing keys used for chronolexograms emitted from it
- is the `competent_authority` (RFC-002) that other cells see and contract with via RFC-009 FSC
- exposes one or more lexostatus query endpoints

A RegelRecht engine is one possible component inside a cell, alongside others (a legacy execution system, a payment intake system, a deurwaarder workflow tool). The cell decides which chronolexograms a component is authorised to write and sign.

The vocabulary mapping between Chronolexografie and RegelRecht is therefore:

| Chronolexografie | RegelRecht |
|---|---|
| chronolexocell | the organisation's juridical+organisational unit that holds chronicles, signing keys, and competence |
| chronolexochronicle | a chronicle owned by the cell; an engine writes decretogram entries into the chronicle named in its config |
| chronolexogram | one entry: a decretogram (engine output) or an executogram (chronicle-stream event) |
| chronolexoreduction | one engine execution: filtering chronicle entries and producing a lexostatus |
| lexostatus | the output of an execution, or the answer to a `lexostatus_query` |
| chronolexosphere | the federation of cells (RFC-009 federated mode) |

This is a clarification, not a code change. RFC-009's `EngineIdentity` is the cell's identity from the engine's perspective. A deployment chooses whether one engine equals one cell, multiple engines share a cell, or a cell mixes engine and non-engine components.

### 3. Two cross-cutting fields, plus rechtsbescherming via RFC-008

Two fields apply to chronolexograms regardless of type. They are introduced here together because they share a pattern: each travels with the chronolexogram when it is signed, propagated, or surfaced. The third concern, rechtsbescherming, is *not* a new field but a derivation through RFC-008; §3.3 explains.

#### 3.1. `modality`

Some besluiten are not new types but operations on existing besluiten. An intrekking, a wijziging, a herziening: each is a handeling *on* a bestaand besluit, not a fresh besluit-type. The `modality` block captures that:

```yaml
produces:
  legal_character: BESCHIKKING
  decision_type: BETALINGSVERPLICHTING
  modality:
    is_intrekking_van: "<beschikking-id>"   # optional
    is_wijziging_van:  "<beschikking-id>"   # optional
```

This matches Awb idiom and avoids polluting `decision_type` with modalities-disguised-as-types. The intrekking is itself a BESCHIKKING with its own AWB lifecycle per RFC-008; `modality.is_intrekking_van` makes the link to the original beschikking explicit so that a downstream consumer (citizen surface, integration) can present the change correctly.

The same field is allowed on executograms in chronicle-stream files, for events such as `kwijtschelding_verleend` that effectively intrekken an earlier vordering.

#### 3.2. `extensions` — namespaced integration hooks

Integrations need to attach configuration to chronolexograms without baking domain assumptions into the base schema. `extensions` is a namespaced map. Each integration owns one namespace key; the integration document for that namespace owns the shape and semantics inside its block.

```yaml
produces:
  legal_character: BESCHIKKING
  decision_type: BETALINGSVERPLICHTING
  extensions:
    mbo_fcid:
      category: ALGEMEEN
```

The same shape applies to chronicle-stream events:

```yaml
events:
  - name: payment_received
    extensions:
      mbo_fcid:
        event_type: BetalingVerwerkt
        category: ALGEMEEN
```

A reader of the schema sees that there is extension-driven behaviour, and which integration drives it, without the base schema having to know what `category: ALGEMEEN` means.

**Stage-aware emission.** When an integration emits an outbound event from a decretogram, it does so at a specific RFC-008 lifecycle stage. FCID emission of a `BetalingsverplichtingOpgelegd` belongs at `BEKENDMAKING`, not at `BESLUIT`: a verplichting that the citizen has not yet been notified of has no juridical existence to display in a betaaloverzicht. Integration documents declare which stage they emit at; the engine fires the emission as part of that stage's hook execution per RFC-008.

**Activation lives in the cell, not in the law.** Whether a cell actually emits FCID for its decretograms is a cell-config concern, not a regulation concern. A gemeente that runs the same Wahv as CJIB but does not connect to MBO simply does not activate `mbo_fcid` in its cell-config; the same lexogram works in both. The presence of an `extensions.mbo_fcid` block is a *hint* for cells that do activate the integration; the absence is a hint that no integration is relevant for the rule.

#### 3.3. Rechtsbescherming: derived from the AWB lifecycle (RFC-008), not a new field

An earlier draft of this RFC introduced a per-rule `bezwaarbaar` block carrying `door`, `via`, `termijn_dagen`, `grondslag`. That was wrong.

RFC-008 already models bezwaar as part of the AWB lifecycle. The bezwaartermijn is computed at `BEKENDMAKING` (AWB 6:8 hooks fire there, not at BESLUIT, because the termijn cannot start before notification). The bezwaarstage runs after BEKENDMAKING. The route to file bezwaar is governed by AWB-7:1 plus the specific bezwaarregeling of the regulation. A nested *besluit op bezwaar* is itself a BESCHIKKING with its own AWB lifecycle.

A per-rule `termijn_dagen: 42` would have been a static hint at BESLUIT-time of a value that RFC-008 correctly computes at BEKENDMAKING-time. A static `via: cell.cjib.bezwaar` would have duplicated routing that the AWB-procedure of the cell already determines. The right design is: the rechtsbescherming-route accompanying a chronolexogram is **derived** from the procedure that produced it, not declared on the rule.

Concretely:

- **For decretograms.** When a cell emits an outbound event for a decretogram, it includes a `bezwaar_route` field. This field is computed by the engine from the RFC-008 procedure of the producing rule: `bezwaar_route.intake` = the cell's bezwaar-intake for that procedure, `bezwaar_route.termijn` = the bezwaartermijn-output that AWB-6:7/6:8 hooks have produced at BEKENDMAKING (so it carries an actual einddatum, not a duration). For UOV procedures (RFC-008 §A.8), `bezwaar_route` is absent and the cell instead includes `beroep_route` per AWB-7:1 lid 1 sub d. For BESLUIT_VAN_ALGEMENE_STREKKING, neither is present except for "concretiserend BAS" (RFC-008 §A.9).

- **For executograms.** An executogram event is a feit, not a besluit. It carries no `bezwaar_route` by default. A specific executogram (such as `kwijtschelding_verleend`) may carry one *because the event implicitly references a nested besluit*; in that case the chronicle-stream event declares `references_decision: <decision-id>` and the engine derives `bezwaar_route` from that decision's procedure.

- **Surface in lexostatus and trace.** A lexostatus returned by a cell includes the `bezwaar_route` for each chronolexogram. The execution trace records, at each BEKENDMAKING-stage node, the computed bezwaartermijn (this already follows from RFC-008 hooks on AWB-6:7/6:8). Citizen-facing surfaces (MBO, MijnOverheid, the engine in WASM) can render the objection-trigger inline.

- **Cross-cell propagation.** When cell A accepts a decretogram from cell B via FSC, the `bezwaar_route` travels with it. Cell A's downstream lexostatus exposes the *original* objection route to the citizen, not just cell A's own route.

This RFC's contribution to rechtsbescherming is therefore not a new field, but the requirement that integrations carry the route in their outbound payloads, derived from RFC-008's lifecycle output. The MBO/FCID integration document specifies the concrete shape of the `bezwaar_route` it serializes.

### 4. Cross-cell queries: `source.kind: lexostatus_query`

A regulation that needs data from another cell uses a new `source.kind` discriminator:

```yaml
input:
  - name: openstaande_vorderingen
    source:
      kind: lexostatus_query
      cell: cjib                          # which cell answers
      lexostatus: openstaande_vorderingen # which lexostatus to retrieve
      parameters:
        bsn: $bsn
```

This says what it does: ask cell X for lexostatus Y. The producing cell does its own chronolexoreductie internally and returns a lexostatus. The consuming engine accepts the result through the RFC-009 ACCEPT path.

The discrimination on `source` happens via a new `kind` field with values `regulation` (today's default) and `lexostatus_query`. Existing source declarations that omit `kind` default to `regulation`, preserving backward compatibility.

The earlier draft of this RFC proposed a wrapper *regulation* in `corpus/regulation/` (`procedureregeling_vorderingenoverzicht_rijk`) as the adapter. That was misleading: it would have placed a non-regulation in the regulation tree under a misleading juridical genre name. A direct `source.kind: lexostatus_query` is more honest and does not pollute the corpus.

### 5. Out of scope

- **Any specific outbound serialisation format.** Belongs in integration documents.
- **Any specific integration partner.** Belongs in integration documents and casus documents.
- **The AWB lifecycle, bezwaar mechanics, nested besluiten op bezwaar.** Already in RFC-008. This RFC builds on RFC-008's procedure model; it does not redefine it.
- **Trust and signing mechanics.** Already in RFC-009 §5.
- **Execution provenance and trace structure.** Already in RFC-013.
- **Federation transport.** Already in RFC-009 (FSC).
- **Federated corpus distribution.** Already in RFC-010, applied here to chronicle-streams as well.
- **The full cell-config format.** This RFC introduces `chronicles/` and the integration-extension activation site; broader cell-config is for a future RFC.
- **The legal basis for inter-organisation data exchange.** Per-case, not at engine level.
- **A statutory Wet gegevensboekhouding** (Nieuwland §7.3.2). Legislative artefact; this RFC is an engineering one.

## Why

### Benefits

- **Lexogram, decretogram and executogram are explicit and separately located.** A future reader sees the type from the path alone.
- **The cell is a juridical unit, not a binary.** Implementations vary widely (one engine, several engines, engine plus legacy) without breaking the conceptual picture.
- **The financial-enforcement domain becomes expressible.** Three new `decision_type` values, one modality block. No mixing of types and modalities.
- **Integrations don't drag domain assumptions into the schema.** The `extensions` namespace mechanism lets integrations evolve without schema surgery.
- **Activation lives in the cell.** A gemeente can run the same Wahv lexogram as CJIB without forking it just because the gemeente doesn't emit FCID.
- **Cross-cell queries are first-class.** No fake-regulation adapters.
- **Rechtsbescherming reuses RFC-008.** No parallel mechanism. The objection route on an emitted event is derived from the AWB procedure that produced the besluit, computed at the right lifecycle stage.
- **Chronicle-streams are reproducible.** They participate in RFC-013 Execution Receipts, so executions that depend on them can be replayed.

### Tradeoffs

- **The `decision_type` enum extension is breaking for downstream consumers of the enum.** Schema bump to v0.6.0 is required.
- **Introducing `chronicles/` adds a second top-level repository directory.** Tooling needs to learn about it. One-time cost.
- **`extensions` blocks are deliberately uninterpreted at the schema level.** A reader still needs an integration document to know what `mbo_fcid.category: VERHOGING` means. The gain is that the schema is honest about this rather than pretending.
- **Activation-in-cell means an FCID-aware regulation YAML can be loaded by a non-FCID cell without effect.** Acceptable: presence is a hint, activation is a cell decision.
- **Integrations must understand RFC-008 stages to emit at the right moment.** This is a constraint on integration authors. The MBO/FCID integration document demonstrates the pattern.
- **The schema-version bump.** v0.5.2 → v0.6.0 is a minor bump because all changes are additive: new enum values (existing values unchanged), new optional `modality` and `extensions` blocks, a new optional `source.kind` discriminator that defaults to `regulation` for backward compatibility. A new chronicle JSON schema lives alongside the existing one and does not affect regulation loading. No existing regulation YAML breaks.

### Alternatives Considered

**Alternative 1: Put executogram-streams under `corpus/`**.
- Rejected: conflates norm and registration. A registration-spec is not a regulation. Tooling that loads `corpus/**` would treat the two alike. The Chronolexografie distinction between registreren and interpreteren collapses at the file-system level.

**Alternative 2: Treat the engine itself as the cell**.
- Rejected: ties the cell-concept to one specific binary. Cells must be able to contain multiple components or none. The cell is juridical and organisational; the engine is one of its tools.

**Alternative 3: Two anonymous `outbound_emit` / `outbound_category` fields on `produces`**.
- Rejected. Anonymous fields hide which integration drives them and force the reader to leave the schema to learn what they mean. A namespaced `extensions.<integration_name>` block makes the dependency explicit.

**Alternative 4: Put activation in the regulation (`fcid_emit: true` on the rule)**.
- Rejected: makes outbound serialisation a property of the law. A gemeente that runs the same wet but does not connect to MBO would have to fork the regulation. Activation belongs in the cell that runs the engine, not in the wet.

**Alternative 5: Keep the wrapper-regulation pattern (`procedureregeling_vorderingenoverzicht_rijk`)**.
- Rejected: a "procedureregeling" is a juridical genre; misusing the term for what is really an adapter invites confusion. `source.kind: lexostatus_query` says what it actually is.

**Alternative 6: Treat `INTREKKING_BESCHIKKING` as a `decision_type`**.
- Rejected: an intrekking is a nested besluit per RFC-008 (its own AWB lifecycle), not a new besluit-type. Captured via `produces.modality.is_intrekking_van`.

**Alternative 7: Introduce a per-rule `bezwaarbaar` block on `produces`** (this was an earlier draft of this RFC).
- Rejected. The AWB lifecycle (RFC-008) already models bezwaar correctly, including the timing constraint that the termijn starts at BEKENDMAKING (not BESLUIT) and the nesting of besluiten op bezwaar. A per-rule field would have duplicated this badly: a static `termijn_dagen: 42` cannot reflect that the einddatum is computed from `bekendmaking_datum + AWB-6:7 + Termijnenwet`. The right derivation is via RFC-008 hooks; this RFC's contribution is to require integrations to *carry* the resulting route in their outbound payloads.

### Implementation Notes

This RFC is the architecture decision. Schema and engine work are in follow-up PRs.

#### Phase 1: Schema

- Bump `schema/v0.5.2` → `schema/v0.6.0`. Update `schema/latest` symlink.
- Add new `decision_type` values: BETALINGSVERPLICHTING, STRAFBESCHIKKING, BESTUURLIJKE_BOETE.
- Add `produces.modality` optional block.
- Add `produces.extensions` as a free-form namespaced map.
- Add `schema/v0.6.0/chronicle.json` for chronicle-stream files.
- Add `source.kind` discriminator with `regulation` (default) and `lexostatus_query`.
- All changes additive; no existing regulation YAML breaks.

#### Phase 2: Engine support

- New: `packages/engine/src/chronicle.rs` — loading, validating, and triggering chronicle-stream entries.
- New types in `packages/engine/src/types.rs`: `ChronicleStream`, `Modality`, `LexostatusQuerySource`.
- Update `packages/engine/src/resolver.rs` to dispatch on `source.kind`.
- Update RFC-013 Execution Receipt builder to include loaded chronicle-streams alongside regulations.
- Wire integration extension emission into RFC-008 stage-hook execution: an integration that declares emission at `BEKENDMAKING` fires its emitter as part of that stage.

#### Phase 3: Vocabulary

- Update [law-format](/concepts/law-format), [multi-org-execution](/concepts/multi-org-execution), [execution-provenance](/concepts/execution-provenance) to use the cell/chronicle/lexogram vocabulary.
- Add glossary entries.

#### Phase 4: Pilot

- One chronicle-stream YAML under `chronicles/` as a working example.
- One integration document under `docs/integrations/` using `extensions.<namespace>`. First integration is Mijn Betaaloverzicht / FCID; see [that document](/integrations/mbo-fcid) for stage-binding, `bezwaar_route` derivation, and field-level mapping.
