# RFC-022: RegelRecht × Mijn Betaaloverzicht — Producer and Consumer Interop

**Status:** Proposed
**Date:** 2026-05-26
**Authors:** Anne Schuth

## Terminology

This RFC uses **Mijn Betaaloverzicht (MBO)** for the citizen-facing service formerly known as *Vorderingenoverzicht Rijk (VORIJK)*. The name change took effect on 2025-07-10. The technical document standard, **Financial Claims Information Document (FCID)**, retains its original name and is the schema this RFC integrates with.

## Context

RegelRecht today turns Dutch law into executable specifications. Its corpus contains 23 regulations across fiscal and social domains. None of them produces a financial-enforcement output: no `BETALINGSVERPLICHTING`, no `STRAFBESCHIKKING`, no `INCASSO_OPGELEGD`. The `produces.decision_type` enum reflects this — it has TOEKENNING, AFWIJZING, AANSLAG, but nothing in the incasso domain.

In parallel, the Dutch government is building [Mijn Betaaloverzicht (MBO)](https://www.eenoverheidsincasso.nl/onze-dienstverlening/vorderingenoverzicht-rijk). MBO consolidates a citizen's financial obligations across at least 15 government opdrachtgevers into one overview, currently administered through CJIB. The standard underlying MBO is FCID, published by the [Blauwe Knop / VORIJK](https://vorijk.nl/docs/financiele-verplichtingen/document_types/financial_claims_information_document/) program. FCID is event-sourced with four event types (`FinancieleVerplichtingOpgelegd`, `BetalingsverplichtingOpgelegd`, `BetalingsverplichtingIngetrokken`, `BetalingVerwerkt`) and four categories (Algemeen, Administratiekosten, Verhoging, Rente).

Today these two worlds are disconnected. A citizen who receives a Wahv fine sees the amount in MBO but cannot trace it to the specific article of the [Wet administratiefrechtelijke handhaving verkeersvoorschriften](https://wetten.overheid.nl/BWBR0004581) that produced it. Conversely, a RegelRecht-style Wsnp engine that needs the list of openstaande vorderingen for a citizen has no formal way to query MBO. The keten "burger ontvangt boete → ziet vordering in Mijn Betaaloverzicht → krijgt juridische uitleg" falls apart at every transition.

[RFC-009 (Multi-Organisation Execution)](./rfc-009) already provides the federation blueprint: FSC-based service discovery, EXECUTE/ACCEPT resolution based on `competent_authority`, signed individual decisions across organizational boundaries. The same eight CRI rijksorganisaties (Belastingdienst, Dienst Toeslagen, CJIB, DUO, SVB, CAK, UWV, RVO) that anchor MBO are the natural Peers in a RegelRecht federation. What is missing is the semantic bridge: a way for a RegelRecht engine to emit FCID events when it produces a financial-enforcement decision, and a way for a RegelRecht regulation to consume MBO claims as input. See [CJIB-uitvoeringslandschap](/concepts/cjib-uitvoeringslandschap) for the full mapping context.

### Design principles

1. **The law remains the source.** An FCID event emitted by a RegelRecht engine carries a direct link to the article that produced it. Provenance does not get lost at the boundary.
2. **No duplicate infrastructure.** Reuse the FSC trust model, the engine identity, and the signature mechanism from RFC-009. One signing key, two output formats.
3. **Bidirectional but asymmetric.** Producing FCID is a transformation of an existing RegelRecht output. Consuming MBO claims is a normal cross-law reference where CJIB is the competent authority. The producer side needs schema extension; the consumer side does not.
4. **Citizen-initiated, data at source.** MBO's principles (no intermediaries, source organizations sign their own data) align with RegelRecht's RFC-009 model. This RFC does not weaken either.

## Decision

### 1. Two interaction roles

A RegelRecht engine that interacts with MBO operates in one or both of two roles:

- **Producer**: when the engine produces a `BESCHIKKING` whose `decision_type` falls in the financial-enforcement family, it can emit a corresponding FCID event to the source organization's MBO endpoint.
- **Consumer**: when a regulation needs the citizen's openstaande vorderingen as input, it references a wrapper regulation whose competent authority is CJIB, and accepts the response via the RFC-009 ACCEPT path.

Both roles are optional and independent. A pure producer (e.g., a CJIB-internal engine running Wahv) emits but never consumes. A pure consumer (e.g., a hypothetical Wsnp engine) consumes but never emits.

### 2. Producer-side mapping

The mapping from RegelRecht output to FCID event is deterministic. The engine derives the FCID event from the `produces` block of the article that produced the output:

| RegelRecht `produces`                                          | FCID `event_type`                  |
|----------------------------------------------------------------|-----------------------------------|
| `BESCHIKKING` + `decision_type: STRAFBESCHIKKING` (totaal)     | `FinancieleVerplichtingOpgelegd`  |
| `BESCHIKKING` + `decision_type: BETALINGSVERPLICHTING`         | `BetalingsverplichtingOpgelegd`   |
| `BESCHIKKING` + `decision_type: BESTUURLIJKE_BOETE`            | `BetalingsverplichtingOpgelegd`   |
| `BESCHIKKING` + `decision_type: INCASSO_OPGELEGD`              | `BetalingsverplichtingOpgelegd`   |
| Override via [RFC-007](./rfc-007) replacing an earlier BESCHIKKING | `BetalingsverplichtingIngetrokken` (for the replaced one) |
| External event from incasso system                             | `BetalingVerwerkt`                |

Field-level derivation:

- `juridische_grondslag_omschrijving` ← first sentence of `article.text`, or `article.title` if shorter
- `juridische_grondslag_bron` ← `article.url` (canonical wetten.overheid.nl link)
- `zaakkenmerk` ← deterministic hash of (engine `organisation_id` + beschikking-id)
- `gebeurtenis_kenmerk` ← UUID v7
- `bedrag` ← currency-typed output × 100 (FCID requires centen, integer)
- `signature` ← signed with the same FSC key from [RFC-009 §5](./rfc-009)

A new optional field on `produces` controls emission:

```yaml
execution:
  produces:
    legal_character: BESCHIKKING
    decision_type: BETALINGSVERPLICHTING
    fcid_emit: true
    fcid_category: ALGEMEEN
```

`fcid_emit` defaults to `false`. `fcid_category` is one of `ALGEMEEN`, `ADMINISTRATIEKOSTEN`, `VERHOGING`, `RENTE`.

### 3. Consumer-side mapping

The consumer side does not need a schema change. A new wrapper regulation `procedureregeling_vorderingenoverzicht_rijk` is added to the corpus, with CJIB as `competent_authority`. It exposes a single article that produces `openstaande_vorderingen[]` for a given BSN. Any regulation that needs this data references the output in the standard `source.regulation` form:

```yaml
input:
  - name: openstaande_vorderingen
    source:
      regulation: "procedureregeling_vorderingenoverzicht_rijk"
      output: "openstaande_vorderingen"
      parameters:
        bsn: $bsn
```

The engine resolves this through the RFC-009 EXECUTE/ACCEPT decision tree. Because the wrapper regulation declares `competent_authority: CJIB`, an engine running under a different identity hits the ACCEPT path and queries CJIB's MBO endpoint via FSC. An engine running as CJIB itself executes the regulation locally, which in turn delegates to CJIB's internal MBO query system.

The wrapper regulation does not duplicate FCID semantics. It is a thin RegelRecht-facing shim that produces a list of vordering records suitable as input to other regulations.

### 4. Schema changes

This RFC bumps the schema to `v0.6.0` with two additive changes:

1. **`produces.decision_type` enum extension**, adding:
   - `BETALINGSVERPLICHTING`
   - `STRAFBESCHIKKING`
   - `BESTUURLIJKE_BOETE`
   - `INCASSO_OPGELEGD`
   - `INCASSO_INGETROKKEN`
   - `BETALING_VERWERKT`

2. **`produces.fcid_emit`** (optional bool, default `false`) and **`produces.fcid_category`** (optional enum: ALGEMEEN, ADMINISTRATIEKOSTEN, VERHOGING, RENTE).

`legal_character` is unchanged — `BESCHIKKING` covers all financial-enforcement cases.

### 5. Trust and identity

The trust model is exactly RFC-009's. Each FCID event is signed with the source organization's FSC key. A citizen-side or peer-side consumer verifies the signature against the FSC Directory's Trust Anchor. No new key infrastructure, no new authentication layer.

For the consumer side: querying CJIB for openstaande vorderingen is one more FSC-mediated call, indistinguishable from any other RFC-009 AuthorityCall.

### 6. Trace and provenance

Producer side: the FCID event carries the W3C `trace_id` from the BESCHIKKING execution trace. A citizen who later wants to know how their vordering was computed can request the trace from the source organization using this `trace_id` (RFC-009 §8, aligned with [Logboek Dataverwerkingen](https://logius-standaarden.github.io/logboek-dataverwerkingen/)).

Consumer side: an `AuthorityCall(CJIB, output: openstaande_vorderingen)` node appears in the consuming engine's trace exactly as any other cross-org call. The accepted list is part of the audit trail for the downstream decision.

### 7. Privacy

Producer events are emitted at the moment of beschikking, never as a bulk export. Each event carries a single citizen's data.

Consumer queries are citizen-initiated by construction: a regulation only fires when invoked for a specific BSN. CJIB enforces BSN-binding on its side and logs the query in its own Logboek Dataverwerkingen. No org receives bulk lists of vorderingen.

### 8. Out of scope

This RFC deliberately does not address:

- **Legal basis for inter-org data exchange**. RFC-009 already notes this is per-case, not addressed at the engine level.
- **MBO user interface**. The citizen-facing portal is built by CJIB and CRI partners, not by RegelRecht.
- **Payment processing**. iDEAL, automatic incasso, and reconciliation are out of scope.
- **The full VORIJK API surface**. FCID is in scope. The Financial Claim Request API and Session API are referenced for context but their integration is left for a follow-up RFC.
- **Citizen authentication**. DigiD-binding on the consumer side is a separate concern, handled at the API gateway layer.

## Why

### Benefits

- **End-to-end traceability**. A citizen seeing a Wahv vordering in MBO can follow the link straight to the article of the Wahv that produced it, with the full execution trace available on request.
- **No new identity infrastructure**. FSC, identity, and signing all come from RFC-009. This RFC adds a serialization, not a security model.
- **First multi-domain coupling**. RegelRecht has been fiscal/social so far. Connecting it to the incasso world via CJIB is the first step into the strafrecht/bestuursrecht-handhaving space.
- **One source of truth for source organizations**. A CJIB engine that runs Wahv emits FCID events directly from its execution path, instead of maintaining a separate FCID-emitter alongside its execution system.
- **Consumer side scales**. The wrapper regulation is a single corpus addition. Any future regulation that needs openstaande vorderingen (Wsnp, Betalingsregeling Rijk, debt-counselling tools) reuses it via standard cross-law references.

### Tradeoffs

- **The `decision_type` enum extension is breaking for downstream consumers of the enum**. Schema bump to v0.6.0 is mandatory.
- **FCID is still evolving**. The standard is at v4.2.0 as of mei 2026. The producer mapping may need updating per FCID minor version. Vendoring the FCID JSON schema (in Phase 3) lets us detect breakage in CI.
- **Several CJIB-domain regulations are not yet in the corpus**. Wahv, Wet OM-afdoening, and the sectoral acts behind bestuurlijke boetes still need to be authored. This RFC enables the integration but does not deliver the regulations themselves.
- **The wrapper regulation is a slight semantic stretch**. `procedureregeling_vorderingenoverzicht_rijk` is not a wet in the traditional sense; it is a RegelRecht-internal shim. Acceptable as long as it is clearly labeled.

### Alternatives Considered

**Alternative 1: Consumer side first, producer later**
- Build only the wrapper regulation and the ACCEPT path in this RFC; defer the producer-side schema extension to a separate future RFC.
- Rejected: the producer side is the side that actually closes the keten for citizens. Splitting reduces the value proposition for CJIB engagement, which is the immediate driver for this RFC.

**Alternative 2: External microservice transforms RegelRecht output into FCID**
- Keep the engine free of FCID knowledge; a separate service tails the trace bus and emits FCID events.
- Rejected: loses the direct connection between article and event. The whole point is that the article-URL travels with the event. A transforming service in the middle re-introduces the staleness and provenance gaps that the RFC tries to fix.

**Alternative 3: Add a full "incasso domain" block to the schema**
- Introduce `produces.financial_enforcement: { ... }` as a structured sub-object with all FCID semantics encoded in the schema.
- Rejected: overfits the schema to one specific external standard. The enum extension + two optional fields keep the schema general and the FCID-specific mapping in engine code.

### Implementation Notes

This RFC is the architecture decision. The actual schema bump and engine implementation are intentionally separated into follow-up PRs so this RFC can be reviewed on its merits.

#### Phase 1: Schema bump

- Bump `schema/v0.5.2` → `schema/v0.6.0`, update `schema/latest` symlink.
- Add new `decision_type` enum values.
- Add `fcid_emit` and `fcid_category` optional fields under `produces`.
- No existing corpus regulation needs to change.

#### Phase 2: Producer module

- New: `packages/engine/src/fcid.rs` — pure mapping from BESCHIKKING output to FcidEvent struct.
- New: `FcidEvent` struct in `packages/engine/src/types.rs`.
- Reuse signing path from RFC-009 implementation (`packages/engine/src/signature.rs`).
- New: `packages/editor-api` endpoint for emitting FCID events from an execution trace (development/inspection use).

#### Phase 3: Schema vendoring and validation

- Vendor FCID v4.x JSON schemas under `schema/external/vorijk/`, with a README documenting the upstream source and snapshot date.
- Snapshot tests: take a fixture Wahv beschikking, run it through `fcid.rs`, validate the output JSON against the vendored FCID schema.

#### Phase 4: Consumer wrapper

- New corpus regulation: `corpus/regulation/nl/procedureregeling/vorderingenoverzicht_rijk/2026-01-01.yaml`.
- Stub CJIB peer entry in development `service-registry.yaml` (RFC-009 §4).
- BDD scenario in `features/`: a hypothetical Wsnp evaluation with three mock vorderingen returned by the stub CJIB peer.

#### Affected components
