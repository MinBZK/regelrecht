# RFC-022: Chronolexogram types in the schema and the cell model

**Status:** Proposed
**Date:** 2026-05-26
**Authors:** Anne Schuth

## Terminology

This RFC adopts the vocabulary from the [Chronolexografie](https://chronolexografie.nl/) position paper (Olthof & Van Andel, december 2025) and the [Nieuwland — een ontwerp voor een digitale rechtsstaat](https://achterkantvandeoverheid.nl/) program (Achterkant van de Overheid, december 2025).

| Term | Working translation | Definition (paraphrased from the position paper) |
|---|---|---|
| **lexogram** | law-recording | vastlegging van een (mogelijke, toekomstige) wijziging in wet- of regelgeving |
| **decretogram** | decision-recording | vastlegging van een concreet besluit of beschikking bij toepassing van wetgeving |
| **executogram** | execution-recording | vastlegging van daadwerkelijke levering of afhandeling van een zaak of dienst |
| **chronolexocel** (cell) | cell | het juridisch en organisatorisch domein waarin chronolexogrammen worden gemaakt, bewaard en verwerkt |
| **chronolexokroniek** (chronicle) | chronicle | tijdsgeordende reeks van chronolexogrammen binnen één cell |
| **chronolexoreductie** (reduction) | reduction | inherent destructieve activiteit van hergebruik van feiten op een later moment |
| **lexostatus** | rechtstoestand-projectie | rechtstoestand zoals gezien vanuit een bepaald (gevraagd) perspectief |

A RegelRecht engine is **not** a cell. The cell is the organisational and juridical unit that owns a chronicle and that signs chronolexograms; an engine is a computational component that can be part of a cell. A cell may contain one RegelRecht engine, several engines, an engine plus a legacy system, or none. This RFC is careful to keep the two concepts apart.

## Context

The Chronolexografie position paper proposes a coherent vocabulary for how Dutch government organisations record the rechtstoestand digitally. It distinguishes three types of recording (lexogram, decretogram, executogram), and places them in cells with their own chronicles. The wider Nieuwland design embeds this in a broader picture of registreren, interpreteren, ontwerpen en normeren.

RegelRecht implements parts of this vocabulary already, without using its names. The 23 regulations in `corpus/regulation/` are de facto lexograms. A BESCHIKKING produced by an engine is a decretogram. Execution traces (RFC-013, Draft) are chronicle-like ordered records of how a decretogram came about. RFC-009 (Multi-Organisation Execution, Proposed) provides each engine deployment with an identity that can act as one of the components of a cell.

What is missing is the third recording type and a clean separation between **normative content** (lexograms in the corpus) and **registration specifications** (which facts a cell records, and how). Today there is no place to declare that a cell records payments, kwijtscheldingen, or service deliveries as facts. Integrations that need them either invent a local format or push the recording into the regulation YAML, which conflates norm and fact.

This RFC introduces:

1. An explicit, named place for executogram-stream definitions, outside the corpus.
2. An extended `produces.decision_type` enum for decretograms in the financial-enforcement domain.
3. A neutral, namespaced **extensions** mechanism on `produces` for integration-specific configuration.
4. A new cross-cell **`source.kind: lexostatus_query`** to ask another cell for a lexostatus, without inventing a fake regulation as adapter.
5. A `bezwaarbaar` affordance on decretograms and executograms, so that the chain that this RFC enables also carries rechtsbescherming, not only visibility.

The RFC is deliberately neutral about any specific external standard. Concrete integrations (with Mijn Betaaloverzicht via FCID, or any other claims standard) live in separate integration documents that reference this RFC.

### Design principles

1. **The corpus is the lexogram store.** Only normative content lives in `corpus/regulation/`.
2. **The chronicle store is separate from the corpus.** Executogram-stream definitions live in `chronicles/`, not in the corpus. A registration specification is not a regulation.
3. **A decretogram is an engine output with `legal_character: BESCHIKKING`.** New `decision_type` values cover the financial-enforcement domain.
4. **An executogram is not a regulation output.** It does not belong in `produces`. It is declared in a chronicle-stream file.
5. **A cell is a juridical unit.** A RegelRecht engine is part of a cell, not the cell itself. The cell owns chronicles, signs chronolexograms, and is what counts as `competent_authority` in RFC-009.
6. **Integration configuration lives in the cell, not in the law.** A regulation's outbound serialisation is not a property of the regulation; it is a property of the cell that runs it. Where outbound configuration must touch the regulation, it is namespaced under `extensions`.
7. **What works as straf deserves rechtsbescherming as straf.** Following Nieuwland §7.2.1, every decretogram and executogram carries an explicit `bezwaarbaar` affordance.

## Decision

### 1. The cell as juridical unit

A cell in this RFC is the organisational and juridical domain that:

- holds one or more chronicles
- owns the signing keys used for chronolexograms emitted from it
- is the `competent_authority` that other cells see and contract with via RFC-009 FSC
- exposes one or more lexostatus query endpoints

A RegelRecht engine is a component that can run inside a cell, alongside other components (a legacy execution system, a payment intake system, a deurwaarder workflow tool). The cell decides which chronolexograms a component is authorised to write and sign.

This is a clarification, not a code change. RFC-009's engine identity remains; a deployment chooses whether one engine identity equals one cell, or whether multiple identities share a cell, or whether one cell contains both an engine and non-engine components.

For practical purposes, a cell has a configuration file. This RFC does not standardise the full cell-config format (that is for a future RFC), but it does locate two specific items that this RFC introduces in the cell's domain: the chronicle-stream files (§3) and the integration-extension activation (§5).

### 2. `produces.decision_type` enum extension

Add the following values to `produces.decision_type`:

- `BETALINGSVERPLICHTING` — generic financial obligation imposed by a bestuursorgaan
- `STRAFBESCHIKKING` — criminal-law settlement under Wetboek van Strafvordering art. 257a
- `BESTUURLIJKE_BOETE` — sectoral administrative fine

That is the entire extension to the enum. Three values, each a distinct **type** of decretogram.

Two values from the first draft of this RFC are deliberately removed:

- `INCASSO_BESCHIKKING` was conflating an executoriale titel (Awb 4:114 dwangbevel) with a primary besluit. Until a concrete case appears that needs a label for a dwangbevel-as-decretogram, we do not invent the label. The dwangbevel itself is closer to an executogram than to a fresh BESCHIKKING.
- `INTREKKING_BESCHIKKING` was treating a modality as a type. An intrekking is a modality on an existing besluit, not a new type of besluit. To capture this, `produces` gains a small modality block (§2.1).

#### 2.1. Modality fields on `produces`

Add the following optional fields to `produces`, applicable to any `decision_type`:

```yaml
produces:
  legal_character: BESCHIKKING
  decision_type: BETALINGSVERPLICHTING
  modality:
    is_intrekking_van: "<beschikking-id>"   # optional
    is_wijziging_van:  "<beschikking-id>"   # optional
```

An intrekking of an earlier beschikking is therefore a `decision_type: <same-type-as-original>` with `modality.is_intrekking_van: <id>`. This matches the Awb idiom: intrekken is a handeling on a bestaand besluit, not a new besluit-type.

### 3. Chronicle-stream files: `chronicles/`, not `corpus/`

Executogram-stream definitions live in a new top-level directory `chronicles/`, parallel to `corpus/`. A chronicle-stream file is **not** a regulation. It is a registration specification: which facts a cell records, who is competent for them, how each is named, what fields it carries.

Minimal example:

```yaml
$id: payments_received
competent_authority: <organisation-id>
chronicle: cjib_main_chronicle      # which chronicle these events join
events:
  - name: payment_received
    intake: external_intake          # where the event enters the cell
    fields:
      case_reference: $external.case_reference
      amount_cents: $external.amount_cents
      received_at: $external.received_at
    bezwaarbaar:
      door: belanghebbende
      via: cell.<organisation>.bezwaar
      termijn_dagen: 42
```

A separate JSON schema for chronicle-stream files lives at `schema/v0.6.0/chronicle.json`.

This relocation matters. A regulation is normative content. A chronicle-stream is a registration spec. Putting them in the same tree invited tooling and validators to treat one as the other, and invited readers to think "the law for recording payments". A chronicle-stream is no such thing.

The directory name `chronicles/` is chosen because the content is closer to "what does the cell's chronicle accept as a recordable event" than to "what does the cell do" (which would suggest a broader cell-config). Future RFCs can introduce a wider `cell-config/` if needed; for now `chronicles/` is the minimum.

### 4. The engine as a component of a cell

The vocabulary mapping is:

| Chronolexografie | RegelRecht today |
|---|---|
| chronolexocel | the organisation's juridical+organisational unit that holds chronicles, signing keys, and competence |
| chronolexokroniek | one of the cell's chronicles; an engine writes decretogram entries into the chronicle named in its config |
| chronolexogram | one entry: a decretogram (engine output, `BESCHIKKING`) or an executogram (event from a chronicle-stream) |
| chronolexoreductie | one engine execution: filtering chronicle entries and producing a lexostatus |
| lexostatus | the output of an engine execution, or the answer to an external `lexostatus_query` |
| chronolexosfeer | the federation of cells (RFC-009 federated mode) |

The intent is to use this vocabulary consistently in future documentation so that readers familiar with Chronolexografie and Nieuwland recognise the architecture without translating.

### 5. Integration extensions

To support concrete integrations without committing the schema to any one of them, `produces` gains a namespaced `extensions` block:

```yaml
produces:
  legal_character: BESCHIKKING
  decision_type: BETALINGSVERPLICHTING
  extensions:
    mbo_fcid:
      emit: true
      category: ALGEMEEN
```

The same shape applies to chronicle-stream events:

```yaml
events:
  - name: payment_received
    intake: external_intake
    extensions:
      mbo_fcid:
        event_type: BetalingVerwerkt
        category: ALGEMEEN
```

Each integration declares its own namespace key (here `mbo_fcid`). The integration document for that namespace owns the shape and semantics inside its block. The base schema permits but does not interpret the inner contents.

A lezer of the schema sees that there is extension-driven behaviour, and which integration drives it, without the base schema having to know what `category: ALGEMEEN` means.

#### 5.1. Activation lives in the cell, not in the law

Whether a cell actually emits FCID for its decretograms is a cell-config concern, not a regulation concern. A gemeente that executes Wahv but does not connect to MBO simply does not enable the `mbo_fcid` integration in its cell-config; the same Wahv lexogram works in both cells. This is why §5 puts the activation in cell-config, while the lexogram only declares `extensions.mbo_fcid.category` as a hint for when the cell does activate it.

The base schema thus permits an integration's `extensions` block on `produces` but does not require it. The presence of the block is a hint to integrating cells; the absence is a hint that no integration is relevant for this rule.

### 6. Cross-cell queries: `source.kind: lexostatus_query`

A regulation that needs data from another cell uses a new `source.kind` discriminator:

```yaml
input:
  - name: openstaande_vorderingen
    source:
      kind: lexostatus_query
      cell: cjib                          # which cell answers
      lexostatus: openstaande_vorderingen # which lexostatus to retrieve
      parameters:
        bsn: $bsn
```

This replaces the fake-regulation pattern from the first draft of this RFC. A `lexostatus_query` is not a regulation reference: it explicitly says "ask cell X for lexostatus Y". The producing cell does its own chronolexoreductie internally and returns a lexostatus. The consuming engine accepts the result through the RFC-009 ACCEPT path.

The discrimination on `source` happens via a new `kind` field with values `regulation` (today's default) and `lexostatus_query`. Existing source declarations that omit `kind` default to `regulation`, preserving backward compatibility.

### 7. `bezwaarbaar` affordance on decretograms and executograms

Nieuwland §7.2.1 ("wat uitwerkt als straf, verdient rechtsbescherming als straf") is taken as a design constraint, not as decoration. Every decretogram and every executogram carries a `bezwaarbaar` block declaring whether and how a belanghebbende can object, and on what timetable.

For decretograms, the block sits on `produces`:

```yaml
produces:
  legal_character: BESCHIKKING
  decision_type: BETALINGSVERPLICHTING
  bezwaarbaar:
    door: belanghebbende                # who may object
    via: cell.<organisation>.bezwaar    # which intake handles it
    termijn_dagen: 42                   # AWB 6:7 default; specific regulations override
    grondslag: "Awb 7:1"
```

For executograms, the block sits on the event in the chronicle-stream file (example in §3).

The `bezwaarbaar` block is required on every decretogram (engine refuses to load a regulation that produces a BESCHIKKING without it) and required on executograms that have legal consequences (the chronicle-stream validator refuses to load a stream whose events lack it unless explicitly marked `bezwaarbaar: niet_van_toepassing` with a reason field).

Concrete consequences:

- **Surface in lexostatus.** A lexostatus returned by a cell carries the `bezwaarbaar` block of every chronolexogram in it. A citizen-facing surface (whether MBO, MijnOverheid, or the engine in WASM) can render an objection-trigger next to the value.
- **Surface in trace.** The execution trace records the `bezwaarbaar` block alongside every BESCHIKKING node. A "samen zien" surface that shows the trace can therefore show, at every step that has consequences, what the objection route is.
- **Surface in inter-cell calls.** When cell A accepts a decretogram from cell B via FSC, the `bezwaarbaar` block travels with it. Cell A's downstream lexostatus exposes the original objection route to the citizen, not just cell A's own route.

This turns the citation of §7.2.1 into infrastructure rather than ornament.

### 8. Out of scope

- **Any specific outbound serialisation format.** Belongs in integration documents.
- **Any specific integration partner.** Belongs in integration documents and casus documents.
- **Trust and signing mechanics.** Already in RFC-009 §5.
- **Execution provenance and trace structure.** Already in RFC-013.
- **Federation transport.** Already in RFC-009 (FSC).
- **The full cell-config format.** This RFC introduces `chronicles/` and the integration-extension activation site; broader cell-config is for a future RFC.
- **The legal basis for inter-organisation data exchange.** Per-case, not at engine level.
- **A statutory Wet gegevensboekhouding** (Nieuwland §7.3.2). Legislative artefact; this RFC is an engineering one.

## Why

### Benefits

- **Lexogram, decretogram and executogram are explicit and separately located.** A future reader of any file sees the type for what it is from the path alone.
- **The cell is a juridical unit, not a binary.** Implementations can vary widely (one engine, multiple engines, engines plus legacy) without breaking the conceptual picture.
- **The financial-enforcement domain becomes expressible.** Three new `decision_type` values, one modality block. No mixing of types and modalities.
- **Integrations don't drag domain assumptions into the schema.** The `extensions` namespace mechanism lets integrations evolve without schema surgery.
- **Activation lives in the cell.** A gemeente can run the same Wahv lexogram as CJIB without forking it just because the gemeente doesn't emit FCID.
- **Cross-cell queries are first-class.** No fake-regulation adapters. `source.kind: lexostatus_query` says what it does.
- **`bezwaarbaar` is infrastructure.** Following Nieuwland §7.2.1 in design, not in citation.

### Tradeoffs

- **The `decision_type` enum extension is breaking for downstream consumers of the enum.** Schema bump to v0.6.0 is required.
- **Introducing `chronicles/` adds a second top-level repository directory.** Tooling needs to learn about it. One-time cost.
- **`extensions` blocks are deliberately uninterpreted at the schema level.** A reader still needs an integration document to know what `mbo_fcid.category: VERHOGING` means. The gain is that the schema is honest about this rather than pretending.
- **Activation-in-cell means an FCID-aware regulation YAML can be loaded by a non-FCID cell without effect.** Acceptable: presence is a hint, activation is a cell decision.
- **`bezwaarbaar` is required on decretograms and most executograms.** This is a constraint on authoring. Authors must consult the relevant Awb sections (typically 6:7, 7:1, 8:1) or the specific bezwaarregeling of the regulation being authored. We consider this a feature, not a bug.

### Alternatives Considered

**Alternative 1: Put executogram-streams under `corpus/`**.
- Rejected: conflates norm and registration. A registration-spec is not a regulation. Tooling that loads `corpus/**` would treat the two alike. The Chronolexografie distinction between registreren and interpreteren collapses at the file-system level.

**Alternative 2: Treat the engine itself as the cell**.
- Rejected: ties the cell-concept to one specific binary. Cells must be able to contain multiple components or none. The cell is juridical and organisational; the engine is one of its tools.

**Alternative 3: Two anonymous `outbound_emit` / `outbound_category` fields on `produces`**.
- Rejected (this was the first draft). Anonymous fields hide which integration drives them and force the reader to leave the schema to learn what they mean. A namespaced `extensions.<integration_name>` block makes the dependency explicit.

**Alternative 4: Put activation in the regulation (`fcid_emit: true` on the rule)**.
- Rejected: makes outbound serialisation a property of the law. A gemeente that runs the same wet but does not connect to MBO would have to fork the regulation. Activation belongs in the cell that runs the engine, not in the wet.

**Alternative 5: Keep the wrapper-regulation pattern (`procedureregeling_vorderingenoverzicht_rijk`)**.
- Rejected: a "procedureregeling" is a juridical genre; misusing the term for what is really an adapter invites confusion. `source.kind: lexostatus_query` says what it actually is.

**Alternative 6: Treat `INTREKKING_BESCHIKKING` as a `decision_type`**.
- Rejected: an intrekking is a modality on an existing besluit, not a new besluit-type. Captured via `produces.modality.is_intrekking_van`.

**Alternative 7: Treat `bezwaarbaar` as a UI concern, outside the schema**.
- Rejected: that is how rechtsbescherming-after-the-fact happens today. Nieuwland §7.2.1 explicitly calls for this to be ontwerp, not afterthought. Putting it in the schema means every authored regulation must answer the question.

### Implementation Notes

This RFC is the architecture decision. Schema and engine work are in follow-up PRs.

#### Phase 1: Schema

- Bump `schema/v0.5.2` → `schema/v0.6.0`. Update `schema/latest` symlink.
- Add new `decision_type` enum values: BETALINGSVERPLICHTING, STRAFBESCHIKKING, BESTUURLIJKE_BOETE.
- Add `produces.modality` optional block with `is_intrekking_van`, `is_wijziging_van`.
- Add `produces.extensions` as a free-form namespaced map.
- Add `produces.bezwaarbaar` as a required block when `legal_character: BESCHIKKING`.
- Add `schema/v0.6.0/chronicle.json` for chronicle-stream files.
- Add `source.kind` discriminator with `regulation` (default) and `lexostatus_query`.

#### Phase 2: Engine support

- New: `packages/engine/src/chronicle.rs` — loading, validating, and triggering chronicle-stream entries on external events.
- New: `packages/engine/src/types.rs` additions for `ChronicleStream`, `Modality`, `Bezwaarbaar`, `LexostatusQuerySource`.
- Update `packages/engine/src/resolver.rs` to dispatch on `source.kind`.

#### Phase 3: Vocabulary update in existing docs

- Update [concepts/law-format](/concepts/law-format): a regulation file is a lexogram.
- Update [concepts/multi-org-execution](/concepts/multi-org-execution): the engine is a component of a cell, not the cell itself.
- Update [concepts/execution-provenance](/concepts/execution-provenance): traces are chronicle-like.
- Add glossary entries for chronolexogram, lexogram, decretogram, executogram, lexostatus, cell, chronicle, reduction.

#### Phase 4: Pilot integration and first chronicle-stream

- One chronicle-stream YAML under `chronicles/` as a working example.
- One integration document under `docs/integrations/` using `extensions.<namespace>`. First integration is Mijn Betaaloverzicht / FCID; see [that document](/integrations/mbo-fcid).
