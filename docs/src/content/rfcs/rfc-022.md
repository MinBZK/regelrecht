## Context

RegelRecht turns Dutch law into executable specifications. Earlier RFCs have built up a vocabulary for doing this at scale: how a *beschikking* is modeled and who is competent to issue it (RFC-002, RFC-007), how delegated legislation is plugged in (RFC-003), how the AWB lifecycle structures the path from application to objection (RFC-008), how multiple organisations execute side by side and federate their corpora (RFC-009, RFC-010), and how each execution leaves an auditable trace (RFC-013). The set of regulations currently in machine-readable form is small relative to the Dutch corpus and grows opportunistically; what matters here is that the conceptual coverage is broad enough to host very different domains under one engine.

That coverage is still being built out. This RFC picks up three pieces that block a concrete next step. First, the financial-enforcement domain (Wahv, OM-strafbeschikking, sectoral *bestuurlijke boetes*) has no fitting `decision_type` and so cannot be expressed today. Second, there is no place in the repository to record factual events (payments received, *kwijtscheldingen*, *deurwaardertrajecten*); these are not regulation outputs and do not belong in `produces`. Third, there is no clean separation between **normative content** (regulations) and **registration specifications** (which facts a deployment records). Today no slot exists for the latter, so integrations either invent a local format or push the recording into a regulation YAML, conflating two different kinds of *vastlegging*.

In december 2025 the Denktank Achterkant van de Overheid published [Nieuwland](https://achterkantvandeoverheid.nl/) and the [Chronolexografie position paper](https://chronolexografie.nl/) (Olthof & Van Andel). Together they offer a coherent vocabulary for digitally recording the *rechtstoestand*: three types of recording (*lexogram*, *decretogram*, *executogram*), held by an organisational and juridical unit called a *chronolexocel*. The vocabulary matches the architecture RegelRecht is already building.

This RFC introduces:

1. The three chronolexogram types as first-class concepts in the schema and repository layout.
2. The cell as a juridical unit (the engine being one possible component of a cell, not the cell itself).
3. Two cross-cutting fields on chronolexograms: modality and integration extensions.
4. Cross-cell queries by reusing the existing `source` block, with the runtime selecting the right transport per context (Blauwe Knop for citizen-clients, FSC for authorised servers).

The RFC is neutral about any specific external standard at the schema level. The first integration it anticipates is the [Blauwe Knop Connect](https://vorijk.nl/standaard/connect/draft-bk-connect-00.html) standard that Mijn Betaaloverzicht is built on; a cell that activates the `blauwe_knop_source` cell-config block serves an FCID-response on a citizen-pull, fully honouring the data-at-source, on-device-aggregation principle of Blauwe Knop. Concrete integration shape lives in the [CJIB pilot proposal](/concepts/cjib-mbo-bridge-proposal); future standards that follow the same pattern (and others that do not) can plug in via the same `extensions`-namespace mechanism.

### RFCs this proposal amends

This RFC reuses two existing mechanisms in ways that ask for a small canonical amendment to the corresponding RFCs. Until those amendments land, the resolver- and signing-decisions specified in §4 and §3.2 are accepted pragmatically:

- **RFC-007** (Cross-Law Execution): the `source.regulation` resolver semantics are extended so that a value naming a *cell* (not a regulation in the loaded corpus) dispatches via the runtime-selected cross-cell transport. RFC-007 currently knows only the cross-law case.
- **RFC-009** (Multi-Org Execution / FSC): the same private key the cell uses for FSC mTLS signing is reused as the JWS-signing key for Blauwe-Knop FCID-responses. RFC-009 is written with FSC-mTLS in mind; the canonical amendment will document the reuse and any constraints (key-rotation alignment, separate key pairs if a deployment prefers segregation).

Neither extension changes the schema or breaks RFC-007 / RFC-009 in their existing use; both are additive. The amendments land as follow-up commits after the CJIB pilot workshop confirms the operational details.

This RFC **does not** introduce a `bezwaarbaar` field. An earlier draft did so. Closer reading of [RFC-008](./rfc-008) showed that the AWB lifecycle (with `BESLUIT`, `BEKENDMAKING`, `BEZWAARTERMIJN`, `BEZWAAR` stages, AWB-3:46/6:7/6:8 hooks, and explicit handling of nested besluiten op bezwaar) already models everything a `bezwaarbaar` field would have duplicated, and does so more correctly: a bezwaartermijn cannot start before bekendmaking, which a per-rule field cannot express. The route from a chronolexogram back to its bezwaar-stage is derived through RFC-008, not declared on the rule. §3.3 below specifies how.

### Design principles

1. **The corpus is the lexogram store.** Only normative content lives in `corpus/regulation/`.
2. **The chronicle store is separate from the corpus.** Registration specifications live in `chronicles/`. A registration-spec is not a regulation.
3. **A decretogram is an engine output with `legal_character: BESCHIKKING`.** New `decision_type` values cover the financial-enforcement domain.
4. **An executogram is not a regulation output.** It does not belong in `produces`. It is declared in a chronicle-stream file.
5. **A cell is a juridical unit.** The engine is one possible component of a cell.
6. **Integration configuration lives in the cell, not in the law.** Where outbound configuration must touch the regulation, it is namespaced under `extensions` so the dependency is explicit.
7. **Rechtsbescherming uses the AWB lifecycle from RFC-008**, not a parallel mechanism in this RFC.
8. **Citizen-side aggregation is preserved.** Where a regulation queries another cell, the runtime selects Blauwe Knop (citizen-client) or FSC (authorised server) on the basis of the cell-context; the schema does not centralise data into a single ground-truth point.

## Terminology

This RFC adopts the vocabulary from the [Chronolexografie](https://chronolexografie.nl/) position paper. Most terms (*lexogram*, *decretogram*, *executogram*, *chronicle*, *reduction*, *lexostatus*) are introduced in context where they are first needed; full definitions live in the [glossary](/reference/glossary).

One distinction needs to be set straight up front because it is easy to misread:

A **chronolexocell** ("cell") is the *juridical and organisational domain* that holds chronicles, owns signing keys, and is the competent authority that other cells contract with via RFC-009 FSC. The cell's identity *is* its `competent_authority` per [RFC-002](./rfc-002); the cell concept simply names the unit that bears that authority across both regulation execution and chronicle-stream recording.

A **RegelRecht engine** is a *computational component* that can run inside a cell. The cell is not the engine, the engine is not the cell. A cell may contain one engine, several engines, an engine alongside a legacy system, or no engine at all. Throughout this RFC, where the difference matters, "cell" refers to the juridical unit and "engine" refers to the binary.

## Decision

### 1. Three chronolexogram types

Chronolexografie distinguishes three types of vastlegging. RegelRecht maps each to a distinct place in the schema and repository layout.

#### 1.1. Lexograms — already exist

A *lexogram* is the recording of a (possible, future) *wijziging in wet- of regelgeving*. A YAML file under `corpus/regulation/<jurisdiction>/<type>/<name>/<valid_from>.yaml` is a lexogram. The sequence of `valid_from` versions for one regulation is its lexogram-chronicle.

No new schema is needed. The contribution here is documentational: existing concepts pages get one paragraph each calling this what it is.

#### 1.2. Decretograms — `decision_type` extension

A *decretogram* **is** an engine output with `produces.legal_character: BESCHIKKING`. The decretogram is the primary artefact; the execution trace from RFC-013 is its body; the cell's FSC signature from RFC-009 §5 is its signature. Wire-formats that an integration serialises (an FCID-record in a Blauwe-Knop-response, an FSC-payload to another cell) are *serialisations* of the decretogram, not derived artefacts: there is no second representation of the decretogram in the cell, only different on-the-wire shapes for different consumers.

A decretogram is **lifecycle-aware** through RFC-008. The `procedure_id` field on `produces` (introduced by RFC-008) selects the AWB procedure under which the BESCHIKKING is taken. The decretogram is not a single emission; it accumulates outputs through RFC-008's stages (BESLUIT, BEKENDMAKING, BEZWAARTERMIJN, ...). Integrations that surface chronolexograms (such as a cell-as-Blauwe-Knop-source serialising FCID) bind to specific stages; see §3.2 and the Blauwe-Knop-source appendix in the [CJIB Blauwe-Knop pilot proposal](/concepts/cjib-mbo-bridge-proposal) for the concrete mapping.

The existing `produces.decision_type` enum has nine values (TOEKENNING, AFWIJZING, GOEDKEURING, GEEN_BESLUIT, ALGEMEEN_VERBINDEND_VOORSCHRIFT, BELEIDSREGEL, VOORBEREIDINGSBESLUIT, ANDERE_HANDELING, AANSLAG). None describes the financial-enforcement domain. This RFC adds three values, each a distinct *type* of decretogram:

- `BETALINGSVERPLICHTING` — generic financial obligation imposed by a *bestuursorgaan*
- `STRAFBESCHIKKING` — criminal-law settlement under Wetboek van Strafvordering art. 257a
- `BESTUURLIJKE_BOETE` — sectoral administrative fine

That is the entire extension. Three values, deliberately small.

Two earlier candidates are excluded:

- `INCASSO_BESCHIKKING` conflated an *executoriale titel* (Awb 4:114 *dwangbevel*) with a primary *besluit*. A *dwangbevel* is closer to an executogram than to a fresh BESCHIKKING; until a concrete case appears that needs a label for a *dwangbevel*-as-decretogram, we do not invent one.
- `INTREKKING_BESCHIKKING` treated a *modality* as a *type*. An *intrekking* is a nested decision in the sense of RFC-008 (Open Question 5, resolved): the *intrekking* is itself a BESCHIKKING with its own AWB lifecycle, that as a side effect changes the original *beschikking*'s state. The schema does not need a separate `decision_type` for this; the *intrekking*-*besluit* is just another BESCHIKKING that references the original via `modality.is_intrekking_van` (§3.1).

#### 1.3. Executograms — `chronicles/` directory

An *executogram* is the recording of *feitelijke levering of afhandeling*: a payment received, a *kwijtschelding* granted, a service delivered. An executogram is **not** a regulation output; it does not belong in any `produces` block. Like a decretogram, an executogram is its own primary artefact (an entry in a chronicle-stream); FCID and other wire-formats are serialisations of that entry, not separately stored derivations.

Executogram-stream definitions live in a new top-level directory `chronicles/`, parallel to `corpus/`. A chronicle-stream file declares: which facts a cell records, who is competent for them, how each is named, what fields it carries. Minimal example:

```yaml
$id: payments_received
competent_authority: <organisation-id>
chronicle: cjib_main_chronicle      # which chronicle these events join
events:
  - name: payment_received
    intake: external_intake          # where the event enters the cell
    fields:
      case_reference: $external.case_reference
      amount_cents: $external.amount_cents
      received_at: $external.received_at
```

A separate JSON schema for chronicle-stream files lives at `schema/v0.6.0/chronicle.json`.

The relocation matters. A regulation is normative content. A chronicle-stream is a registration spec. Putting them in the same tree invites tooling to treat one as the other, and invites readers to think "the law for recording payments". A chronicle-stream is no such thing.

The name `chronicles/` is chosen because the content is closer to "which events does the cell's chronicle accept" than to "what does the cell do" (which would suggest a broader cell-config). Future RFCs may introduce a wider `cell-config/`; for now `chronicles/` is the minimum needed.

**Anticipating a Wet gegevensboekhouding.** Nieuwland §7.3.2 sketches a future *Wet gegevensboekhouding* that would put cell-side registration of *welke* data is processed, *door wie*, *op welke grondslag* on a statutory footing. The `chronicles/` mechanism is deliberately shaped to be compatible: each chronicle-stream entry records *what* the cell records, *by whom*, on *which grondslag*, at *which moment*. Today this is an engineering convention; if the law lands, the same files become the cell's *gegevensboekhouding* artefact with no schema rework. This is not a claim about whether the law will be enacted; it is a constraint on the schema so that we do not paint ourselves out of the option.

**Federation and reproducibility.** Chronicle-stream files participate in the same federated-corpus mechanism as regulations (RFC-010): a cell can host its chronicle-streams in its own repository alongside the regulations it implements. They also participate in the [RFC-013](./rfc-013) Execution Receipt mechanism on the same footing as regulations: when a chronicle-stream contributes to an execution (typically as the source of `BetalingVerwerkt`-style events that an engine consults during reduction), its content-hash and version are recorded in the Receipt's `scope.loaded_artifacts`. Without this, an execution that depends on a chronicle-stream cannot be reproduced months later.

### 2. Naming the cell: it is RFC-002's `competent_authority`

The Chronolexografie *chronolexocell* is the same juridical unit that RFC-002 calls `competent_authority` and that RFC-009 represents at runtime as `EngineIdentity`. This RFC introduces no new concept; it adopts the Chronolexografie name to describe what is already in the schema and engine.

Picking up the existing pieces, the cell:

- **Holds** one or more chronicles. A chronicle is an ordered record of the cell's chronolexograms (decretograms and executograms); typical CJIB topology is several chronicles (Wahv-decisions, OM-uitvoering, mandate-executions, betalingen, kwijtscheldingen), not one centralised log. Exact topology is a per-cell deployment choice.
- **Owns** the signing keys used for chronolexograms produced from it (RFC-009 §5). The same key signs FCID-responses (Blauwe Knop) and FSC-responses; per the amendment noted in Context, a deployment may opt for separate keys.
- **Is** the `competent_authority` (RFC-002) that other cells see and contract with via RFC-009 FSC. The cell encapsulates: that competent_authority, plus the signing keys, plus the chronicle store, plus the lexostatus capability declarations (see §4).
- **Declares** which lexostatussen it exposes and how each is reduced from its chronicles. This is the cell's `capabilities` artefact; see §4.
- **Runs** zero or more engines that evaluate regulations and accumulate decretograms into its chronicles. A cell that runs no engine but a legacy system is also a cell.

A RegelRecht engine is one component a cell can use, alongside others (a legacy execution system, a payment intake system, a *deurwaarder* workflow tool). RFC-009 supports a deployment where one engine equals one cell, where multiple engines share a cell, or where a cell mixes engine and non-engine components; this RFC does not change those choices.

The vocabulary mapping between Chronolexografie and the existing RegelRecht concepts:

| Chronolexografie | RegelRecht | RFC |
|---|---|---|
| chronolexocell | `competent_authority` + signing keys + chronicles | RFC-002, RFC-009 |
| chronolexochronicle | the cell's trace store + chronicle-streams | RFC-013, §1.3 |
| chronolexogram | one decretogram (engine output) or one executogram (chronicle-stream event) | §1.2, §1.3 |
| chronolexoreduction | a filter/aggregate over one or more chronicles producing a lexostatus | §4 |
| lexostatus | the *result* of a chronolexoreduction; the canonical answer-shape for a cross-cell query | §4 |
| chronolexosphere | the federation of cells | RFC-009 federated mode |

### 3. Two cross-cutting fields, plus *rechtsbescherming* via RFC-008

Two fields apply to chronolexograms regardless of type. They are introduced here together because they share a pattern: each travels with the chronolexogram when it is signed, propagated, or surfaced. The third concern, *rechtsbescherming*, is *not* a new field but a derivation through RFC-008; §3.3 explains.

#### 3.1. `modality`: a backlink within RFC-008's nested-procedure model

RFC-008 already settled the semantics of *intrekking*, *wijziging*, and *herziening* (Open Question 5, resolved): an *intrekking* is itself a fresh BESCHIKKING with its own AWB lifecycle, that as a side effect changes the original *besluit*'s state. RFC-022 adds no new semantics here. What it adds is a small **backlink field** so a downstream consumer (citizen surface, integration) can quickly find the original *besluit* a new BESCHIKKING modifies, without having to reconstruct the relation from procedure-state.

```yaml
produces:
  legal_character: BESCHIKKING
  decision_type: BETALINGSVERPLICHTING
  modality:
    is_intrekking_van: "<beschikking-id>"   # optional backlink, per RFC-008
    is_wijziging_van:  "<beschikking-id>"   # optional backlink, per RFC-008
```

This avoids the temptation to invent INTREKKING_BESCHIKKING as a `decision_type` (which would have duplicated RFC-008 and conflated modality with type). The same backlink is allowed on executograms in chronicle-stream files, for events such as `kwijtschelding_verleend` that effectively *intrekken* an earlier *vordering*; the executogram references its underlying decretogram via the same field.

#### 3.2. `extensions` — namespaced integration hooks

Integrations need to attach configuration to chronolexograms without baking domain assumptions into the base schema. `extensions` is a namespaced map. Each integration owns one namespace key; the integration document for that namespace owns the shape and semantics inside its block.

The first integration this RFC anticipates is `blauwe_knop`: the cell announces that it is willing to surface this rule's decretograms (or this chronicle-stream's events) in its [Blauwe Knop](https://vorijk.nl/standaard/connect/draft-bk-connect-00.html) source-endpoint. Mijn Betaaloverzicht is the most visible Blauwe-Knop-client today; other citizen-side aggregators following the same standard would consume the same source-endpoint.

```yaml
produces:
  legal_character: BESCHIKKING
  decision_type: BETALINGSVERPLICHTING
  extensions:
    blauwe_knop:
      payload: fcid                       # for now always fcid; future-proof
      category: ALGEMEEN
      visible_from_stage: BEKENDMAKING    # optional; defaults per integration
```

The same shape applies to chronicle-stream events:

```yaml
events:
  - name: payment_received
    extensions:
      blauwe_knop:
        payload: fcid
        event_type: BetalingVerwerkt
        category: ALGEMEEN
```

A reader of the schema sees that there is extension-driven behaviour, and which integration drives it, without the base schema having to know what `category: ALGEMEEN` means.

**Stage-aware visibility, not stage-aware emission.** Blauwe Knop is a pull-based protocol: a citizen-client authenticates (via DigiD + the App Manager) and pulls the cell's FCID-response on demand. There is no push, no central destination, no emit-moment. The relevant RFC-008 lifecycle binding is therefore *visibility*, not *emission*: a `BetalingsverplichtingOpgelegd` becomes visible in the cell's FCID-response from `BEKENDMAKING` onward, because a *verplichting* that the citizen has not yet been notified of has no juridical existence to display in a *betaaloverzicht*.

**The visibility check is a hard guard, not a hint.** On every pull, the cell-runtime evaluates `current_stage >= visible_from_stage` per candidate decretogram and excludes any decretogram that does not satisfy the guard. Excluded decretograms simply do not appear in the response: no placeholder, no null record, no partial `bezwaar_route`. This is mandatory because the `bezwaar_route`-fields (`termijn_einddatum` in particular) are only populated by RFC-008 hooks at the BEKENDMAKING-stage; serving a decretogram before that stage would either omit those fields or fall back to defaults, both of which would mislead the citizen about their objection window. Lowering `visible_from_stage` below BEKENDMAKING is therefore not permitted for decretograms that require a `bezwaar_route` (which is all financial-enforcement decision types this RFC adds). By convention each integration accepts a stage-binding field inside its own namespace with a sensible default in the integration document; the base schema does not validate the stage value, but the cell-runtime is required to enforce the guard.

**Activation lives in the cell, not in the law.** Whether a cell actually serves a Blauwe-Knop-source endpoint for its decretograms is a cell-config concern, not a regulation concern. A *gemeente* that runs the same Wahv as CJIB but does not run a Blauwe-Knop-source simply omits the `blauwe_knop_source` block from its cell-config; the same lexogram works in both cells. The presence of an `extensions.blauwe_knop` block on a rule is a *hint* for cells that do activate the source; the absence is a hint that this rule is not meant to surface in any citizen-side aggregator. Note the framing: Blauwe Knop has no concept of toggleable "roles"; `blauwe_knop_source` is simply a direct cell-config feature-block that declares the endpoint, FCID-version, signing-key reference, and which lexostatus the cell serves on the source.

#### 3.3. Rechtsbescherming: derived from the AWB lifecycle (RFC-008), not a new field

An earlier draft of this RFC introduced a per-rule `bezwaarbaar` block carrying `door`, `via`, `termijn_dagen`, `grondslag`. That was wrong.

RFC-008 already models *bezwaar* as part of the AWB lifecycle. The *bezwaartermijn* is computed at `BEKENDMAKING` (AWB 6:8 hooks fire there, not at BESLUIT, because the termijn cannot start before notification). The bezwaar-stage runs after BEKENDMAKING. The route to file bezwaar is governed by AWB-7:1 plus the specific *bezwaarregeling* of the regulation. A nested *besluit op bezwaar* is itself a BESCHIKKING with its own AWB lifecycle.

A per-rule `termijn_dagen: 42` would have been a static hint at BESLUIT-time of a value that RFC-008 correctly computes at BEKENDMAKING-time. A static `via: cell.cjib.bezwaar` would have duplicated routing that the AWB-procedure of the cell already determines. The right design is: the *rechtsbescherming*-route accompanying a chronolexogram is **derived** from the procedure that produced it, not declared on the rule.

Concretely:

- **For decretograms.** When a cell emits an outbound event for a decretogram, it includes a `bezwaar_route` field. This field is computed by the engine from the RFC-008 procedure of the producing rule: `bezwaar_route.intake` = the cell's bezwaar-intake for that procedure, `bezwaar_route.termijn` = the *bezwaartermijn*-output that AWB-6:7/6:8 hooks have produced at BEKENDMAKING (so it carries an actual *einddatum*, not a duration). For UOV procedures (RFC-008 §A.8), `bezwaar_route` is absent and the cell instead includes `beroep_route` per AWB-7:1 lid 1 sub d. For BESLUIT_VAN_ALGEMENE_STREKKING, neither is present except for "concretiserend BAS" (RFC-008 §A.9).

- **For executograms.** An executogram event is a *feit*, not a *besluit*. It carries no `bezwaar_route` by default. A specific executogram (such as `kwijtschelding_verleend`) may carry one *because the event implicitly references a nested besluit*; in that case the chronicle-stream event declares `references_decision: <decision-id>` and the engine derives `bezwaar_route` from that decision's procedure.

- **Surface in lexostatus and trace.** A lexostatus returned by a cell includes the `bezwaar_route` for each chronolexogram. The execution trace records, at each BEKENDMAKING-stage node, the computed *bezwaartermijn* (this already follows from RFC-008 hooks on AWB-6:7/6:8). Citizen-facing surfaces (MBO, MijnOverheid, the engine in WASM) can render the objection-trigger inline.

- **Cross-cell propagation.** When cell A accepts a decretogram from cell B via FSC, the `bezwaar_route` travels with it. Cell A's downstream lexostatus exposes the *original* objection route to the citizen, not just cell A's own route.

This RFC's contribution to *rechtsbescherming* is therefore not a new field, but the requirement that integrations carry the route in their payloads, derived from RFC-008's lifecycle output. The Blauwe-Knop-source appendix in the CJIB pilot proposal specifies the concrete shape of the `bezwaar_route` it serialises into an FCID-response.

### 4. Cross-cell queries: lexostatus, reuse `source`, runtime chooses the transport

#### 4.1 Lexostatus and chronolexoreductie

A **lexostatus** is the canonical answer-shape a cell offers to consumers. It is the *result* of a *chronolexoreductie*: a filter-and-aggregate over one or more of the cell's chronicles, with rules for selection, exclusion, grouping, and ordering. A lexostatus is not an output of a single regulation execution; it is a cell-level capability that may combine outputs from multiple regulations and chronicle-streams.

Each cell declares which lexostatussen it exposes and how each is reduced. The declaration is a cell-level capability artefact (provisional shape; the full cell-config format is for a future RFC):

```yaml
# cells/<cell-id>/capabilities.yaml (sketch)
lexostatus_definitions:
  - name: openstaande_vorderingen
    description: >
      All open payment obligations for the given BSN, drawn from the
      relevant decision-chronicles, filtered on procedure-stage and
      net of withdrawals.
    inputs:
      - name: bsn
        type: string
    reduction:
      sources:
        - chronicle: <chronicle-id>
          select_where: "decision_type IN (BETALINGSVERPLICHTING) AND current_stage >= BEKENDMAKING"
        # ... possibly more chronicles
      exclude:
        - "modality.is_intrekking_van IS NOT NULL OR has_intrekking_in_chronicle = true"
      group_by: zaakkenmerk
      order_by: bekendmaking_datum DESC
```

The cell-runtime ships the lexostatus result in the transport-specific format: an FCID-response over Blauwe Knop, an ACCEPT-payload over FSC. Same reduction, different serialisations. The Blauwe-Knop-source response is itself one such lexostatus (it can be named explicitly in capabilities so the source-endpoint and the cross-cell-query path are unified, not parallel).

The cell is the authority for its own lexostatussen. A consumer cannot inject a custom reduction; it can only ask for a published lexostatus and supply the documented parameters. This keeps the reduction logic with the cell that owns the chronicles, which is the only place where the right filter set is knowable.

#### 4.2 Source resolution

A regulation that needs data from another cell uses the existing `source` block from RFC-007. No new fields, no `kind` discriminator: the regulation just names the producing cell and the lexostatus it wants.

```yaml
input:
  - name: openstaande_vorderingen
    source:
      regulation: cjib                       # cell-id, resolved by the runtime
      output: openstaande_vorderingen        # the lexostatus the cell exposes
      parameters:
        bsn: $bsn
```

Resolution semantics: `source.regulation` names something the runtime resolves. When it resolves to a regulation in the loaded corpus, the engine evaluates it as today (RFC-007). When it resolves to another cell, the engine performs a cross-cell query for the named lexostatus. When neither, error. Same syntax in all cases; the schema does not change. (This extends RFC-007's resolver semantics; see the amendment note in Context.)

#### 4.3 Transport selection

What does change is **how the cross-cell query travels**. Two transports are supported, depending on the context in which the engine runs:

- **Citizen-client context (Blauwe Knop).** When the engine runs inside a citizen-client (RegelRecht-WASM in a browser, a mobile app), the cross-cell query is resolved via [Blauwe Knop Connect](https://vorijk.nl/standaard/connect/draft-bk-connect-00.html). The engine acts as a Blauwe-Knop-client: the citizen has a DigiD session, the engine pulls the cell's Blauwe-Knop-source endpoint, receives a signed FCID-response, verifies the signature against the App Manager's trust material, and returns the result. No server-side aggregation; the data stays on-device.

- **Authorised-server context (FSC).** When the engine runs on a server of a *bevoegde instantie* with a specific statutory ground (or an explicit citizen mandate) to query another cell, the cross-cell query is resolved via FSC. The engine follows the RFC-009 ACCEPT path: the receiving cell verifies the FSC-signed call and the asserted ground, and answers from its own internal *chronolexoreductie* for the requested lexostatus.

The selection between transports is a property of the cell-runtime, not of the regulation. A cell-config with a `blauwe_knop_client` block resolves cross-cell queries via Blauwe Knop. A cell-config with an `fsc_client` block resolves them via FSC. These are direct cell-config feature-blocks (not Blauwe-Knop-defined "roles"; the standard has no role-concept) that declare client behaviour. A cell that has both blocks (rare; for example a *bevoegde instantie* that also embeds a citizen-facing tool) selects on the basis of the present authorisation context. The same regulation YAML therefore works unchanged in a citizen-application, in a *gemeente*-systeem, or in a research environment that mocks both transports.

Blauwe Knop and FSC coexist in the Common-Ground landscape. They are not alternatives; they cover different juridical contexts. Blauwe Knop is citizen-to-source under the data-at-source principle Nieuwland §5 articulates: the citizen aggregates on-device, no organisation sees the combined picture. FSC is organisation-to-organisation under a statutory ground or a recorded mandate. A source-cell (such as CJIB for *openstaande vorderingen*) typically hosts both endpoints, with different access-control regimes for each, both serving the same underlying lexostatus.

The earlier draft of this RFC made two simpler claims that were both wrong and have been replaced:

- Rejected wrapper *regulation* in `corpus/regulation/` (`procedureregeling_vorderingenoverzicht_rijk`) as the adapter. That placed a non-regulation in the regulation tree under a misleading juridical genre name.
- Rejected a single discriminator (`source.kind: lexostatus_query`) with fresh `cell:` and `lexostatus:` fields. That invented schema for what `source.regulation` and `source.output` already express.
- Rejected an FSC-only resolution. Treating every cross-cell query as a server-to-server call was the right answer for the *bevoegde instantie* context but the wrong answer for citizen-facing tools. The on-device aggregation principle that Blauwe Knop implements is non-negotiable in the citizen-context; the runtime-selects-the-transport pattern above keeps both contexts first-class.

### 5. Out of scope

- **Any specific outbound serialisation format.** Belongs in integration documents.
- **Any specific integration partner.** Belongs in integration documents and casus documents.
- **The AWB lifecycle, *bezwaar* mechanics, nested *besluiten op bezwaar*.** Already in RFC-008. This RFC builds on RFC-008's procedure model; it does not redefine it.
- **Trust and signing mechanics.** Already in RFC-009 §5.
- **Execution provenance and trace structure.** Already in RFC-013.
- **Federation transport.** Already in RFC-009 (FSC).
- **Federated corpus distribution.** Already in RFC-010, applied here to chronicle-streams as well.
- **The full cell-config format.** This RFC introduces `chronicles/` and the integration-extension activation site; broader cell-config is for a future RFC.
- **The legal basis for inter-organisation data exchange.** Per-case, not at engine level.
- **A statutory Wet gegevensboekhouding** (Nieuwland §7.3.2). Legislative artefact; this RFC is an engineering one.

## Why

### Benefits

- **Lexogram, decretogram and executogram are explicit and separately located.** A future reader sees the type from the path alone.
- **The cell is a juridical unit, not a binary.** Implementations vary widely (one engine, several engines, engine plus legacy) without breaking the conceptual picture.
- **The financial-enforcement domain becomes expressible.** Three new `decision_type` values, one modality block. No mixing of types and modalities.
- **Integrations don't drag domain assumptions into the schema.** The `extensions` namespace mechanism lets integrations evolve without schema surgery.
- **Activation lives in the cell.** A *gemeente* can run the same Wahv lexogram as CJIB without forking it just because the *gemeente* doesn't emit FCID.
- **Cross-cell queries are first-class.** No fake-regulation adapters.
- ***Rechtsbescherming* reuses RFC-008.** No parallel mechanism. The objection route on an emitted event is derived from the AWB procedure that produced the *besluit*, computed at the right lifecycle stage.
- **Chronicle-streams are reproducible.** They participate in RFC-013 Execution Receipts, so executions that depend on them can be replayed.

### Tradeoffs

- **The `decision_type` enum extension is breaking for downstream consumers of the enum.** A schema bump (proposed: v0.5.2 → v0.6.0) is required as part of this RFC's acceptance and Phase 1 implementation.
- **Introducing `chronicles/` adds a second top-level repository directory.** Tooling needs to learn about it. One-time cost.
- **`extensions` blocks are deliberately uninterpreted at the schema level.** A reader still needs an integration document to know what `blauwe_knop.category: VERHOGING` means. The gain is that the schema is honest about this rather than pretending.
- **Activation-in-cell means an FCID-aware regulation YAML can be loaded by a non-FCID cell without effect.** Acceptable: presence is a hint, activation is a cell decision.
- **Integrations must understand RFC-008 stages to decide visibility at the right moment.** This is a constraint on integration authors. The Blauwe-Knop-source appendix in the CJIB pilot proposal demonstrates the pattern.
- **The proposed schema-version bump** (v0.5.2 → v0.6.0) is small because all changes are additive: new enum values (existing values unchanged), new optional `modality` and `extensions` blocks, and a new chronicle JSON schema living alongside the existing schema. Cross-cell queries require no schema change at all; they reuse the existing `source` block with a registry-driven resolution rule. No existing regulation YAML breaks. The bump itself lands in Phase 1; until then, v0.5.2 remains the validated schema.

### Alternatives Considered

**Alternative 1: Put executogram-streams under `corpus/`**.
- Rejected: conflates norm and registration. A registration-spec is not a regulation. Tooling that loads `corpus/**` would treat the two alike. The Chronolexografie distinction between *registreren* and *interpreteren* collapses at the file-system level.

**Alternative 2: Treat the engine itself as the cell**.
- Rejected: ties the cell-concept to one specific binary. Cells must be able to contain multiple components or none. The cell is juridical and organisational; the engine is one of its tools.

**Alternative 3: Two anonymous `outbound_emit` / `outbound_category` fields on `produces`**.
- Rejected. Anonymous fields hide which integration drives them and force the reader to leave the schema to learn what they mean. A namespaced `extensions.<integration_name>` block makes the dependency explicit.

**Alternative 4: Put activation in the regulation (`fcid_emit: true` on the rule)**.
- Rejected: makes integration-participation a property of the law. A *gemeente* that runs the same *wet* but does not run a Blauwe-Knop-source would have to fork the regulation. Activation belongs in the cell-config (the `blauwe_knop_source` block) that the cell runs the engine in, not in the *wet*.

**Alternative 5: Keep the wrapper-regulation pattern (`procedureregeling_vorderingenoverzicht_rijk`)**.
- Rejected: a "procedureregeling" is a juridical genre; misusing the term for what is really an adapter invites confusion. The reused-source approach (§4) is the right answer.

**Alternative 6: Treat `INTREKKING_BESCHIKKING` as a `decision_type`**.
- Rejected: an *intrekking* is a nested *besluit* per RFC-008 (its own AWB lifecycle), not a new *besluit*-type. Captured via `produces.modality.is_intrekking_van`.

**Alternative 7: Introduce a per-rule `bezwaarbaar` block on `produces`** (this was an earlier draft of this RFC).
- Rejected. The AWB lifecycle (RFC-008) already models *bezwaar* correctly, including the timing constraint that the *termijn* starts at BEKENDMAKING (not BESLUIT) and the nesting of *besluiten op bezwaar*. A per-rule field would have duplicated this badly: a static `termijn_dagen: 42` cannot reflect that the *einddatum* is computed from `bekendmaking_datum + AWB-6:7 + Termijnenwet`. The right derivation is via RFC-008 hooks; this RFC's contribution is to require integrations to *carry* the resulting route in the payloads they serialise (such as the `bezwaar_route` in a Blauwe-Knop FCID-response).

**Alternative 8: Treat every cross-cell query as a server-to-server FSC call** (earlier wording in §4).
- Rejected. That answer was right for the *bevoegde-instantie* context but wrong for citizen-facing tools. Routing a citizen-application's lookup of *openstaande vorderingen* through a server-to-server federation would centralise data that MBO's Blauwe-Knop architecture deliberately keeps decentralised. The runtime-selects-transport pattern in §4 keeps both contexts first-class while leaving the regulation YAML transport-agnostic.

**Alternative 9: Treat the Blauwe-Knop-source as a push-emit (`emit_at_stage`)**.
- Rejected. Blauwe Knop has no central destination to push to; the architecture is pull-based by design (citizen-client authenticates, source serves on demand). The stage-binding that matters is *visibility* at pull-time, not *emission*. `visible_from_stage` reflects this; an integration that genuinely is push-based (a future internal monitoring system, say) can define its own stage-binding field inside its own `extensions` namespace.

### Implementation Notes

This RFC is the architecture decision. Schema and engine work are in follow-up PRs.

#### Phase 1: Schema

- Bump `schema/v0.5.2` → `schema/v0.6.0`. Update `schema/latest` symlink.
- Add new `decision_type` values: BETALINGSVERPLICHTING, STRAFBESCHIKKING, BESTUURLIJKE_BOETE.
- Add `produces.modality` optional block.
- Add `produces.extensions` as a free-form namespaced map.
- Add `schema/v0.6.0/chronicle.json` for chronicle-stream files.
- No schema change for cross-cell queries; the existing `source` block is reused, with runtime-context-driven transport selection (Phase 2).
- All changes additive; no existing regulation YAML breaks.

#### Phase 2: Engine support

- New: `packages/engine/src/chronicle.rs` — loading, validating, and triggering chronicle-stream entries.
- New types in `packages/engine/src/types.rs`: `ChronicleStream`, `Modality`.
- Update `packages/engine/src/resolver.rs`: when resolving a `source.regulation` value, check whether it names a cell; if so, dispatch via the cell-context's transport (Blauwe Knop for cells with a `blauwe_knop_client` config block, FSC for cells with an `fsc_client` block); otherwise fall through to today's regulation-resolution. Each cell-context evaluates the requested lexostatus via the receiving cell's declared chronolexoreductie.
- New: `packages/engine/src/cell_features/blauwe_knop_source.rs` — when activated by the cell-config block, exposes a Blauwe-Knop-source endpoint that, on a verified citizen-pull, runs the cell's declared lexostatus-reduction, **enforces the `visible_from_stage` guard per candidate decretogram**, and serialises an FCID-response with the cell's signing key.
- New: `packages/engine/src/cell_features/blauwe_knop_client.rs` — when activated by the cell-config block, resolves cross-cell queries by pulling the named cell's Blauwe-Knop-source on behalf of the active citizen session.
- New: `packages/engine/src/lexostatus.rs` — loading cell `capabilities.yaml` declarations, executing chronolexoreductions, serialising to the requested transport format.
- Update RFC-013 Execution Receipt builder to include loaded chronicle-streams and lexostatus-definitions alongside regulations.

#### Phase 3: Vocabulary

- Update [law-format](/concepts/law-format), [multi-org-execution](/concepts/multi-org-execution), [execution-provenance](/concepts/execution-provenance) to use the cell/chronicle/lexogram vocabulary.
- Add glossary entries.

#### Phase 4: Pilot

- One chronicle-stream YAML under `chronicles/` as a working example.
- One integration spec using `extensions.<namespace>`. First integration is `blauwe_knop` (a cell-as-source for the Blauwe Knop Connect standard, with FCID as the payload type); see the Blauwe-Knop-source appendix of the [CJIB pilot proposal](/concepts/cjib-mbo-bridge-proposal) for stage-binding, `bezwaar_route` derivation, and field-level mapping.
