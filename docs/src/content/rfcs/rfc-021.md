---
title: "RFC-021: Canonical Law-Model Crate and Schema/Type Codegen"
---

**Status:** Proposed
**Date:** 2026-06-11
**Authors:** Tim de Jager
**Short title:** Canonical Law Model

## Context

A repo-wide architecture audit (2026-06-11) found that the law-YAML document
model — the shape of the files in `corpus/regulation/` — is implemented **six
times**, and that a law-format change requires touching **seven to nine
uncoordinated places** with no code generation connecting any of them.

### Finding 1: six implementations of the same model

| # | Location | What it is |
|---|----------|------------|
| 1 | `packages/engine/src/article.rs` (~1,900 lines) | The canonical, full serde model: ~30 structs/enums (`ArticleBasedLaw`, `Article`, `MachineReadable`, `Execution`, `ActionOperation`, ...), plus model-facing types from `engine/src/types.rs` (`Value`, `Operation`, `ParameterType`) |
| 2 | `packages/corpus/src/source_map.rs` | A private partial mirror (`LawDoc`, `LawArticle`, `LawMr`, `LawExec`, `LawOutput`, ...) used by `collect_law_outputs` / `collect_law_implements`, **plus** line-based string parsing of `$id` and `name` (`extract_law_id`, `extract_law_name`) |
| 3 | `packages/harvester/src/yaml/writer.rs` | A third private struct set (`YamlLaw`, `YamlArticle`, `YamlPreamble`, `YamlReference`) that *serializes* law YAML — the write side of the format — with a stringly-typed `regulatory_layer: String` |
| 4 | `packages/tui/src/backend/corpus_scanner.rs` | A fourth hand-rolled line scanner (`extract_metadata`: `strip_prefix("$id:")`, counts `- number:` lines for article count). The tui depends on `regelrecht-engine` but not on `regelrecht-corpus`, so it could not reuse #2 even if it wanted to |
| 5 | `packages/pipeline/src/enrich.rs` | Untyped `serde_yaml_ng::Value` tree-walking (`count_articles_in_value`, `machine_readable` presence checks) |
| 6 | `frontend/` (18 files) | JavaScript property access (`machine_readable.execution.input`, ...) on js-yaml-parsed objects, with no type definitions at all |

The root cause is structural, not carelessness: `regelrecht-corpus` deliberately
does not depend on `regelrecht-engine` (the dependency is reversed — corpus
*dev*-depends on engine for integration tests), and the engine is the only crate
that has the full model. So every other consumer re-derives the subset it needs.
`regelrecht-shared`, the natural home for cross-crate types, today holds only
`RegulatoryLayer` and a telemetry helper.

The duplication is not just redundant — the copies *disagree*. The harvester
writer treats `regulatory_layer` as a free-form string while the engine parses
it as an enum; the corpus mirror and the tui scanner silently ignore fields
the engine considers meaningful; the tui's article counter (`- number:` line
matching) counts things that are not articles if a nested list ever uses a
`number` key at the wrong indentation.

### Finding 2: a format change touches 7–9 places, none generated

Adding or changing a field in the law format currently requires manually
updating, in order:

1. `schema/vX.Y.Z/schema.json` — a **new** version directory; released
   directories are immutable, enforced by the `protect-schema` CI job
2. the `schema/latest` symlink
3. `packages/engine/src/article.rs` — the serde structs
4. `packages/engine/Cargo.toml` — `package.metadata.regelrecht.supported-schemas`
   (a hand-maintained version list)
5. `packages/engine/src/config.rs` — `SUPPORTED_SCHEMAS`, a second copy of that
   list ("must match the metadata in Cargo.toml", says the comment)
6. `packages/engine/src/bin/validate.rs` — a manual `include_str!` per schema
   version plus a hand-written `detect_version` if/else chain
7. `packages/corpus/src/source_map.rs` — the mirror, if the field matters there
8. `packages/harvester/src/yaml/writer.rs` — the write side, if harvested laws
   carry the field
9. frontend JS — find the affected property accesses by grep
10. `conformance/` fixtures — if the change affects execution semantics

No schemars, typify, ts-rs, utoipa or comparable tooling exists in the repo in
either direction. Nothing fails to compile when step 3 and step 1 drift apart;
we find out when a law fails to load, or worse, loads with a silently dropped
field. Recent format work (units/`type_spec`, `valid_to` in RFC-019) has paid
this tax in full.

### Current dependency graph

The internal crate graph is acyclic and layered:

```text
engine     → shared
harvester  → shared
corpus     → (no internal deps; dev-dep on engine)
pipeline   → corpus, harvester, shared
admin      → auth, shared, corpus, harvester, pipeline
editor-api → auth, shared, corpus, pipeline
tui        → engine
frontend   → engine (via WASM), js-yaml (untyped)
```

There is room at the bottom of this graph for a leaf crate that everyone can
depend on.

## Decision

This RFC proposes four things: (1) extract the law model into a new leaf crate,
(2) generate `schema.json` from the Rust structs (schemars direction) with a CI
gate, (3) generate TypeScript types from the same structs with ts-rs, and
(4) do this in six stages that each keep every package compiling. Each decision
is argued below; alternatives are in the Why section.

### 1. A new leaf crate: `regelrecht-law-model`

Extract the document model from `packages/engine/src/article.rs` and the
model-facing parts of `packages/engine/src/types.rs` into a new workspace
member `packages/law-model` (crate name `regelrecht-law-model`).

**Contents:**

- All document structs/enums from `article.rs`: `ArticleBasedLaw`, `Article`,
  `MachineReadable`, `Execution`, `Input`, `Output`, `Parameter`, `Source`,
  `TypeSpec`, `Produces`, `Action`, `ActionValue`, `ActionOperation`, `Case`,
  `Definition`, `OpenTerm`, `OpenTermDefault`, `ImplementsDeclaration`,
  `HookPoint`, `HookFilter`, `HookDeclaration`, `OverrideDeclaration`,
  `StageRequirement`, `Stage`, `ProcedureDefinition`, `ProcedureAppliesTo`,
  `UntranslatableEntry`, `CompetentAuthority`, `LegalBasis`, ...
- From `types.rs`, the types the document model references: `Value`,
  `Operation` (used by `Action.operation`), `ParameterType`, and related
  enums that describe the *format* rather than the *evaluator*.
- A re-export of `RegulatoryLayer` from `regelrecht-shared` (it stays in
  shared; `law-model` depends on `shared`, mirroring what engine and harvester
  already do).
- The YAML loading entry points (`from_yaml_str`, `from_yaml_file`) including
  the existing size limits, and `schema_version()`.
- A new **`parse_law_header()`**: a cheap, *deliberately tolerant* extraction
  of `$id`, `name`, `$schema`, `regulatory_layer`, `publication_date`,
  `valid_from`, `bwb_id` and an article count, for the consumers that today
  hand-roll line scanners (corpus `SourceMap`, tui). Important nuance: the
  line-based parsing in `source_map.rs` is not just a shortcut — it works on
  YAML that does **not** parse as a whole (e.g. a draft mid-edit in the
  editor). Tolerance to malformed bodies is a *requirement* of the header
  parser, not an implementation accident. It must be specified and tested as
  such (header fields are read even when the document body is invalid YAML),
  so it can replace both existing line scanners without behavior change.

What stays in the engine: evaluation (`resolver`, `service`, operations
semantics), `Operation::is_comparison()`-style evaluator helpers if they prove
engine-specific, receipts, config limits for execution depth, and everything
WASM-binding related. The engine re-exports the model types from their old
paths (`regelrecht_engine::article::*`) so downstream code and the WASM API
are unaffected.

**Consumers after extraction:** engine, corpus, harvester, tui, pipeline (and
transitively admin, editor-api). The corpus crate gets the model as a *regular*
dependency — which it can, because `law-model` is a leaf below both corpus and
engine. The corpus→engine dev-dependency can stay for end-to-end tests or be
re-evaluated; it no longer carries model duty.

**Dependency graph after:**

```text
law-model  → shared
engine     → law-model, shared
harvester  → law-model, shared
corpus     → law-model (regular dep; dev-dep engine optional)
tui        → engine, law-model
pipeline   → corpus, harvester, shared, law-model
```

**Why a new crate and not growing `regelrecht-shared`:** both were considered
(see Alternatives). In short: `shared` is the grab-bag for small cross-cutting
types (`RegulatoryLayer`, telemetry) with near-zero dependencies, consumed by
most crates including ones that never touch law YAML (auth-adjacent code in
admin/editor-api). The law model is ~3,000 lines of serde with a YAML
dependency, will grow `schemars`/`ts-rs` feature gates, and is conceptually a
*specification artifact* — per RFC-013, the schema is a spec and the engine an
implementation; the Rust model is the spec's reference representation and
deserves its own versioning. Mixing that into `shared` couples its release
cadence to telemetry helpers and forces YAML-related dependencies (or feature
gymnastics) on crates that only want `RegulatoryLayer`.

### 2. Codegen direction: schemars (structs → schema), CI-verified

Derive `schemars::JsonSchema` on the law-model structs (behind a `schema-gen`
feature, off by default) and add a generator binary plus a CI gate:

- `cargo run -p regelrecht-law-model --features schema-gen --bin generate-schema`
  writes the schema derived from the structs.
- CI job `schema-sync`: regenerates and diffs against
  `schema/latest/schema.json`. A mismatch fails the build with "the model and
  the released schema have drifted; cut a new schema version or fix the model".

This direction — Rust structs as the source of truth, JSON Schema as a
generated, then *frozen* artifact — was chosen over typify (schema → structs)
because of two repo constraints:

**The immutable-versions constraint.** `schema/vX.Y.Z/` directories are
append-only; the `protect-schema` CI job rejects any modification to a version
directory that exists on main. Generated artifacts fit this naturally: when the
model changes, the generator output no longer matches `schema/latest`, the gate
fails, and the author must create `schema/vNEXT/` (with the generated content)
and repoint the symlink — exactly the release flow we have today, but with the
schema content produced mechanically instead of hand-edited. The historic
directories are never regenerated and never touched. Under typify, the
hand-written schema would be the source and the structs generated — but then
*which* schema? We support eight versions simultaneously (`v0.2.0`–`v0.5.2`);
typify would either generate eight struct sets (the engine has exactly one
model, see Risks) or generate from `latest` only, in which case the historic
schemas constrain nothing and we have gained little.

**Serde-attribute fidelity.** The existing model is not mechanically derivable
from the schema: it uses `#[serde(untagged)]` (`ActionValue`,
`CompetentAuthority`), an internally-tagged operation enum with an alias
(`IF`/`SWITCH`), field aliases (`url`/`ref`), `$schema`/`$id` renames, and a
`#[serde(skip)]` computed field (`content_hash`). typify's generated code
cannot reproduce these hand-tuned attributes, and the harvester *serializes*
through this model, so round-trip shape fidelity matters. schemars works in the
direction that preserves the hand-tuned serde model and derives the schema from
it.

**Honest caveat:** the first generated schema will *not* be byte-identical to
the hand-written `schema/v0.5.2/schema.json` (different `description` strings,
different `oneOf` factoring, jsonschema-draft idioms). The gate therefore
cannot start as "generated == latest". The realistic adoption path is: generate,
manually reconcile semantic differences (places where the hand-written schema
is stricter or looser than the serde model — finding those is itself valuable),
release the reconciled output as the next schema version, and *from then on*
enforce equality. If reconciliation surfaces real divergence between what the
schema promises and what the engine accepts, that is a bug-fix discussion, not
a blocker for this RFC.

The same mechanism shrinks the version-list duplication: a small build-time
check (or `build.rs`) asserts that `config.rs::SUPPORTED_SCHEMAS`, the
Cargo.toml `supported-schemas` metadata, and the versions embedded in
`validate.rs` agree, and `validate.rs`'s per-version `include_str!` list can be
generated from the `schema/` directory listing. This does not remove the
human decision of *which* versions are supported — that stays a deliberate
table per RFC-013 — it removes the three hand-synchronized copies of it.

### 3. TypeScript types: ts-rs on the law model

Derive `ts_rs::TS` on the law-model structs (behind a `ts-gen` feature, off by
default) and generate `frontend/src/types/law-model.d.ts` (checked in;
regenerated and diffed by the same CI gate as the schema).

ts-rs is preferred over an OpenAPI/utoipa pipeline for *this* problem because
the frontend's law object does not arrive through a typed HTTP API: the editor
fetches raw YAML and parses it client-side with js-yaml. There is no OpenAPI
surface that describes "the parsed law document" — utoipa would document
editor-api's endpoints, which mostly transport YAML strings and corpus
metadata, not the law shape. ts-rs maps the structs (including the untagged
and internally-tagged enums) directly to the TypeScript shapes the 18
`machine_readable.*`-touching files actually consume.

Scope note: the frontend is JavaScript, not TypeScript, so the generated
`.d.ts` is advisory until files opt in (JSDoc `@type` imports or incremental
TS adoption). That migration is explicitly *not* part of this RFC; the RFC only
guarantees that an authoritative, generated type definition exists and cannot
drift. utoipa/OpenAPI for the *API DTOs* (trajects, notes, scenarios) remains a
reasonable separate future step and is not foreclosed by this choice.

### 4. Migration plan: six stages, every stage compiles

Each stage is one reviewable PR; the workspace builds and all tests pass at
every stage boundary.

- **Stage 1 — extract.** Create `packages/law-model`, move the structs, add
  `pub use regelrecht_law_model::...` re-exports in
  `regelrecht_engine::article` and `regelrecht_engine::types` at the old
  paths. No downstream crate changes; the WASM build and public API are
  byte-compatible. Add `parse_law_header()` with tests covering the
  malformed-body tolerance cases.
- **Stage 2 — corpus.** Replace `source_map.rs`'s private mirror
  (`LawDoc`/`LawMr`/`LawExec`/...) with law-model types and replace
  `extract_law_id`/`extract_law_name` with `parse_law_header()`. Corpus gains
  `regelrecht-law-model` as a regular dependency. Existing source-map tests
  (including the editor-draft tolerance behavior) must pass unchanged.
- **Stage 3 — harvester.** Replace the writer's `YamlLaw`/`YamlArticle`
  structs with serialization through the law-model types, turning
  `regulatory_layer` from `String` into the enum. Golden-file tests pin the
  emitted YAML byte-for-byte before/after (the corpus on disk must not churn).
- **Stage 4 — tui + pipeline.** tui's `corpus_scanner::extract_metadata` and
  pipeline's `count_articles_in_value` move to `parse_law_header()` /
  law-model helpers. Note these also must tolerate partially-enriched or
  malformed files (enrichment counts articles on files an LLM is mid-rewrite
  on), so they use the tolerant header parser, not strict deserialization.
- **Stage 5 — codegen gates.** Add `schema-gen` (schemars) and the
  `schema-sync` CI job, including the one-time reconciliation release
  described above; add the supported-schemas consistency check and the
  generated `include_str!` list in `validate.rs`.
- **Stage 6 — TS types.** Add `ts-gen` (ts-rs), generate
  `frontend/src/types/law-model.d.ts`, add the CI diff gate, annotate one or
  two high-traffic composables (`useLaw.js`, `useDependencies.js`) with JSDoc
  type imports as a demonstrator.

Stages 2–4 are independent of each other (any order); 5 and 6 depend only on 1.

### Non-goals

- **No law-format changes.** The YAML on disk is identical before and after;
  no schema version is introduced by stages 1–4 (stage 5's reconciliation
  release is content-neutral unless reconciliation finds actual bugs, which
  would be handled separately).
- **No behavior changes.** Loading, validation outcomes, harvester output
  bytes, source-map conflict resolution, and engine execution results are
  unchanged; tests pin this at each stage.
- **No frontend TypeScript migration.** Only generated type definitions plus
  a demonstrator.
- **No engine/schema versioning policy changes.** RFC-013's
  supported-schemas table remains a deliberate, human-curated list.

## Why

### Benefits

- One model, six consumers: a format change becomes "edit the law-model crate,
  follow the compiler", instead of grepping for five shadow copies and hoping.
- The schema can no longer silently drift from the engine: CI fails on
  divergence (Finding 2's steps 1, 3, 4, 5, 6 collapse into generated or
  checked artifacts).
- The frontend gets its first authoritative types for the law object.
- Known copy disagreements (stringly-typed `regulatory_layer` in the
  harvester, divergent article counting) are eliminated rather than patched.
- The corpus crate sheds its awkward position of re-implementing the model it
  is forbidden to import.

### Tradeoffs

- **A new workspace member** and the coordination cost of a crate whose API
  many crates depend on; model changes now ripple as semver-relevant changes
  to a shared crate (this is the point, but it is also friction).
- **The generated schema will differ cosmetically** from the hand-written one;
  the one-time reconciliation in stage 5 is real work and may surface
  uncomfortable spec-vs-implementation discrepancies that then demand
  decisions.
- **schemars describes the *current* model only.** Historic schema versions
  remain hand-written artifacts; nothing mechanically ties `v0.3.x` to any
  code anymore (it never did — see Risks for why this is acceptable).
- **Feature-gate complexity**: `schema-gen` and `ts-gen` must stay out of
  default and WASM builds, which is one more thing to police (a CI check on
  the WASM artifact's dependency set is cheap insurance).
- **ts-rs types are only as useful as their adoption** in a JS codebase; until
  files opt in, the gate prevents drift of the artifact, not misuse in code.

### Alternatives considered

**Grow `packages/shared` instead of a new crate.**
- Pro: no new workspace member; `RegulatoryLayer` already lives there; most
  crates already depend on it.
- Con: `shared` is deliberately dependency-light and consumed by crates with
  no law-YAML business; the model brings serde_yaml_ng plus optional
  schemars/ts-rs; release cadence and feature flags get entangled with
  telemetry. The audit's root cause was "no correct home for the model" — a
  grab-bag is a home, but not a correct one.
- **Rejected** in favor of a dedicated leaf crate; `shared` keeps
  `RegulatoryLayer` (re-exported by law-model) so nothing breaks.

**typify (schema → structs).**
- Pro: the JSON Schema (already the official spec per RFC-013) becomes the
  single source of truth; conceptually clean for third-party engines.
- Con: cannot reproduce the hand-tuned serde attributes the model relies on;
  unclear story for eight coexisting schema versions versus one engine model;
  would replace ~3,000 lines of reviewed, security-annotated code (size
  limits, depth limits) with generated code of larger blast radius.
- **Rejected**; the schema remains the *interchange* spec, generated and
  frozen, while the Rust model is its reference implementation.

**OpenAPI/utoipa for frontend types.**
- Pro: types and HTTP contract in one artifact; standard tooling.
- Con: the law document does not travel through a typed JSON API — it is
  client-side-parsed YAML; utoipa would type the envelope, not the payload.
- **Rejected for the law model**; remains open for API DTOs later.

**Do nothing / document the copies.**
- Pro: zero migration risk.
- Con: Finding 2 is a recurring tax on every format RFC (units, valid_to,
  foreach are all recent or in flight), and the copies already disagree.
- **Rejected.**

### Risks and open questions

- **WASM build weight.** The engine's `wasm` feature builds a cdylib for the
  frontend. The law-model crate's default feature set must be exactly what
  the engine needs today (serde, serde_yaml_ng, thiserror); schemars and
  ts-rs stay behind non-default features that the WASM build never enables.
  Risk is low but regressions are silent (bundle grows); add a size check or
  dependency-tree assertion to CI.
- **Compile times.** One more crate is one more codegen unit, but moving
  ~3,000 lines out of the engine lets corpus/harvester/pipeline build against
  the small model crate without waiting for the engine. Expected neutral to
  slightly positive; worth measuring in stage 1's PR.
- **The corpus↔engine dev-dep.** After stage 2 there is no *model* reason for
  corpus to dev-depend on engine; keeping it for end-to-end tests is fine and
  creates no cycle (dev-dependencies do not participate in the build graph of
  dependents). Dropping it is a nice-to-have, not a requirement.
- **Schema-version multiplicity.** The model tracks ONE current shape; older
  documents (v0.2.0–v0.5.1) load because the model is structurally permissive
  (optional fields, defaults, aliases), and *validation* of older documents
  continues to use the frozen per-version JSON schemas in `validate.rs` —
  that mechanism is untouched. The honest statement: this was already true
  before this RFC (article.rs never had per-version variants); codegen makes
  it visible rather than making it worse. If the permissive-superset strategy
  ever breaks (a field changing *meaning* between versions rather than being
  added), that is a format-versioning problem for a future RFC regardless of
  where the structs live.
- **Tolerant parsing is load-bearing.** `extract_law_id` working on broken
  YAML is relied on by the editor flow (drafts) and enrichment (mid-rewrite
  files). `parse_law_header()` must encode this contract explicitly; stage 2
  and 4 reviews should treat any tightening of accepted inputs as a breaking
  change.
- **Who owns the law-model crate's API stability?** Proposal: the same review
  bar as `schema/` changes — model PRs that alter (de)serialization require a
  schema-version decision, enforced mechanically by the stage-5 gate.
