# RFC-016: Burger-demo met geïsoleerd corpus

**Status:** Proposed
**Date:** 2026-04-21
**Authors:** Tim de Jager

## Context

De externe PoC `poc-machine-law` (Python/FastAPI) laat zien hoe wet-als-code voor burgers voelt: je kiest een persona, klikt op "Zorgtoeslag", en ziet direct het bedrag, de redenering, en — als je wilt — een simulatie over een synthetische populatie. Dat narratief wil regelrecht ook tonen, maar dan op de eigen stack (Rust engine + Vue + `@minbzk/storybook`) en zonder de architecturale en juridische compromissen van de PoC mee te slepen.

Twee van die compromissen staan op gespannen voet met onze principes:

1. **Niet-wet-getrouwe YAML.** Een demo moet 100 % werken voor één datum. De PoC bevat daarom hardgecodeerde bedragen waar de wet zegt "bij ministeriële regeling", en simplificaties rond temporale randgevallen. Voor een demo acceptabel, voor het hoofdcorpus niet.

2. **Demo-eigen vormgeving.** Een burger-UI heeft andere behoeften dan de editor — dashboard-kaarten, formulieren per wet, simulatie-grafieken — die niet thuishoren in de editor-Vue-app.

We hebben een weg nodig om de demo te bouwen zonder het hoofdcorpus of de editor te vervuilen.

## Decision

We bouwen een geïsoleerde burger-demo als vierde frontend-app, gekoppeld aan de bestaande Rust-engine via WASM.

De demo-scope v1 is: één wet (Zorgtoeslag), één persona (Merijn), execution-trace zichtbaar, LLM-uitleg via een kleine backend-proxy, en een populatie-simulatie in een Web Worker.

De demo deelt alleen de engine en het RR-Components-ontwerpsysteem met de rest van het monorepo. Alle demo-specifieke YAML (persona's, demo-index) leeft in een nieuwe `corpus-demo/` die buiten `just validate` valt. Demo-wetten die al law-faithful in `corpus/regulation/` staan (Zorgtoeslag) worden hergebruikt, niet gekopieerd; nieuwe demo-only wetten landen in `corpus-demo/regulation/` met een README die uitdrukkelijk vermeldt dat afwijkingen van de wet zijn toegestaan mits gedocumenteerd.

Repo-layout na deze RFC:

```
corpus/regulation/    # ongewijzigd, law-faithful, shared
corpus-demo/          # nieuw, demo-only
  README.md           # beleid + checklist
  demo-index.yaml     # welke wetten in de demo zichtbaar zijn
  profiles/           # persona-data
  regulation/         # demo-only YAML (v1 leeg — zorgtoeslag wordt hergebruikt)
frontend-demo/        # nieuwe Vue-app, alleen RR-Components
packages/demo-api/    # Axum-proxy met één route POST /api/explain
```

Uitvoering gebeurt volledig client-side (WASM); de Axum-proxy is er uitsluitend voor LLM-uitleg, omdat `ANTHROPIC_API_KEY` niet in de browser mag. De proxy heeft rate-limiting per IP om misbruik op de publieke demo-URL te voorkomen.

## Why

### Benefits

- **Hoofdcorpus blijft law-faithful.** Demo-vereenvoudigingen kunnen niemand in de war brengen, omdat ze letterlijk in een ander pad leven met een README die het beleid vermeldt.
- **Editor wordt niet vervuild.** `frontend/` blijft een editor; burgerzaken zitten in `frontend-demo/`.
- **Minimale nieuwe infra.** WASM in de browser betekent geen extra execute-service; alleen een dunne LLM-proxy.
- **Hergebruik waar het kan.** Zorgtoeslag-YAML wordt niet gekopieerd — één bron, één waarheid voor wetten die al kloppen.
- **Engine roadmap profiteert.** Alles wat de demo nodig heeft en nu ontbreekt (bv. meer trace-metadata voor uitleg), wordt als aparte engine-issue opgevoerd en komt ten goede aan iedereen — niet als demo-hack.

### Tradeoffs

- **Vierde frontend-app.** Meer CI-werk, meer dependencies bij te houden. Beperkt via het delen van het storybook-package.
- **Kleine Axum-proxy erbij.** Extra deploy-unit alleen voor LLM-uitleg. Als we LLM-uitleg schrappen, kan de demo puur statisch.
- **Twee corpus-paden.** Auteurs moeten weten waar wat landt. Mitigatie: `corpus-demo/README.md` is expliciet, `just validate` slaat het over zodat demo-YAML nooit per ongeluk law-faithful wordt beoordeeld.
- **WASM-bundle wordt groter** door ingebakken corpus. Voor één wet verwaarloosbaar; bij uitbreiding nadenken over lazy-loading.

### Alternatives Considered

**Alternative 1: Demo in `frontend/` als extra route**
- Het bestaande editor-frontend krijgt een `/demo`-route.
- Afgewezen: de editor is een professional-tool met eigen deps en styling. Een burger-demo eraan hangen verhoogt de onderhoudslast zonder dat we iets hergebruiken dat de moeite waard is; routing, auth, layouts en state-model zijn wezenlijk anders.

**Alternative 2: Volledige kopie van Zorgtoeslag in `corpus-demo/`**
- 100 % gescheiden corpus, geen cross-reads.
- Afgewezen: dubbele bron van waarheid betekent dat elke wet-update op twee plekken moet gebeuren, met risico op divergentie. Voor law-faithful wetten is hergebruik veiliger.

**Alternative 3: Uitvoering via HTTP i.p.v. WASM**
- Dedicated `packages/demo-api/` met `POST /api/execute`.
- Afgewezen voor v1: voegt een stateful-unit toe zonder meerwaarde voor deze use-case. De editor doet al WASM-in-browser en het werkt. Bij een tweede client (MCP-server, Slack-bot) kan dit alsnog.

**Alternative 4: LLM-uitleg volledig client-side met publieke key**
- Anthropic-key in frontend-bundle.
- Afgewezen: keys horen niet publiek. Rate-limiting ook niet betrouwbaar client-side.

**Alternative 5: Demo-simplifications inline in `corpus/regulation/` met een `demo: true`-veld**
- Markeer velden die niet law-faithful zijn in het hoofdcorpus.
- Afgewezen: zet de deur open voor sluipende simplifications. Fysieke scheiding maakt het beleid afdwingbaar.

### Implementation Notes

- `justfile` sluit `corpus-demo/` uit van `validate`; pre-commit YAML-lint gaat wel door.
- `frontend-demo/` gebruikt een poort in het bereik 7100–7300 (container port-mapping) en bindt op `0.0.0.0`.
- `packages/demo-api/` draait standalone; in productie zit het achter pad-routing of op een subdomein naast de demo-frontend.
- Demo-YAML die wel in `corpus-demo/regulation/` landt moet in zijn eigen README-sectie de afwijkingen vermelden, zodat een lezer altijd ziet wat bewust vereenvoudigd is.
- De RFC legt geen eisen op aan toekomstige demo-wetten buiten Zorgtoeslag; die uitbreiding is een beslissing per wet.
