# RFC-012: Untranslatables — Handling Legal Constructs Beyond Engine Expressiveness

**Status:** Proposed
**Date:** 2026-04-01
**Authors:** Eelco Hotting

## Context

The law-generate skill instructs an LLM to produce `machine_readable` sections for Dutch law YAML files. It iterates up to 3 times through a generate→validate→test loop until the output passes schema validation and BDD tests.

The engine's operation set is deliberately small — arithmetic, comparison, logical, conditional, and a handful of date operations. We only extend it when real legal texts demand a new construct. This RFC is about streamlining what happens at that moment of discovery.

However, Dutch law regularly uses constructs that fall outside the current operation set. When the LLM encounters these constructs, it has no "I can't express this" path. The skill tells it to iterate and fix failures, so it will:

1. Flatten a bracket table into a giant `IF`/cases tree (fragile, error-prone for large tables)
2. Pre-compute values that should be dynamic
3. Silently omit conditions it cannot express
4. Invent creative encodings that pass validation but misrepresent the law

This is dangerous because the result looks correct — it passes schema validation, may even pass BDD tests for the tested cases — but silently diverges from what the law actually says. The reverse-validate skill checks traceability to legal text, but not whether the chosen operations faithfully capture the semantics.

We call these constructs **untranslatables**: the law is clear about what it means, but the engine's formal language cannot express it. The term is deliberately borrowed from translation theory — the entire law-generate process *is* translation (legal Dutch → formal operations), and some things don't cross that boundary.

## Decision

Introduce an **untranslatables mechanism** across three layers: generation-time detection, schema annotation, and engine runtime behavior.

### Layer 1: Generation-time detection (immediate)

Add explicit instructions to the law-generate skill that define when the model must bail out instead of improvise. The model must recognize patterns like:

- "afronden" (rounding) → no rounding operation exists
- "totaal van" / "som van" over a variable-length collection → no aggregation exists
- Bracket tables with >2 dimensions → IF/cases becomes unreliable
- "het aantal dagen/maanden/jaren tussen" → no DATE_DIFF exists

When the model identifies such a construct, it must:
1. Add an `untranslatables` entry to the article's `machine_readable` section (see Layer 2)
2. Skip the untranslatable parts of the execution logic — do not approximate
3. Generate execution logic for the parts that *are* translatable, if any
4. Include all untranslatables in the Phase 5 report

Additionally, add heuristics to the law-reverse-validate skill to detect likely workarounds that indicate the model improvised instead of flagging:

- `IF` with >8 cases → possible inlined bracket table
- Arithmetic chains that approximate rounding (`MULTIPLY` → `DIVIDE` → `MULTIPLY` with powers of 10)
- Hardcoded values that look like pre-computed aggregations
- Output values that don't correspond to any single operation's semantics

These are flagged as "possible untranslatable workaround — verify with human."

### Layer 2: Schema annotation (short-term)

Add an optional `untranslatables` field to the `machine_readable` schema:

```yaml
machine_readable:
  untranslatables:
    - construct: "afronden op hele euro's"
      reason: "Rounding is not available as an engine operation"
      suggestion: "Add ROUND/CEIL/FLOOR operation to engine"
      legal_text_excerpt: "Het bedrag wordt naar boven afgerond op hele euro's"
  execution:
    # ... execution logic for the parts that ARE translatable
```

This makes untranslatables:
- **Structured** — queryable by tooling
- **Persistent** — survives across generate/validate cycles
- **Visible** — the admin dashboard and pipeline can surface them
- **Co-located** — lives next to the article it applies to

Articles with untranslatables may still have partial `execution` logic for the parts that *are* expressible.

### Layer 3: Engine runtime behavior

When the engine loads a law file and encounters articles with `untranslatables`, its behavior is controlled by an execution mode flag:

| Mode | On untranslatable | Use case |
|------|-------------------|----------|
| `--untranslatable=error` (default) | Hard error, stop execution | CI, production — safest default |
| `--untranslatable=propagate` | Produce `UNTRANSLATABLE` value that taints downstream outputs | Audit, analysis — "show me what's affected" |
| `--untranslatable=warn` | Skip article, log warning in trace | Development, exploration |
| `--untranslatable=ignore` | Execute anyway (pretend field isn't there) | Human-verified workarounds |

**Default is fail-fast.** You must explicitly opt in to tolerating gaps. This follows the principle of least surprise: if a law has untranslatable constructs, running it without acknowledgement should fail loudly rather than produce partial or tainted results.

**The trace always records untranslatables regardless of mode.** The flag controls what the engine *does* about them, not whether it *notices*.

In `propagate` mode, the `UNTRANSLATABLE` value behaves like `NaN` in floating point: any operation involving an untranslatable input produces an untranslatable output. At the end of execution, the result shows exactly which outputs are tainted and which are trustworthy. The trace captures the origin point (which article, which construct).

## Why

### Benefits

- **Prevents silent divergence** between law text and machine-readable interpretation — the most dangerous failure mode
- **Fail-fast default** ensures untranslatables are never silently ignored in CI or production
- **Propagation mode** gives full visibility into which outputs are affected, without stopping execution
- **Prioritizes engine roadmap** — untranslatables directly tell us which operations to add next, driven by real legal constructs rather than speculation
- **Preserves trust** — a flagged gap is better than a plausible-looking wrong answer
- **Works incrementally** — Layer 1 (skill instructions) is zero-cost and can ship today; Layers 2-3 add structure over time
- **Makes partial progress visible** — an article with untranslatables is better than a silently wrong article or a completely skipped one

### Tradeoffs

- **More articles will be incomplete** — the generate skill will produce fewer complete `machine_readable` sections in the short term, because it can no longer paper over gaps
- **Schema change required** for Layer 2 — adds an `untranslatables` field that all tooling must handle gracefully
- **Heuristic detection has false positives** — a legitimate 10-case IF is not necessarily a workaround; human judgment is still needed
- **LLM compliance is imperfect** — even with explicit instructions, models may still improvise; the reverse-validate heuristics provide a safety net but are not foolproof
- **Propagation adds complexity to the engine** — a new value type that must be handled in every operation

### Alternatives Considered

**Alternative 1: Expand the engine operation set preemptively**
- Add ROUND, SUM, TABLE_LOOKUP, etc. before they're needed
- Rejected: violates the project's principle of minimal, auditable operations; we should add operations when we have concrete legal texts that need them, not speculatively

**Alternative 2: Use YAML comments only (no schema field)**
- Flag issues as `# UNTRANSLATABLE: ...` comments
- Rejected as sole mechanism: comments are invisible to tooling, can't be queried or tracked, and are easily lost during edits

**Alternative 3: Fail the entire law if any article is untranslatable**
- Abort generation for the whole file if one article can't be expressed
- Rejected: too aggressive. Most laws have a mix of translatable and untranslatable articles. Partial coverage with clear annotations is more useful than no coverage.

**Alternative 4: Always propagate `UNTRANSLATABLE` (no fail-fast default)**
- Default to propagation mode, require `--strict` for errors
- Rejected: silent partial results in production are dangerous. The safe default is to require explicit acknowledgement of gaps.

### Implementation Notes

**Layer 1** requires only editing the law-generate and law-reverse-validate skill definitions. No code changes.

**Layer 2** requires:
- Adding `untranslatables` to `schema/v0.5.0/schema.json` (or a new schema version)
- Updating `packages/corpus/` to parse the field
- Updating `packages/admin/` to display untranslatables
- Updating `packages/pipeline/` to track untranslatable counts per law

**Layer 3** requires:
- Adding an `Untranslatable` variant to the engine's `Value` type
- Propagation logic in every operation (similar to NaN handling)
- CLI flag parsing (`--untranslatable=propagate|warn|ignore`)
- Trace entries for untranslatable encounters
- Updating the execution result to distinguish clean outputs from tainted outputs

**Ordering**: Layer 1 → Layer 2 → Layer 3. Each layer is independently useful.
