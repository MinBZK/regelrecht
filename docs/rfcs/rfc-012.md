# RFC-012: Expressibility Limits — Escalation When Machine-Readable Generation Hits Engine Boundaries

**Status:** Proposed
**Date:** 2026-04-01
**Authors:** Eelco Hotting

## Context

The law-generate skill instructs an LLM to produce `machine_readable` sections for Dutch law YAML files. It iterates up to 3 times through a generate→validate→test loop until the output passes schema validation and BDD tests.

The engine's operation set is deliberately small: arithmetic (`ADD`, `SUBTRACT`, `MULTIPLY`, `DIVIDE`, `MIN`, `MAX`), comparison, logical, `IF`/cases, `IN`, `LIST`, and four date operations (`AGE`, `DATE_ADD`, `DATE`, `DAY_OF_WEEK`). This is by design — a minimal, auditable set.

However, Dutch law regularly uses constructs that fall outside this set:

| Legal construct | Example | Missing operation |
|----------------|---------|-------------------|
| Rounding | "afgerond op hele euro's naar boven" | `ROUND`, `CEIL`, `FLOOR` |
| Aggregation over collections | "het totaal van alle inkomsten" | `SUM`, `COUNT`, `FOREACH` |
| String operations | "de aanhef luidt: ..." | `CONCAT`, `CONTAINS` |
| Table/bracket lookups | Multi-dimensional tax brackets | `TABLE_LOOKUP` |
| Modulo/periodicity | "per volle kalendermaand" | `MODULO` |
| Date difference in units | "het aantal maanden tussen X en Y" | `DATE_DIFF` |

When the LLM encounters these constructs, it has no "I can't express this" path. The skill tells it to iterate and fix failures, so it will:

1. Flatten a bracket table into a giant `IF`/cases tree (fragile, error-prone for large tables)
2. Pre-compute values that should be dynamic
3. Silently omit conditions it cannot express
4. Invent creative encodings that pass validation but misrepresent the law

This is dangerous because the result looks correct — it passes schema validation, may even pass BDD tests for the tested cases — but silently diverges from what the law actually says. The reverse-validate skill checks traceability to legal text, but not whether the chosen operations faithfully capture the semantics.

## Decision

Introduce an **expressibility escalation mechanism**: when machine-readable generation encounters a legal construct that cannot be faithfully represented with the available engine operations, it must **stop and flag it for human review** rather than approximate.

This mechanism has three layers:

### Layer 1: Skill instructions (immediate)

Add explicit instructions to the law-generate skill that define when the model must bail out instead of improvise. The model must recognize patterns like:

- "afronden" (rounding) → no rounding operation exists
- "totaal van" / "som van" over a variable-length collection → no aggregation exists
- Bracket tables with >2 dimensions → IF/cases becomes unreliable
- "het aantal dagen/maanden/jaren tussen" → no DATE_DIFF exists

When the model identifies such a construct, it must:
1. Skip the article (do not generate a `machine_readable` section)
2. Add a YAML comment explaining why
3. Include it in the Phase 5 report under a new "Escalations" section

### Layer 2: Schema annotation (short-term)

Add an optional `escalations` field to the `machine_readable` schema:

```yaml
machine_readable:
  escalations:
    - construct: "afronden op hele euro's"
      reason: "Rounding is not available as an engine operation"
      suggestion: "Add ROUND/CEIL/FLOOR operation to engine"
      legal_text_excerpt: "Het bedrag wordt naar boven afgerond op hele euro's"
  execution:
    # ... rest of the machine_readable section, if partially expressible
```

This makes escalations:
- **Structured** — queryable by tooling
- **Persistent** — survives across generate/validate cycles
- **Visible** — the admin dashboard and pipeline can surface them
- **Co-located** — lives next to the article it applies to

The `escalations` field is optional and does not affect engine execution. Articles with escalations may still have partial `execution` logic for the parts that *are* expressible.

### Layer 3: Reverse-validate detection (short-term)

Add heuristics to the law-reverse-validate skill to detect likely workarounds:

- `IF` with >8 cases → possible inlined bracket table
- Arithmetic chains that approximate rounding (`MULTIPLY` → `DIVIDE` → `MULTIPLY` with powers of 10)
- Hardcoded values that look like pre-computed aggregations
- Output values that don't correspond to any single operation's semantics

These are flagged as "possible expressibility workaround — verify with human."

## Why

### Benefits

- **Prevents silent divergence** between law text and machine-readable interpretation — the most dangerous failure mode
- **Prioritizes engine roadmap** — escalations directly tell us which operations to add next, driven by real legal constructs rather than speculation
- **Preserves trust** — a flagged gap is better than a plausible-looking wrong answer
- **Works incrementally** — Layer 1 (skill instructions) is zero-cost and can ship today; Layers 2-3 add structure over time
- **Makes partial progress visible** — an article with escalations is better than a silently wrong article or a completely skipped one

### Tradeoffs

- **More articles will be skipped** — the generate skill will produce fewer `machine_readable` sections in the short term, because it can no longer paper over gaps
- **Schema change required** for Layer 2 — adds an `escalations` field that all tooling must handle gracefully
- **Heuristic detection (Layer 3) has false positives** — a legitimate 10-case IF is not necessarily a workaround; human judgment is still needed
- **LLM compliance is imperfect** — even with explicit instructions, models may still improvise; the reverse-validate layer provides a safety net but is not foolproof

### Alternatives Considered

**Alternative 1: Expand the engine operation set preemptively**
- Add ROUND, SUM, TABLE_LOOKUP, etc. before they're needed
- Rejected: violates the project's principle of minimal, auditable operations; we should add operations when we have concrete legal texts that need them, not speculatively

**Alternative 2: Use YAML comments only (no schema field)**
- Flag issues as `# HUMAN_REVIEW: ...` comments
- Rejected as sole mechanism: comments are invisible to tooling, can't be queried or tracked, and are easily lost during edits. Acceptable as Layer 1 alongside the structured approach.

**Alternative 3: Fail the entire law if any article is inexpressible**
- Abort generation for the whole file if one article can't be expressed
- Rejected: too aggressive. Most laws have a mix of expressible and inexpressible articles. Partial coverage with clear escalations is more useful than no coverage.

### Implementation Notes

**Layer 1** requires only editing the law-generate and law-reverse-validate skill definitions. No code changes.

**Layer 2** requires:
- Adding `escalations` to `schema/v0.5.0/schema.json` (or a new schema version)
- Updating `packages/corpus/` to parse the field
- Updating `packages/admin/` to display escalations
- Updating `packages/pipeline/` to track escalation counts

**Layer 3** requires only editing the law-reverse-validate skill definition with detection heuristics. No code changes, but effectiveness depends on the heuristics being well-calibrated — expect iteration.

**Ordering**: Layer 1 → Layer 2 → Layer 3. Each layer is independently useful.
